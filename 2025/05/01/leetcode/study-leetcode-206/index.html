Leetcode-206:反转链表-迭代双指针法

Created2025-05-01|Updated2025-05-01|Java学习笔记Leetcode Hot100
|Word Count:3.1k|Reading Time:10mins|Post Views:

1、题目

给你单链表的头节点 head，请你反转链表，并返回反转后的链表的头节点。

示例

示例 1：

输入：head = [1,2,3,4,5]
输出：[5,4,3,2,1]

示例 2：

输入：head = [1,2]
输出：[2,1]

示例 3：

输入：head = []
输出：[]

提示

  • 链表中节点的数目范围是 [0, 5000]
  • -5000 <= Node.val <= 5000

进阶

可以使用迭代或递归两种方法解决此题，本文重点讲解迭代双指针法（空间复杂度更优，O(1)）。

2、分析

2.1 问题核心

单链表反转的核心是改变节点的next指向，让每个节点从“指向下一个节点”变为“指向前一个节点”。由于单链表无法反向遍历，需通过双指针记录前后节点位置，同时用临时变量备份后续节点，避免断链，这是迭代法的核心逻辑。

2.2 常见错误分析（避坑重点）

这是链表反转中最易踩的坑，核心错误是过早修改下一个节点的指向（如错误执行nxt.next = cur），该操作会直接导致链表断链、后续节点丢失、形成环结构，也是绝大多数反转逻辑出错的根源。

错误危害拆解

  1. 断链丢失节点：修改未处理节点的next指向，会断开其与后续节点的连接，且无任何变量持有后续节点的引用，导致这些节点成为“游离节点”，造成内存泄漏，程序再也无法访问；
  2. 形成环结构：让下一个节点提前指向当前已反转的节点，会使已处理节点和未处理节点互相指向，后续指针遍历会陷入死循环；
  3. 逻辑越界：反转的核心是“处理当前节点，暂存后续节点”，提前操作未处理节点违背该基本逻辑，导致整个反转流程混乱，后续所有操作都失去意义。

错误执行过程图示

错误核心总结：未备份后续节点就修改其指向，或跳过当前节点直接操作下一个节点，是链表反转中所有错误的本质原因。

2.3 核心思路（迭代双指针法）

采用pre（前节点）、cur（当前节点） 双指针，搭配next（临时备份节点），遵循“先备份、再反转、后推进” 的固定三步法，从根本上避免断链、成环和丢失节点问题：

  1. 定义指针：pre初始为null（反转后头节点的next为null），cur初始为head（从原头节点开始处理），next用于临时存储cur.next；
  2. 核心三步：
    • 备份：next = cur.next → 先保存当前节点的下一个节点，这是防止断链的核心步骤；
    • 反转：cur.next = pre → 仅修改当前节点的指向，让其指向前一个节点，不触碰任何未处理节点；
    • 推进：pre = cur、cur = next → 双指针整体后移，准备处理下一个节点；
  3. 循环终止：当cur == null时，所有节点处理完毕，pre指向原链表最后一个节点，即反转后的新头节点。

该方法**时间复杂度O(n)（仅线性遍历一次链表），空间复杂度O(1)**（仅使用3个指针变量，原地修改），是链表反转的最优解。

3.2 代码逐行解释

  • 初始化三个核心指针，各司其职无冗余：
    • pre：记录当前节点的前一个节点，作为反转后的指向目标，初始为null适配反转链表的尾节点要求；
    • cur：指向当前正在处理、需要修改指向的节点，从原链表头节点开始遍历；
    • next：纯临时备份指针，无其他作用，仅用于保存后续节点，避免断链。
  • 循环条件：只要当前节点不为null就继续处理，确保原链表所有节点都被反转（包括最后一个节点），彻底避免节点遗漏。
  • 整个算法的灵魂步骤：修改cur.next之前，必须先备份其下一个节点。若跳过此步，修改cur.next后将彻底失去后续节点的引用，直接导致断链。
  • 反转的核心操作：仅对当前已定位的cur节点进行修改，让其从“指向下一个节点”变为“指向前一个节点”，不涉及任何未处理节点，从根本上避免成环。
  • 双指针同步后移，遵循“先更新pre，再更新cur” 的顺序：
    • pre移动到当前已完成反转的cur节点，作为下一个节点的反转指向目标；
    • cur移动到之前备份的next节点，成为新的待处理节点；
    • 顺序不可颠倒，否则会丢失当前节点的引用。
  • 最易忽略的细节：循环结束时cur已遍历至null（原链表尾节点的下一个位置），而pre恰好停在原链表的最后一个节点，该节点就是反转后的新头节点，返回pre而非cur。

4、算法性能

  • 时间复杂度：O(n)
    仅对链表进行一次线性遍历，每个节点仅被处理一次，操作数与节点数量严格成正比，无任何重复操作。
  • 空间复杂度：O(1)
    仅使用pre、cur、next三个指针变量，无额外数组、哈希表、递归栈等空间开销，实现原地链表反转，满足所有空间复杂度要求。
  • 优势对比：
    与递归法相比，迭代法无需承担递归调用栈的空间开销（递归法空间复杂度O(n)），在节点数量较多（如n=5000）时，更节省内存且无栈溢出风险；与其他暴力方法相比，时间效率最优，是工业界和算法题中的首选解法。

5、测试示例（分步详细演示）

以经典测试用例 链表1 → 2 → 3 → 4 → null 为例，分步演示指针移动和链表反转的完整过程，直观理解每一步的逻辑和链表变化：

初始状态

pre = null，cur = 1，next = null
链表结构：null ← pre 1(cur) → 2 → 3 → 4 → null

第1次循环（处理节点1）

  1. next = cur.next → next = 2
  2. cur.next = pre → 1.next = null，链表：null ← 1(cur) 2(next) → 3 → 4
  3. pre = cur → pre = 1
  4. cur = next → cur = 2
    当前状态：pre=1，cur=2，next=2

第2次循环（处理节点2）

  1. next = cur.next → next = 3
  2. cur.next = pre → 2.next = 1，链表：null ← 1(pre) ← 2(cur) 3(next) → 4
  3. pre = cur → pre = 2
  4. cur = next → cur = 3
    当前状态：pre=2，cur=3，next=3

第3次循环（处理节点3）

  1. next = cur.next → next = 4
  2. cur.next = pre → 3.next = 2，链表：null ← 1 ← 2(pre) ← 3(cur) 4(next) → null
  3. pre = cur → pre = 3
  4. cur = next → cur = 4
    当前状态：pre=3，cur=4，next=4

第4次循环（处理节点4）

  1. next = cur.next → next = null
  2. cur.next = pre → 4.next = 3，链表：null ← 1 ← 2 ← 3(pre) ← 4(cur) null(next)
  3. pre = cur → pre = 4
  4. cur = next → cur = null
    当前状态：pre=4，cur=null，next=null

循环终止

cur == null，退出循环，返回pre=4，反转后最终链表：4 → 3 → 2 → 1 → null，结果完全正确。

6、拓展与总结

6.1 拓展：递归法实现（代码简洁，作为补充）

递归法的核心思路是从后往前反转，先递归反转当前节点的后续链表，再修改当前节点的指向，代码简洁但存在递归调用栈的空间开销（空间复杂度O(n)），作为拓展参考，适合理解递归思想：

6.2 链表反转核心避坑指南（必记）

结合本次错误分析和正确解法，总结4条必记避坑原则，从根本上避免反转出错：

  1. 先备份，再修改：修改任何节点的next指向之前，必须用临时变量备份其后续节点，这是避免断链的唯一方法；
  2. 只处理当前节点：一次循环仅专注处理cur节点，不提前操作next节点，拒绝“画蛇添足”的提前反转；
  3. 指针更新有顺序：严格遵循“先反转，再更新pre，最后更新cur”的顺序，不可颠倒或跳过；
  4. 返回pre不返回cur：循环结束后cur必然为null，pre才是反转后的新头节点，这是最易犯的返回值错误。

6.3 整体总结

迭代双指针法是反转单链表的工业级最优解，其核心价值在于将复杂的链表反转问题，拆解为“备份-反转-推进”的简单重复步骤，既保证了时间和空间的极致效率，又从逻辑上规避了断链、成环、丢失节点等所有常见错误。

该方法是单链表操作的基础核心技巧，掌握后可轻松应对各类链表反转衍生问题（如反转链表前N个节点、反转链表指定区间、K个一组反转链表等），也是LeetCode、校招、笔试中的高频考点，务必熟练掌握每一步的逻辑细节和指针变化，做到“手写无BUG”。

掌握迭代双指针法，不仅能解决链表反转问题，更能培养“操作前备份、分步执行、逻辑闭环” 的算法思维，这种思维在所有链表、树等数据结构操作中都至关重要。
