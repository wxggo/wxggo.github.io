Leetcode-023:合并K个升序链表-分治归并最优解法（对比顺序合并）

Created2026-02-05|Updated2026-02-05|Java学习笔记Leetcode Hot100
|Word Count:4.2k|Reading Time:15mins|Post Views:

前言

合并K个升序链表是Leetcode链表模块的经典Hard难题，也是面试中考察分治思想和归并排序应用的高频压轴题。本题的基础解法是顺序合并（逐个合并链表），实现简单但效率极低；而分治归并解法是最优选择，通过分而治之的思想将时间复杂度从平方级别优化至对数级别，是本题的面试首选写法。

1、题目

给你一个链表数组，每个链表都已经按升序排列。请你将所有链表合并到一个升序链表中，返回合并后的链表。

示例

示例 1：
输入：lists = [[1,4,5],[1,3,4],[2,6]]
输出：[1,1,2,3,4,4,5,6]
解释：合并后的升序链表为所有输入链表的有序合并。

示例 2：
输入：lists = []
输出：[]

示例 3：
输入：lists = [[]]
输出：[]

示例 4：
输入：lists = [[5],[3],[1]]
输出：[1,3,5]

提示

  • k == lists.length
  • 0 <= k <= 10^4
  • 0 <= lists[i].length <= 500
  • -10^4 <= lists[i][j] <= 10^4
  • lists[i] 按升序排列
  • lists[i].length 总和不超过 10^4

2、核心基础：合并两个升序链表

无论是顺序合并还是分治归并，其核心子过程都是合并两个升序链表，这是本题所有解法的基础，必须先掌握。

2.1 合并两个升序链表的核心思路

利用哑节点（dummy） 简化头节点处理，双指针分别遍历两个链表，逐个比较节点值，将较小的节点接在结果链表后，直至其中一个链表遍历完毕，最后将未遍历完的链表直接接在结果链表末尾。

2.3 子方法性能分析

设两个链表的节点数分别为n1、n2，则该方法的时间复杂度O(n1+n2) （每个节点仅遍历一次），空间复杂度O(1) （仅使用有限个指针变量）。

3、两种解法思路与核心差异

本题的两种主流解法均基于上述merge子方法，核心差异在于合并K个链表的策略：顺序合并是「逐个合并，累积结果」，分治归并是「分而治之，逐层合并」。

3.1 解法一：顺序合并（基础版）

3.1.1 核心思路

  1. 初始化结果链表ans为null；
  2. 遍历链表数组，依次将ans与当前链表lists[i]调用merge方法合并，更新ans为合并后的结果；
  3. 遍历结束后，ans即为合并后的最终链表。

简单来说，就是把合并K个链表的问题，转化为K-1次合并两个链表的问题，逐个“吞并”。

3.2 解法二：分治归并（最优版）

3.2.1 核心思路

分治算法的核心是分而治之：将大问题拆分为若干个规模相同的小问题，递归解决小问题后，再将小问题的结果合并，得到大问题的解。
针对本题，分治步骤为：

  1. 分：将链表数组lists按中间位置拆分为左、右两个子数组，递归合并左、右子数组，得到两个有序的合并结果；
  2. 治：将上述两个有序结果调用merge方法合并，得到最终的有序链表；
  3. 终止条件：当子数组的长度为0时返回null，长度为1时直接返回该链表（单个链表本身就是有序的，无需合并）。

简单来说，就是把合并K个链表的问题，转化为两次合并K/2个链表的问题，逐层拆分直至单个链表，再逐层合并。

4、深度剖析：为什么分治归并比顺序合并快很多？

这是本题的核心问题，也是面试中高频追问的考点。答案的关键在于两种解法的时间复杂度存在本质差异：顺序合并为平方级别，分治归并为线性对数级别，当K（链表数）增大时，效率差距会呈指数级扩大。

为了清晰推导，先做统一假设：

  • 设链表数组中有K个链表，每个链表的平均节点数为n；
  • 所有链表的总节点数为N = K * n（N为固定值，题目中N≤10⁴）；
  • 合并两个总节点数为M的链表，时间复杂度为O(M)（由merge子方法的特性决定）。

4.1 顺序合并的时间复杂度推导：O(K²n) = O(KN)

顺序合并是K-1次连续的两个链表合并，每次合并的节点数会逐步累积，我们逐一分析每次合并的时间开销：

  1. 第1次合并：null + 第1个链表 → 节点数n → 时间O(n)；
  2. 第2次合并：n个节点 + 第2个链表n → 节点数2n → 时间O(2n)；
  3. 第3次合并：2n个节点 + 第3个链表n → 节点数3n → 时间O(3n)；
    …
  4. 第K次合并：(K-1)n个节点 + 第K个链表n → 节点数Kn → 时间O(Kn)。

总时间开销为所有次合并的时间之和：O(n + 2n + 3n + ··· + Kn) = O(n × (1+2+3+···+K))
根据等差数列求和公式，1+2+···+K = K×(K+1)/2
忽略常数项后，总时间复杂度为：O(n × K²) = O(K²n)
又因为N=Kn，可转化为O(KN) —— 与链表数K成平方关系，与总节点数N成线性关系。

4.2 分治归并的时间复杂度推导：O(NlogK)

分治归并的核心是逐层合并，分治的递归树有log_2K层，每一层的总合并节点数都是N（所有节点仅参与每一层的一次合并），我们逐层分析：

  1. 分治层级：将K个链表拆分为 K/2 对，每对2个链表（n+n）→ 共 k/2次合并，每层总节点数k/2 × 2n = Kn = N → 每层时间 O(N)；
  2. 递归树的层数：每次将K拆分为 k/2，直至拆分为1个链表，层数为 log_2K（如K=8时，层数为3：8→4→2→1）；
    总时间开销为层数 × 每层时间：O(logK × N) = O(NlogK)

—— 与总节点数N成线性关系，与链表数K成对数关系。

4.3 两种复杂度的直观对比：差距随K增大呈指数级扩大

我们用具体数值举例，直观感受两者的效率差距（设N=10⁴，固定总节点数）：

链表数K 顺序合并O(KN) 分治归并O(NlogK) 效率差距（顺序/分治）
8 8×10⁴=8×10⁴ 10⁴×3=3×10⁴ 2.67倍
16 16×10⁴=1.6×10⁵ 10⁴×4=4×10⁴ 4倍
100 100×10⁴=10⁶ 10⁴×7≈7×10⁴ 14.29倍
10⁴ 10⁴×10⁴=10⁸ 10⁴×14≈1.4×10⁵ 714倍

从表格可清晰看出：K越大，分治归并的优势越明显，当K达到题目提示的10⁴时，分治的效率是顺序的700多倍，这就是分治归并比顺序合并快很多的根本原因。

4.4 空间复杂度对比

  • 顺序合并：O(1)，仅使用有限个指针变量，无额外空间开销；
  • 分治归并：O(logK)，递归调用栈的深度为分治的层数logK，属于常数级别的空间开销（题目中K≤10⁴时，logK≈14，可忽略）。

分治仅用极少量的栈空间，换来了时间复杂度的质的提升，是典型的空间换时间的最优策略。

5、分治归并代码核心点逐行解释

分治归并的代码分为主方法入口、分治递归方法、合并两个链表子方法三部分，其中分治递归方法是核心，以下重点解释关键细节：

5.1 主方法入口：简化递归调用

  • 采用左闭右开区间[0, lists.length)划分链表数组，符合编程中的区间划分习惯，避免边界索引错误；
  • 无需额外判空（递归方法内部会处理空区间），代码更简洁。

5.2 分治递归方法：核心拆分与合并

  • 左闭右开区间：[i, mid)和[mid, j)，保证两个子区间无重叠、无遗漏，覆盖原区间[i, j)；
  • 终止条件：是分治算法的关键，避免无限递归：
    1. m=0：空区间，返回null（适配lists为空的情况）；
    2. m=1：单个链表，直接返回（单个链表本身就是有序的，是最小的解）；
  • 递归拆分：将原问题拆分为两个规模为m/2的子问题，递归求解后合并结果，符合分治的核心思想。

5.3 合并子方法：与顺序合并完全一致

无需额外修改，复用合并两个升序链表的通用方法，保证代码的复用性和一致性。

6、经典示例手动模拟（分治归并过程）

以示例1：lists = [[1,4,5],[1,3,4],[2,6]]（K=3，N=8） 为例，手动模拟分治归并的完整过程，直观理解“分-治-合”的步骤：

6.1 初始化

调用mergeKLists(lists, 0, 3)，m=3，不满足终止条件，mid=0+3/2=1。

6.2 分：递归拆分

  1. 左子问题：mergeKLists(lists, 0, 1) → m=1，返回第一个链表[1,4,5]；
  2. 右子问题：mergeKLists(lists, 1, 3) → m=2，继续拆分：
    • 右左子问题：mergeKLists(lists,1,2) → m=1，返回第二个链表[1,3,4]；
    • 右右子问题：mergeKLists(lists,2,3) → m=1，返回第三个链表[2,6]；
    • 合并右左、右右结果：merge([1,3,4], [2,6]) → 得到[1,2,3,4,6]。

6.3 治：合并最终两个结果

调用merge([1,4,5], [1,2,3,4,6])，合并过程：

  1. 1 vs 1 → 接1 → 指针后移；
  2. 4 vs 1 → 接1 → 指针后移；
  3. 4 vs 2 → 接2 → 指针后移；
  4. 4 vs 3 → 接3 → 指针后移；
  5. 4 vs 4 → 接4 → 指针后移；
  6. 5 vs 6 → 接5 → 指针后移；
  7. 接剩余节点6；
    最终得到合并结果：[1,1,2,3,4,4,5,6]，与示例1一致。

7、拓展思考（面试高频考点）

7.1 分治归并的迭代实现（避免递归栈）

递归实现的分治归并有O(logK)的递归栈空间，若面试官要求O(1)空间的分治实现，可采用迭代版分治，核心是逐层遍历链表数组，两两合并后更新数组，直至数组中只剩一个链表：

  • 时间复杂度仍为O(NlogK)，空间复杂度优化为O(1)，面试中可作为分治解法的拓展回答。

7.2 优先级队列（小根堆）解法（另一种O(NlogK)解法）

除了分治归并，小根堆也是本题的最优解法之一，核心是利用堆的特性维护当前所有链表的头节点，每次取出最小值节点接入结果链表，再将该节点的下一个节点入堆：

  • 时间复杂度O(NlogK)（每个节点入堆/出堆的时间为logK）；
  • 空间复杂度O(K)（堆中最多存储K个节点）；
  • 与分治归并相比，堆解法更适合流式数据场景（如链表动态添加），分治解法更适合静态数组场景。

7.3 三种解法的综合对比

解法 时间复杂度 空间复杂度 优点 缺点 适用场景
顺序合并 O(KN) O(1) 实现最简单 效率极低 K极小的场景（如K<10）
分治归并（递归） O(NlogK) O(logK) 效率高、代码简洁 少量递归栈 面试首选、静态数组
分治归并（迭代） O(NlogK) O(1) 效率高、无栈空间 代码稍复杂 要求O(1)空间的面试场景
优先级队列 O(NlogK) O(K) 适配流式数据 空间开销较大 链表动态添加的场景

8、总结

合并K个升序链表的核心是基于合并两个升序链表的子过程，选择更优的合并策略，本文核心总结如下：

  1. 合并两个升序链表是所有解法的基础，采用哑节点+双指针实现，时间O(m+n)、空间O(1)；
  2. 顺序合并是基础解法，思路为逐个合并累积结果，时间复杂度O(K²n)=O(KN)，效率极低，仅适用于K极小的场景；
  3. 分治归并是最优解法，核心是分而治之、逐层合并，时间复杂度O(NlogK)，空间复杂度O(logK)（递归）/O(1)（迭代）；
  4. 分治比顺序快的根本原因：时间复杂度的量级差异——顺序与K成平方关系，分治与K成对数关系，K越大差距越明显；
  5. 优先级队列是另一种O(NlogK)的解法，适配流式数据场景，可作为面试的拓展回答。

在面试中，分治归并（递归版） 是本题的首选书写版本，代码简洁、逻辑清晰，且能体现对分治思想和归并排序的理解，掌握其时间复杂度的推导过程，能从容应对面试官的追问。
