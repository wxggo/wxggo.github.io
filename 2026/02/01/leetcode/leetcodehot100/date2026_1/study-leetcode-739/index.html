Leetcode-739:每日温度问题-反向遍历跳跃优化&单调栈双解法

Created2026-02-01|Updated2026-02-01|Java学习笔记Leetcode Hot100
|Word Count:2.7k|Reading Time:10mins|Post Views:

1、题目

给定一个整数数组 temperatures，表示每天的温度，返回一个数组 answer，其中 answer[i] 是指对于第 i 天，下一个更高温度出现在几天后。如果气温在这之后都不会升高，请在该位置用 0 来代替。

示例

示例 1：
输入：temperatures = [73,74,75,71,69,72,76,73]
输出：[1,1,4,2,1,1,0,0]
解释：

  • 第0天（73℃）：下一个更高温度在第1天（74℃），间隔1天；
  • 第1天（74℃）：下一个更高温度在第2天（75℃），间隔1天；
  • 第2天（75℃）：下一个更高温度在第6天（76℃），间隔4天；
  • 后续天数以此类推。

示例 2：
输入：temperatures = [30,40,50,60]
输出：[1,1,1,0]

示例 3：
输入：temperatures = [30,20,10]
输出：[0,0,0]

提示

  • 1 <= temperatures.length <= 10⁵
  • 30 <= temperatures[i] <= 100

2、分析

2.1 问题核心

每日温度问题的本质是「找数组中每个元素右侧第一个更大元素的距离」，常规暴力解法（对每个元素向后逐个遍历）时间复杂度为 O(n²)，面对 n=10⁵ 的数据量会超时。

本文提供两种线性时间复杂度（O(n)） 的高效解法：

  • 解法一：反向遍历+结果跳跃优化（复用已计算结果，无额外数据结构）；
  • 解法二：单调栈（利用栈的后进先出特性，一次遍历匹配目标元素）。

两种解法均能适配大数据量，且各有优势：反向遍历代码更简洁，单调栈逻辑更通用。

2.2 解法一：反向遍历+跳跃优化

2.2.1 核心优化点

（1）反向遍历

从数组末尾向前处理：

  • 最后一天的结果必然是 0（无后续天数）；
  • 处理第 i 天时，i+1 到 n-1 的结果已计算完成，可直接复用。
（2）结果跳跃

传统暴力遍历需逐个检查 i+1, i+2, ...，而跳跃优化利用 result[j]（j > i）的结果：

  • 若 temperatures[j] <= temperatures[i]，说明 j 到 j+result[j] 之间的元素都小于等于 temperatures[j]（自然也小于 temperatures[i]），可直接跳转到 j + result[j]，跳过无效区间；
  • 若 result[j] = 0，说明 j 之后无更大温度，i 的结果也为 0。

2.2.2 算法流程

  1. 初始化结果数组（长度与温度数组一致，默认值为 0）；
  2. 从倒数第二天（i = n-2）向前遍历到第0天；
  3. 初始化 j = i+1，循环检查：
    • 若 temperatures[j] > temperatures[i]：result[i] = j - i，跳出循环；
    • 若 result[j] = 0：result[i] = 0，跳出循环；
    • 否则：j += result[j]（跳跃到下一个可能的位置）。

2.3 解法二：单调栈

2.3.1 核心优化点

（1）栈存索引而非值

栈中存储温度数组的索引（而非温度值），既可以通过索引获取温度，又能直接计算距离（当前索引 - 栈顶索引）。

（2）单调递减栈

栈内索引对应的温度值保持单调递减：

  • 遍历到第 i 天时，若当前温度大于栈顶索引对应的温度，说明栈顶索引是「当前温度左侧第一个更小的元素」，即当前温度是栈顶索引的「右侧第一个更大元素」；
  • 弹出栈顶并计算距离，重复此过程直到栈为空或当前温度不大于栈顶温度，再将当前索引压入栈。

2.3.2 算法流程

  1. 初始化结果数组（默认值为 0）和空栈（存储索引，初始栈顶指针 top = -1）；
  2. 从第0天到最后一天遍历数组：
    • 若栈非空且当前温度 > 栈顶索引对应温度：
      • 弹出栈顶索引 j，计算 result[j] = i - j；
      • 重复上述操作直到栈为空或当前温度 ≤ 栈顶索引对应温度；
    • 将当前索引 i 压入栈；
  3. 遍历结束后，未匹配到更大温度的元素结果仍为 0。

3、程序

3.1 完整代码

3.2 代码逐行解释

解法一：反向遍历+跳跃优化

  • length：缓存数组长度，避免多次调用 temperatures.length；
  • result：结果数组，默认值为 0（最后一天无需处理）。
  • 外层循环：反向遍历，从倒数第二天到第0天；
  • 内层循环：j 初始为 i+1，通过 j += result[j] 跳跃，跳过无效区间。
  • 找到更大温度：记录间隔并跳出循环；
  • j 之后无更大温度：i 的结果置0并跳出循环。

解法二：单调栈

  • result：结果数组，默认值为 0；
  • st：数组模拟栈（避免Java内置栈的性能损耗），存储温度数组索引；
  • top：栈顶指针，top=-1 表示空栈。
  • 正向遍历每个天数，记录当前温度 curTemp；
  • 内层循环：栈非空且当前温度更大时，弹出栈顶索引，计算并赋值结果；
  • 压入当前索引：保持栈的单调递减特性。

4、算法性能

解法 时间复杂度 空间复杂度 核心优势 核心缺点
反向遍历+跳跃优化 O(n) O(n) 无额外数据结构、代码简洁 逻辑依赖结果复用，仅适用于「右侧第一个更大元素」
单调栈 O(n) O(n) 逻辑通用（适配左右侧/大小元素）、正向遍历 需要额外栈空间，代码稍繁琐

补充说明

  • 时间复杂度：两种解法每个元素仅被访问1~2次（反向遍历的跳跃、单调栈的入栈/出栈各一次），整体为 O(n)；
  • 空间复杂度：均为 O(n)（结果数组为必要空间，单调栈额外使用 O(n) 栈空间，反向遍历无额外空间）。

5、测试示例

以 temperatures = [73,74,75,71,69,72,76,73] 为例验证两种解法：

5.1 解法一：反向遍历+跳跃优化执行过程

  1. 初始化 result = [0,0,0,0,0,0,0,0]；
  2. i=6（76℃）：j=7（73≤76且result[7]=0）→ result[6]=0；
  3. i=5（72℃）：j=6（76>72）→ result[5]=1；
  4. i=4（69℃）：j=5（72>69）→ result[4]=1；
  5. i=3（71℃）：j=4（69≤71）→ j += 1 = 5（72>71）→ result[3]=2；
  6. i=2（75℃）：j=3（71≤75）→ j +=2=5（72≤75）→ j +=1=6（76>75）→ result[2]=4；
  7. i=1（74℃）：j=2（75>74）→ result[1]=1；
  8. i=0（73℃）：j=1（74>73）→ result[0]=1；
  9. 最终结果：[1,1,4,2,1,1,0,0]。

5.2 解法二：单调栈执行过程

  1. 初始化 result=[0,0,0,0,0,0,0,0]，st=[]，top=-1；
  2. i=0（73）：栈空 → 压入0 → st=[0]；
  3. i=1（74）：74>73 → 弹出0，result[0]=1 → 压入1 → st=[1]；
  4. i=2（75）：75>74 → 弹出1，result[1]=1 → 压入2 → st=[2]；
  5. i=3（71）：71<75 → 压入3 → st=[2,3]；
  6. i=4（69）：69<71 → 压入4 → st=[2,3,4]；
  7. i=5（72）：72>69 → 弹出4，result[4]=1；72>71 → 弹出3，result[3]=2；72<75 → 压入5 → st=[2,5]；
  8. i=6（76）：76>72 → 弹出5，result[5]=1；76>75 → 弹出2，result[2]=4；压入6 → st=[6]；
  9. i=7（73）：73<76 → 压入7 → st=[6,7]；
  10. 最终结果：[1,1,4,2,1,1,0,0]。

6、扩展思考

6.1 解法适配场景

  • 反向遍历+跳跃优化：适合「仅需找右侧第一个更大元素」且追求代码简洁的场景，面试中可快速实现；
  • 单调栈：适合「找左右侧第一个更大/更小元素」的通用场景（如接雨水、柱状图最大面积、下一个更大元素系列问题）。

6.2 边界情况处理

  • 数组长度为1：两种解法均返回 [0]（反向遍历无循环，单调栈仅压入索引无弹出）；
  • 所有元素递减：结果数组全为 0（反向遍历每次触发 result[j]=0，单调栈无弹出操作）；
  • 存在连续相同温度：反向遍历通过跳跃跳过无效区间，单调栈仅在温度严格更大时弹出，结果正确。

6.3 优化方向

  • 反向遍历：若题目允许修改原数组，可将结果存储在原数组（覆盖无用温度值），空间复杂度降至 O(1)；
  • 单调栈：使用Java内置 Deque（如 LinkedList）替代数组模拟栈，代码更易读（但性能略低）。
