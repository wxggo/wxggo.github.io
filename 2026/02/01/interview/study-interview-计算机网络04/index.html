1、前言

“请说一下GET和POST的区别”、”HTTPS是如何保证安全的”、”HTTP/1.1到HTTP/3做了哪些优化”——这是计算机网络面试中的送分题也是送命题。看似基础，但能否说出RFC规范定义与实际实现的差异、能否讲清底层加密原理、能否对比版本演进的核心痛点，直接决定了面试的评价档次。

本文基于小林coding的HTTP面试题梳理，从HTTP基础概念、方法对比、缓存机制、HTTPS安全、协议演进五大维度，系统拆解HTTP面试八股文的核心要点，标注高频考点与易错陷阱，助你面试中从容应对。

2、核心知识体系总览

HTTP面试知识体系可归纳为**”基础概念→方法语义→缓存策略→安全传输→版本演进”五大模块，核心考察点在于协议规范的严谨性与工程实践的差异性**。

3、分模块详解（含面试高频考点）

3.1 HTTP基础：状态码与语义

核心工作流程

HTTP状态码由3位数字构成，按首位数字分为5大类，面试需掌握常见状态码的精确含义及使用场景：

1xx：提示信息（中间状态）

  • 100 Continue：服务器已收到请求头，客户端可继续发送请求体（用于优化大文件上传）。

2xx：成功处理

  • 200 OK：通用成功状态，响应体包含请求数据。
  • 204 No Content：成功但无响应体（如DELETE请求删除成功）。
  • 206 Partial Content：断点续传/分块下载成功，响应体为部分内容（范围请求）。

3xx：重定向与缓存

  • 301 Moved Permanently：永久重定向，资源已移至新URL，搜索引擎会更新索引。
  • 302 Found：临时重定向，资源仍存在但临时用其他URL访问，搜索引擎不更新索引。
  • 304 Not Modified：协商缓存命中，资源未修改，客户端使用本地缓存（无响应体）。
  • 307 Temporary Redirect：临时重定向（严格版），要求保持原请求方法（POST不能变GET）。

4xx：客户端错误

  • 400 Bad Request：请求报文语法错误（参数缺失/格式错误）。
  • 401 Unauthorized：未认证（需登录）。
  • 403 Forbidden：禁止访问（权限不足，与认证无关）。
  • 404 Not Found：资源不存在。

5xx：服务端错误

  • 500 Internal Server Error：服务器内部错误（代码异常）。
  • 502 Bad Gateway：网关/代理从上游服务器收到无效响应。
  • 503 Service Unavailable：服务器暂时无法处理（超载或维护）。

关键考点解析

  • 301 vs 302 vs 307：301永久迁移（SEO权重转移），302临时迁移（但会导致POST变GET的歧义），307严格保持方法不变（POST仍POST）。
  • 304触发条件：客户端发送If-Modified-Since（时间）或If-None-Match（ETag），服务端校验后返回304，无响应体，节省带宽。

面试考点

  • 问：301和302的区别？什么时候用307？
  • 答：301永久重定向（域名变更），302临时重定向；302存在POST重定向后变为GET的问题（非幂等性引发），307强制要求保持原请求方法，避免该问题。
  • 问：304状态码的含义？触发条件是什么？
  • 答：304表示资源未修改，使用缓存；触发条件是客户端发送协商缓存验证头（If-Modified-Since/If-None-Match），服务端校验无变化后返回。

3.2 请求方法：GET与POST的本质（RFC规范 vs 现实）

核心工作流程

RFC 7231等规范对HTTP方法的语义有明确定义，这与实际开发中的滥用存在差异：

RFC规范定义

  • GET：安全且幂等，用于获取资源，不应产生副作用（不修改服务器状态）。
  • POST：不安全且不幂等，用于提交数据，可能创建新资源或触发处理操作。
  • PUT：幂等，用于全量更新资源。
  • DELETE：幂等，用于删除资源。

关键概念界定

  • 安全（Safe）：请求方法不会”破坏”服务器资源（只读）。
  • 幂等（Idempotent）：多次执行相同操作，结果相同（无副作用或副作用一致）。

关键考点解析

  • GET与POST的核心差异（RFC层面）：

    1. 语义：GET是获取，POST是提交；
    2. 幂等性：GET幂等（多次获取结果一致），POST不幂等（多次提交创建多条记录）；
    3. 缓存性：GET可缓存（浏览器、代理均可），POST不可缓存；
    4. 书签：GET可保存为书签，POST不可。

  • **现实实现的”越界”**：

    • GET带Body：RFC未禁止，某些场景（如Elasticsearch查询）会用GET+Body，但浏览器/代理可能不支持，不推荐。
    • POST查询：实际开发中常用POST做查询（参数多放Body避免URL过长），此时POST是安全的，但仍不可缓存。

  • 安全性误区：

    • GET参数在URL中 exposed，POST在Body中，但两者都是明文传输（HTTPS下均加密）。
    • 真正安全靠HTTPS，而非GET/POST之分。

面试考点

  • 问：GET和POST的区别？从RFC规范和实际实现两个层面回答。
  • 答：RFC层面：GET安全幂等（获取资源、可缓存），POST不安全不幂等（提交数据、不可缓存）；实际层面：GET可用Body但兼容性差，POST也可用于查询但失去缓存能力。安全上两者皆明文，需HTTPS保障。
  • 问：什么是幂等性？哪些HTTP方法是幂等的？
  • 答：幂等性指多次执行相同操作结果一致；GET、PUT、DELETE是幂等的，POST不幂等（多次提交创建多个资源）。
  • 问：GET请求可以带Body吗？POST可以当作GET用吗？
  • 答：RFC未禁止GET带Body，但协议语义和浏览器实现不支持；POST可以查询（参数放Body），但违背语义且不可缓存。

3.3 HTTP缓存：强缓存与协商缓存（性能优化核心）

核心工作流程

HTTP缓存是减少网络传输、提升性能的关键机制，分为强缓存（不发请求）和协商缓存（发请求但无响应体）两级：

强缓存（本地缓存生效）

  • Cache-Control（HTTP/1.1）：max-age=3600（秒），优先级高于Expires。
    • no-cache：跳过强缓存，强制协商缓存（发请求）。
    • no-store：禁止缓存（不存不取）。
    • private：仅客户端缓存，public：客户端+代理均可缓存。
  • Expires（HTTP/1.0）：绝对过期时间（GMT），受本地时间影响可能不准，已被Cache-Control取代。

协商缓存（服务器验证）

  • Last-Modified + If-Modified-Since：最后修改时间（秒级精度），可能因频繁修改（1秒内多次）导致缓存失效。
  • ETag + If-None-Match：资源唯一标识符（优先级高于Last-Modified），基于内容生成（如MD5），精度更高。

缓存决策流程

  1. 检查Cache-Control/Expires，未过期 → 强缓存（200 from cache）。
  2. 已过期 → 携带If-Modified-Since/If-None-Match发送请求。
  3. 服务端校验，无变化 → 304 Not Modified（使用缓存）。
  4. 有变化 → 200，返回新资源并更新缓存标识。

关键考点解析

  • 缓存优先级：Cache-Control > Expires，ETag > Last-Modified。
  • 200 vs 304：200（from cache/disk）是强缓存命中，304是协商缓存命中（发了请求但无响应体）。
  • 刷新按钮 vs 强制刷新：
    • 普通刷新（F5）：可能携带Cache-Control: max-age=0强制协商缓存。
    • 强制刷新（Ctrl+F5）：携带Cache-Control: no-cache和Pragma: no-cache，跳过所有缓存。

面试考点

  • 问：HTTP缓存机制是怎样的？强缓存和协商缓存的区别？
  • 答：强缓存通过Cache-Control/Expires判断，有效期内直接使用本地缓存（200）；协商缓存通过Last-Modified/ETag向服务器验证，返回304则使用缓存，200则更新。
  • 问：为什么有了Last-Modified还需要ETag？
  • 答：Last-Modified秒级精度无法识别1秒内多次修改；且文件修改但内容未变（touch）会误判，ETag基于内容生成更准确。
  • 问：Ctrl+F5刷新和F5刷新的区别？
  • 答：F5可能触发协商缓存，Ctrl+F5强制禁用所有缓存（携带no-cache头）。

3.4 HTTPS：安全传输原理（TLS握手详解）

核心工作流程

HTTPS = HTTP + SSL/TLS（安全层），解决HTTP的窃听、篡改、冒充三大风险：

SSL/TLS（位于传输层与应用层之间）

  • 握手阶段（2 RTT，TLS 1.2）：协商加密算法、交换密钥、验证身份。
  • 加密传输阶段：使用协商密钥对称加密通信。

TLS 1.2 握手详细流程（4次握手）

  1. ClientHello：客户端发送支持的TLS版本、加密套件列表、Client Random（随机数）。
  2. ServerHello：服务端确认TLS版本、加密套件、Server Random（随机数），并发送数字证书（含公钥）。
  3. 客户端验证与密钥交换：
    • 验证证书（CA链、有效期、域名匹配）。
    • 取出服务器公钥，加密生成Pre-master Random（第三个随机数，仅客户端和服务器知晓）。
    • 发送加密通信变更通知（后续用会话密钥加密）。
  4. Server Finish：服务端用私钥解密Pre-master，计算会话密钥（Client+Server+Pre-master通过PRF生成），发送握手结束通知。

加密机制

  • 混合加密：
    • 非对称加密（RSA/ECDHE）：仅用于握手阶段交换密钥，解决”密钥配送问题”。
    • 对称加密（AES）：用于后续数据传输，性能比非对称高3个数量级。
  • 摘要算法（SHA-256）：生成消息指纹，验证数据完整性（防篡改）。
  • 数字证书（CA签发）：解决身份冒充问题，包含公钥、域名、有效期、CA签名。

关键考点解析

  • 三个随机数作用：Client Random + Server Random（明文传输）+ Pre-master Random（非对称加密传输），三者共同生成会话密钥，防止重放攻击和前向保密（即使私钥泄露，历史会话无法解密）。
  • 证书验证链：客户端内置根CA证书 → 验证服务器证书签名 → 验证中间CA → 信任链完整则通过。
  • TLS 1.3优化：握手简化至1 RTT（甚至0 RTT重连），废弃RSA密钥交换（不支持前向保密），仅支持ECDHE。

面试考点

  • 问：HTTPS解决了HTTP的哪些问题？如何解决？
  • 答：解决窃听（混合加密）、篡改（摘要算法哈希校验）、冒充（数字证书CA验证）。
  • 问：HTTPS握手过程中涉及哪些加密技术？为什么用混合加密？
  • 答：非对称加密交换密钥，对称加密传输数据；非对称加密计算量大（慢3个数量级），对称加密快但密钥配送困难，混合加密兼顾安全与性能。
  • 问：TLS握手为什么需要三次随机数？两次可以吗？
  • 答：Client和Server随机数加上Pre-master生成会话密钥，确保随机性防止预测；若仅用两个明文随机数，可能被中间人计算，Pre-master通过非对称加密保护，保证只有通信双方知晓。
  • 问：证书验证失败可能是什么原因？
  • 答：证书过期、域名不匹配、签发CA不受信任（自签证书）、证书链不完整。

3.5 HTTP演进：1.1 vs 2 vs 3（性能优化史）

核心工作流程

HTTP版本演进主线是解决队头阻塞、提升连接利用率、降低延迟：

HTTP/1.1（1997）改进

  • 持久连接（Keep-Alive）：默认Connection: keep-alive，单个TCP连接可复用多个请求，减少三次握手开销。
  • 管道化（Pipelining）：允许客户端未收到响应前连续发送多个请求，但服务器必须按序响应，仍存在队头阻塞（Head-of-Line Blocking）。
  • 分块传输（Chunked）、断点续传（Range请求）、Host头部（支持虚拟主机）。

HTTP/1.1痛点

  • 请求/响应队头阻塞：前一个请求阻塞，后续请求必须等待（即使管道化，响应仍需按序）。
  • TCP连接数限制：浏览器同域名仅6-8个并发连接，需域名分片（Sharding）突破。
  • 头部冗余：每次请求都携带完整Header（Cookie等），未压缩。

HTTP/2（2015）革命性改进

  • 二进制分帧层：HTTP消息拆分为帧（Frame），采用二进制编码（非文本），更高效且利于流控制。
  • 多路复用（Multiplexing）：单一TCP连接上并行传输多个Stream（请求/响应），帧可乱序发送，按Stream ID重组，彻底解决HTTP层队头阻塞。
  • 头部压缩（HPACK）：客户端和服务端维护静态/动态表，重复Header字段用索引标识，减少传输（如Cookie从数百B降至几B）。
  • 服务器推送（Push）：服务端可主动向客户端推送资源（如HTML中引用的CSS/JS），无需等待请求。

HTTP/2痛点（TCP层队头阻塞）

  • TCP丢包阻塞：HTTP/2虽然解决了HTTP层队头阻塞，但底层仍是TCP，单个TCP连接内任一丢包将阻塞所有Stream（TCP保证有序，丢包需重传，后续数据等待）。

HTTP/3（草案）基于QUIC

  • 传输层改用UDP：基于QUIC协议（Quick UDP Internet Connections），在应用层实现可靠传输。
  • 无队头阻塞：QUIC内部多个独立Stream，单个Stream丢包仅阻塞该Stream，不影响其他。
  • 0-RTT/1-RTT握手：首次连接1-RTT（类似TLS 1.3），重连0-RTT（携带之前会话密钥参数）。
  • 连接迁移：基于Connection ID而非四元组（IP+Port），网络切换（WiFi切4G）无需重连。

关键考点解析

  • 队头阻塞对比：

    • HTTP/1.1：应用层队头阻塞（请求/响应串行）。
    • HTTP/2：TCP层队头阻塞（多路复用但TCP单连接丢包阻塞）。
    • HTTP/3：无队头阻塞（QUIC基于UDP，Stream独立）。

  • HTTP/2多路复用局限：虽然单连接多Stream，但TCP层面仍是单连接，丢包重传阻塞所有Stream，高丢包环境下性能可能不如HTTP/1.1多连接。

面试考点

  • 问：HTTP/1.1 Keep-Alive和HTTP/2多路复用的区别？
  • 答：Keep-Alive是串行复用（请求-响应依次），多路复用是并行复用（帧可乱序，Stream独立）。
  • 问：HTTP/2解决了HTTP队头阻塞，为什么还说它有队头阻塞问题？
  • 答：HTTP/2解决的是HTTP协议层的队头阻塞（请求并发），但底层TCP协议存在队头阻塞（丢包重传阻塞整个连接），HTTP/3通过QUIC解决TCP层队头阻塞。
  • 问：HTTP/3为什么基于UDP？QUIC如何保证可靠性？
  • 答：UDP无连接状态灵活，QUIC在应用层实现类似TCP的拥塞控制、重传机制、有序传输，且支持Stream独立（单丢包不阻塞其他Stream）。
  • 问：HTTP/2的头部压缩HPACK原理？
  • 答：客户端和服务端维护静态表（预定义常见Header）和动态表（会话中累积的Header），发送时用索引代替字符串，减少冗余传输。

5、面试高频考点汇总（直接背诵）

5.1 核心问答（必背）

Q1：HTTP和HTTPS的区别？

  • 端口：HTTP 80，HTTPS 443。
  • 安全性：HTTP明文传输，HTTPS通过SSL/TLS加密（混合加密+摘要算法+数字证书）。
  • 性能：HTTPS握手阶段增加RTT（TLS握手），且加密解密消耗CPU资源，但现代硬件影响较小。
  • 证书：HTTPS需向CA申请数字证书，HTTP无需。

Q2：HTTPS握手（TLS 1.2）过程？为什么用三个随机数？

  • 过程：ClientHello（版本、算法、Client Random）→ ServerHello（确认、Server Random、证书）→ 客户端验证+Pre-master Random → Server Finish生成会话密钥。
  • 三随机数：Client Random（客户端）+ Server Random（服务端）+ Pre-master Random（加密传输），确保密钥随机性且仅双方知晓，防止重放攻击。

Q3：POST和GET能否互换使用？有什么问题？

  • 技术上：可以互换（GET带Body，POST当GET用），但违背语义。
  • 问题：GET用于修改数据不安全（误触、爬虫、预加载会触发副作用）；POST用于查询不可缓存（浪费带宽），且URL长度限制导致参数受限（若放URL）。

Q4：什么是幂等性？实际开发中如何保证接口幂等？

  • 定义：同一操作执行多次结果一致（无副作用或副作用相同）。
  • 实现：数据库唯一索引（防重复插入）、Token机制（防重复提交）、乐观锁（版本号）、分布式锁（Redis/ZooKeeper）。

Q5：浏览器输入URL到显示页面发生了什么？（HTTP层面）

  1. DNS解析：域名→IP（递归查询+缓存）。
  2. 建立连接：TCP三次握手（HTTP/1.1）或QUIC握手（HTTP/3）。
  3. TLS握手（HTTPS）：交换密钥、验证证书（1-2 RTT）。
  4. 发送请求：构建HTTP报文（方法、URL、Headers）。
  5. 服务器处理：路由、业务逻辑、数据库查询。
  6. 返回响应：状态码、响应头、HTML/JSON。
  7. 渲染页面：解析DOM、CSSOM，执行JS（前端范畴）。

5.2 易错点提醒（避坑）

  1. GET vs POST安全性：两者都是明文（HTTP下），POST的Body也能被抓包，真正安全靠HTTPS。
  2. 304不是重定向：304是”未修改”使用缓存，虽带3字头但无Location头，不算跳转。
  3. 301 vs 302的SEO：301会转移权重（旧URL权重给新URL），302不会（临时跳转保留旧URL权重）。
  4. HTTP/2的局限：虽然支持多路复用，但TCP层队头阻塞仍存在（单连接丢包阻塞所有Stream）。
  5. TLS与SSL叫法：面试中说SSL/TLS均可，但技术细节应提TLS 1.2/1.3（SSL 3.0已废弃）。
  6. Cookie与Session：Cookie存放于客户端（4KB限制），Session存放于服务端（依赖Cookie中的SessionID或URL重写）。

6、总结

HTTP面试八股文的掌握程度，直接反映候选人对Web基础协议的理解深度。核心复习路径建议：

  1. 先抓大放小：死磕GET/POST本质、HTTPS握手、缓存机制（强缓存/协商缓存）、HTTP/2特性，这四块占面试80%分值。
  2. 再抠细节：状态码的精确含义（301/302/307差异）、幂等性实现、HPACK原理、QUIC优势。
  3. 结合实践：不仅要知道RFC规范，还要说明实际开发中的trade-off（如POST查询虽违背语义但解决URL长度限制，GET带Body虽可行但兼容性差）。
