Leetcode-169:多数元素-摩尔投票法极简实现

Created2026-02-06|Updated2026-02-06|Java学习笔记Leetcode Hot100
|Word Count:1.5k|Reading Time:5mins|Post Views:

1、题目

给定一个大小为 n 的数组 nums ，返回其中的多数元素。多数元素是指在数组中出现次数 大于 n/2 的元素。

你可以假设数组是非空的，并且给定的数组总是存在多数元素。

示例

示例 1：
输入：nums = [3,2,3]
输出：3

示例 2：
输入：nums = [2,2,1,1,1,2,2]
输出：2

提示

  • n == nums.length
  • 1 <= n <= 5 * 10^4
  • -10^9 <= nums[i] <= 10^9

2、分析

2.1 问题核心

找多数元素的核心是利用“出现次数超过一半”的特性，在不额外占用大量空间的前提下高效定位目标元素。常规思路（哈希表统计次数、排序取中间值）虽能解决问题，但要么空间复杂度高，要么时间复杂度非最优。

摩尔投票法（Boyer-Moore 投票算法） 是该题的最优解：基于“多数元素的出现次数超过一半，其投票数无法被其他元素完全抵消”的核心思想，通过一次遍历完成投票抵消，最终剩下的“擂主”即为多数元素。该方法无需额外空间，时间复杂度为线性级别，是极致的性能最优解。

2.2 核心思路

摩尔投票法的核心是“比武投票，同阵营加血，不同阵营扣血，血条为空换擂主”：

  1. 初始化状态：定义ans记录当前擂主（候选多数元素），count记录擂主的“生命值”（投票数），初始ans=0、count=0；
  2. 遍历投票：逐个遍历数组元素：
    • 若count=0（当前无擂主/擂主血条为空）：将当前元素设为新擂主ans，并将生命值置为1；
    • 若count≠0（已有擂主）：对比当前元素与擂主：
      • 相同（同阵营）：擂主生命值+1；
      • 不同（不同阵营）：擂主生命值-1；
  3. 结果确定：由于多数元素出现次数超过一半，遍历结束后，最终的擂主ans必然是多数元素（其他元素的投票数无法完全抵消它）。

3.2 代码简洁解释

  • 状态初始化：ans初始无意义（因count=0），count为0表示当前无有效擂主；
  • 血条为空逻辑：count==0时更换擂主，保证每次更换的擂主都是当前未被抵消的候选元素；
  • 投票抵消逻辑：相同元素增加生命值，不同元素减少生命值，利用“多数元素生命值无法被完全抵消”的特性锁定目标；
  • 结果返回：遍历结束后无需额外验证（题目保证存在多数元素），直接返回最终擂主ans。

4、算法性能

  • 时间复杂度：O(n)（n为数组长度）。仅需一次线性遍历，每个元素仅做一次判断和数值操作，无重复计算；
  • 空间复杂度：O(1)。仅使用了ans和count两个变量，无额外数组、哈希表等空间开销，属于极致的原地操作；
  • 核心优势：
    1. 相比哈希表法（O(n)空间复杂度），摩尔投票法完全无额外空间占用；
    2. 相比排序法（O(n log n)时间复杂度），线性遍历的效率更高；
    3. 代码极简，仅几行核心逻辑，易理解、易书写、易调试。

5、测试示例

示例2：nums = [2,2,1,1,1,2,2]（长度7，多数元素2出现4次）

  1. 初始化：ans=0，count=0；
  2. 遍历num=2：count=0 → ans=2，count=1；
  3. 遍历num=2：num==ans → count=2；
  4. 遍历num=1：num≠ans → count=1；
  5. 遍历num=1：num≠ans → count=0；
  6. 遍历num=1：count=0 → ans=1，count=1；
  7. 遍历num=2：num≠ans → count=0；
  8. 遍历num=2：count=0 → ans=2，count=1；
  9. 遍历结束，返回ans=2（与示例结果一致）。

示例1：nums = [3,2,3]（长度3，多数元素3出现2次）

  1. 初始化：ans=0，count=0；
  2. 遍历num=3：count=0 → ans=3，count=1；
  3. 遍历num=2：num≠ans → count=0；
  4. 遍历num=3：count=0 → ans=3，count=1；
  5. 遍历结束，返回ans=3（与示例结果一致）。

6、扩展思考

6.1 其他解法对比

解法 时间复杂度 空间复杂度 核心缺点
摩尔投票法 O(n) O(1) 仅适用于“多数元素占比超1/2”场景
哈希表统计 O(n) O(n) 额外占用哈希表空间
排序取中值 O(n log n) O(1)/O(n) 排序耗时，效率低于线性遍历

6.2 摩尔投票法适用场景

该算法仅适用于“存在绝对多数元素（出现次数>n/2）”的场景：

  • 若题目不保证存在多数元素，遍历结束后需额外遍历一次数组，验证ans的出现次数是否超过n/2；
  • 可扩展至“找出现次数>n/3的元素”（需维护两个擂主），是处理“绝对多数元素”类问题的通用思路。

6.3 边界情况处理

代码天然适配所有边界场景，无需额外判断：

  • 数组长度为1：直接返回唯一元素（count=0→设为擂主，遍历结束返回）；
  • 多数元素连续出现：生命值持续增加，不会被替换；
  • 多数元素分散出现：其他元素的抵消无法耗尽其生命值，最终仍会成为擂主。
