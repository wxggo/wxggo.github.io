Leetcode-011:盛最多水的容器-双指针贪心最优解

Created2026-02-02|Updated2026-02-02|Java学习笔记Leetcode Hot100
|Word Count:2.8k|Reading Time:9mins|Post Views:

1、题目

给定一个长度为 n 的整数数组 height。有 n 条垂线，第 i 条线的两个端点是 (i, 0) 和 (i, height[i])。
请你找出其中的两条线，使得它们与 x 轴共同构成的容器可以容纳最多的水。返回容器可以储存的最大水量。

说明：你不能倾斜容器，且 n 至少为 2。

示例

示例 1：
输入：height = [1,8,6,2,5,4,8,3,7]
输出：49
解释：选取下标1（高度8）和下标8（高度7）的垂线，容器宽度为7，高度取最小值7，面积=7×7=49，为最大水量。

示例 2：
输入：height = [1,1]
输出：1
解释：仅有的两条垂线，宽度1，高度1，面积=1×1=1。

示例 3：
输入：height = [4,3,2,1,4]
输出：16
解释：选取下标0（高度4）和下标4（高度4）的垂线，宽度4，高度4，面积=4×4=16。

提示

  • 2 <= height.length <= 10^5
  • 0 <= height[i] <= 10^4

2、分析

2.1 问题核心

盛最多水的容器问题的核心遵循木桶短板效应：容器的盛水高度由两条垂线中更短的那条决定，盛水面积公式为：
面积 = 两条垂线的最小高度 × 两条垂线的水平间距
即 area = Math.min(height[left], height[right]) × (right - left)。

问题的关键是找到一对垂线，让上述公式的计算结果最大。暴力解法需枚举所有垂线对，时间复杂度为 O(n²)，面对 10^5 级别的数据会超时；双指针+贪心 是最优解，通过从两端向中间移动指针，仅一次遍历即可找到最大面积，时间复杂度降至 O(n)。

2.2 核心难点：为什么要移动「短板指针」而非长板指针？

这是整个解法的核心，也是面试考察的重点，底层逻辑是贪心决策：移动短板指针才有可能让容器的盛水面积增大，移动长板指针则面积必然减小或不变。
具体推导：

  1. 设当前左指针left、右指针right，且height[left] < height[right]（左为短板，右为长板），当前面积为min(L,R)×W = L×W（W为间距）；
  2. 若移动长板指针（right–）：新的间距W’ = W-1，新的最小高度min(L, R') 要么等于L（R’≥L），要么小于L（R’<L），因此新面积≤ L×(W-1) < 原面积，面积必然减小；
  3. 若移动短板指针（left++）：新的间距W’ = W-1，但新的最小高度min(L', R) 可能大于L（L’>L），此时新面积可能 > 原面积，存在增大的可能性。

总结：短板是限制面积的核心因素，只有移动短板指针，才有可能找到更长的短板，从而突破当前面积的限制；移动长板指针只会让宽度减小，短板不变或更短，面积不可能增大。

2.3 核心思路

  1. 双指针初始化：左指针left指向数组起始位置（0），右指针right指向数组末尾位置（height.length-1），初始最大面积ans=0；
  2. 双指针遍历：当left < right时，循环执行以下步骤：
    • 计算当前指针组合的容器面积，利用短板效应计算高度；
    • 比较当前面积与ans，更新ans为较大值；
    • 贪心移动指针：移动高度更小的那个指针（短板指针），若两指针高度相等，移动任意一个均可；
  3. 返回结果：遍历结束后，ans即为最大盛水面积。

该思路通过贪心决策避免了无效的枚举，仅一次遍历即可覆盖所有可能的最优解情况，是线性时间复杂度的最优解。

3.2 代码核心点逐行解释

（1）双指针初始化

  • 左右指针分别从两端开始，覆盖最大的初始水平间距，符合「间距越大，面积越有可能更大」的直观认知；
  • 最大面积ans初始为0，逐步更新为遍历过程中的最大值。

（2）面积计算（核心公式）

  • 严格遵循木桶短板效应，用Math.min取两条垂线的最小高度作为容器有效高度；
  • 用right - left计算两条垂线的水平间距，即容器的宽度；
  • 两者相乘得到当前指针组合的实际盛水面积，无冗余计算。

（3）贪心指针移动（面试必考）

  • 核心逻辑：移动短板指针，仅这一步就将时间复杂度从O(n²)降至O(n)；
  • 特殊情况：当height[left] == height[right]时，移动左或右指针均可，因为后续无论移动哪个，新的面积都不会超过当前面积（宽度减小，短板不变）；
  • 注释中关键结论：height[left]作为短板，与右边任意垂线的组合，都无法形成比当前更大的面积，因此可以直接舍弃该短板，右移左指针。

（4）循环终止条件

  • 当left == right时，两条垂线重合，水平间距为0，面积为0，无计算意义，循环终止；
  • 确保所有有效的垂线对都被遍历（无需遍历left > right的情况，为重复组合）。

4、算法性能

  • 时间复杂度：O(n)（n为数组长度）。仅一次遍历数组，左右指针合计移动n次，每次循环仅做常数次计算（面积计算、指针移动）；
  • 空间复杂度：O(1)。仅使用了left、right、ans、area四个变量，无额外数据结构，属于原地操作，空间利用率拉满；
  • 核心优势（面试重点）：
    1. 线性时间复杂度，完美适配题目10^5级别的数据规模，无超时风险；
    2. 空间复杂度为常数级，远优于其他解法；
    3. 代码极简，逻辑清晰，面试中可快速书写，不易出错；
    4. 贪心决策思想典型，可迁移至同类数组双指针问题。

5、测试示例

以经典示例：height = [1,8,6,2,5,4,8,3,7] 为例，完整模拟双指针执行过程，直观理解贪心逻辑：

  1. 初始化：left=0(1)、right=8(7)、ans=0；
  2. 第一次循环：area=min(1,7)×8=8 → ans=8，左为短板→left=1(8)；
  3. 第二次循环：area=min(8,7)×7=49 → ans=49，右为短板→right=7(3)；
  4. 第三次循环：area=min(8,3)×6=18 → ans不变，右为短板→right=6(8)；
  5. 第四次循环：area=min(8,8)×5=40 → ans不变，等高移右→right=5(4)；
  6. 第五次循环：area=min(8,4)×4=16 → ans不变，右为短板→right=4(5)；
  7. 第六次循环：area=min(8,5)×3=15 → ans不变，右为短板→right=3(2)；
  8. 第七次循环：area=min(8,2)×2=4 → ans不变，右为短板→right=2(6)；
  9. 第八次循环：area=min(8,6)×1=6 → ans不变，右为短板→right=1(8)；
  10. 循环终止：left=1 == right=1，退出循环；
  11. 返回结果：ans=49，与示例一致。

关键观察：第二次循环得到最大面积49后，后续所有指针移动的面积均小于该值，贪心决策成功跳过了所有无效的垂线组合，无需枚举全部n²种可能。

6、扩展思考

6.1 暴力解法的局限性

暴力解法通过双层循环枚举所有(left, right)组合，计算每对的面积并更新最大值，代码如下：

  • 时间复杂度O(n²)，当n=10^5时，计算量达到10^10，远超程序运行时间限制，必然超时；
  • 空间复杂度O(1)，虽空间最优，但时间效率过低，无工程实用价值。

6.2 边界情况天然适配

代码无需额外处理边界，天然支持所有题目规定的场景：

  • 数组长度为2：仅一次循环计算面积，直接返回结果；
  • 数组元素全递增/递减：双指针会逐步移动至最优解位置；
  • 存在多个相同高度的短板：移动任意一个短板指针均可，不影响最终结果；
  • 元素高度为0：短板为0时面积为0，指针会自动移动，跳过无效组合。

6.3 贪心思想的延伸

本题的贪心决策属于局部最优推全局最优：每次移动短板指针，都是做出当前步骤的最优选择（放弃不可能更优的短板），最终通过一系列局部最优选择，得到全局最优解。
该思想在双指针问题中应用广泛，如「三数之和」「最接近的三数之和」等，掌握后可快速解决同类问题。

6.4 为什么两板等高时可移动任意指针？

当height[left] == height[right]时，无论移动左还是右指针，新的面积都不可能超过当前面积：

  • 新间距W' = W-1，比当前小1；
  • 新的最小高度min(L', R)或min(L, R') 最大为原高度L（R），因此新面积≤ L×(W-1) < L×W = 当前面积；
  • 因此移动任意一个指针均可，不影响最终的最大面积结果。

总结

盛最多水的容器问题的双指针+贪心解法，是数组问题中「时间最优、实现最简」的经典解法，其核心在于抓住木桶短板效应和贪心决策的底层逻辑：移动短板指针才有可能增大面积，移动长板指针则面积必然减小。

本文从问题核心、贪心逻辑、代码实现、执行过程四个维度，全面拆解了该解法，重点剖析了面试中必考的「指针移动原因」。该解法仅用数行代码实现了线性时间复杂度，且空间复杂度为常数级，是面试中解决该问题的推荐解法。
