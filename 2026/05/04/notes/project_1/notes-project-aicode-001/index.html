Notes-项目复盘:基于LangChain4j的AI编程小助手学习与复现

发表于2026-05-04|更新于2026-05-04|项目回顾
|字数总计:6.4k|阅读时长:22分钟|阅读量:

本文是对 AI 编程小助手 - LangChain4j 实战项目 的一次学习复现整理。这个项目的目标不是简单地调用一次大模型 API，而是把大模型接入、Prompt 设计、会话记忆、流式响应、RAG 检索增强、Tool Calling 工具调用、MCP 工具扩展、安全护轨和前端聊天界面串成一个相对完整的 AI 应用。
我在复现过程中主要关注后端 Agent 服务的组织方式、LangChain4j 的工程化使用、前后端流式通信、RAG 和工具调用的接入方式，以及这类“代码开发助手”项目和普通聊天机器人的区别。

一、前言

最近在学习 AI 应用开发时，我明显感觉到一个变化：
以前做项目更多是围绕 CRUD、权限、数据库、接口设计展开，而现在很多项目开始围绕“大模型能力如何接入业务系统”展开。

这次复现的项目是一个 AI 编程小助手。从表面上看，它像是一个面向程序员的聊天机器人，可以回答学习路线、求职面试、代码问题等内容；但真正把代码跑起来并阅读后端实现后，我发现它并不是一个简单的 Controller 调大模型接口 的 Demo，而是更接近一个小型 Agent 应用原型。

它大致串起了下面这些能力：

我这次复现主要想解决几个问题：

  1. 如何用 Java 技术栈开发一个 AI 对话应用；
  2. LangChain4j 在项目中到底承担什么角色；
  3. AI Service、ChatModel、StreamingChatModel、ChatMemory 之间如何协作；
  4. RAG、工具调用、MCP、安全护轨分别解决什么问题；
  5. 为什么 Agent 代码开发助手不能只停留在“问答机器人”层面；
  6. 如果后续自己扩展一个代码开发助手，还可以从哪些方向继续做。

二、项目目标与整体方案

2.1 项目定位

这个项目的定位是一个面向程序员的 AI 编程学习与求职辅导助手。它主要覆盖三类场景：

场景 说明
编程学习导师 根据用户问题给出学习路线、技术选型建议和学习资料方向
求职面试助手 辅助准备简历、面试题、项目亮点和八股文
代码答疑专家 解答编程技术问题，给出代码示例和排查思路

如果只从产品形态来看，它像一个聊天应用；但从技术实现来看，它实际上是一个 基于大模型的 Agent 应用雏形。

它不仅让模型回答问题，还给模型配置了：

  1. 系统提示词；
  2. 会话记忆；
  3. 结构化输出；
  4. RAG 本地知识库检索；
  5. 面试题搜索工具；
  6. MCP 联网搜索工具；
  7. 输入安全护轨；
  8. 流式输出能力；
  9. 前端 Markdown 渲染能力。

这里的两个参数可以理解为：

参数 说明
1000 每个文本片段最大字符数
200 相邻片段之间的重叠字符数

重叠的意义是避免切分时把上下文截断。
比如一个知识点的前半句在上一个片段，后半句在下一个片段，如果没有重叠，模型可能拿不到完整语义。

参数 作用
maxResults(5) 最多取 5 个相关片段
minScore(0.75) 过滤掉相关度较低的片段

复现时我遇到过一个问题：
如果 minScore 设置太高，模型经常检索不到内容；如果设置太低，又会把不太相关的片段带进去，导致回答跑偏。

所以 RAG 并不是“接上向量库就结束”，检索参数、文档质量、切分方式都会影响最终效果。

十二、个人收获

这次复现对我最大的帮助，是让我把“调用大模型 API”和“开发 AI 应用”区分开了。

以前我理解的 AI 应用大概是：

但这个项目让我看到更完整的链路：

我的主要收获有以下几点。

1. LangChain4j 更像是 AI 应用的编排层

它不是简单封装模型 API，而是把模型、记忆、RAG、工具、Prompt、结构化输出等能力组织起来。
这对 Java 后端开发者很友好，因为可以继续使用熟悉的 Spring Boot 工程方式。

2. Agent 的关键是“能不能使用工具”

普通聊天机器人只回答文本。
而 Agent 代码开发助手应该能读取文档、搜索资料、分析代码、调用接口，甚至辅助执行部分开发流程。

这个项目中的 InterviewQuestionTool 和 MCP 搜索工具虽然简单，但已经体现了 Agent 的基本思想。

3. RAG 解决的是“回答依据”的问题

没有 RAG 时，模型回答更多依赖自身知识。
有了 RAG 后，可以把本地文档、项目经验、面试题库、学习资料接入进来，让回答更贴近指定资料。

但 RAG 的效果很依赖文档质量和切分策略，不能只看功能有没有跑通。

4. 流式响应直接影响用户体验

AI 应用里，响应时间往往比较长。
如果没有流式输出，用户会觉得系统卡住了。
SSE 虽然实现简单，但已经能明显提升聊天体验。

5. 安全护轨在 Agent 项目中必须提前考虑

如果 AI 只是回答文本，风险相对有限。
但一旦 AI 可以调用工具，安全问题就会放大。
后续做代码开发助手时，工具权限、参数校验、调用日志都应该认真设计。

十三、后续优化方向

如果继续在这个项目基础上扩展，我觉得可以从以下方向入手：

  1. 会话持久化
    当前内存会话重启后会丢失，可以接入 Redis 或数据库。

  2. 用户系统
    增加登录、用户隔离、历史记录、个人知识库等功能。

  3. 向量库持久化
    将 InMemoryEmbeddingStore 替换为 Milvus、PGVector、Elasticsearch 或 Redis Vector。

  4. 代码仓库分析工具
    支持上传或绑定 GitHub 仓库，让 AI 分析项目结构、生成 README、总结模块职责。

  5. 更多 MCP 工具
    接入文档搜索、Issue 查询、CI 日志分析、依赖漏洞检查等能力。

  6. 输出结构化报告
    针对学习路线、简历修改、项目优化建议，生成固定格式报告。

  7. 工具调用审计
    记录模型调用了什么工具、传了什么参数、返回了什么结果。

  8. 前端体验优化
    增加会话列表、停止生成、重新生成、复制代码、代码高亮等功能。

十四、总结

这次 AI 编程小助手项目复现，让我对 Java 技术栈下的大模型应用开发有了更完整的认识。

如果用一句话总结这个项目：

它不是一个简单的 AI 聊天框，而是一个把 Spring Boot、LangChain4j、大模型、RAG、工具调用、MCP、安全护轨和 Vue 前端串起来的 Agent 应用练习项目。

对我来说，最大的收获不是学会了某一个 API，而是理解了一个 AI 应用的基本结构：

后续如果要做一个真正可用的 Agent 代码开发助手，我认为不能只停留在“帮我解释代码”或“帮我写函数”这类问答能力上，而应该继续往项目级能力扩展，例如：

  1. 读取项目目录；
  2. 理解模块依赖；
  3. 分析报错日志；
  4. 生成修改方案；
  5. 调用工具验证结果；
  6. 输出可落地的代码变更建议。

这也是我觉得这个项目最有价值的地方：
它用一个相对轻量的实战项目，把 AI 应用开发中最重要的几块能力都串了一遍。
