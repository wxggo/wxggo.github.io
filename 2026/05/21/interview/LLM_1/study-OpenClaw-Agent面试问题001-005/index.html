Question001: 请介绍一下 AI 超级智能体项目的整体架构设计，有哪些模块？

知识点详解

AI 超级智能体项目可以理解成一个面向用户任务的智能应用平台。它不是单纯调用一次大模型 API，而是把前端交互、后端接口、AI 应用编排、模型能力、RAG、工具调用、MCP 服务、文件存储、向量数据库等模块组合起来，让系统具备“理解任务、规划步骤、调用工具、返回结果”的能力。

整体架构可以按从上到下的方式分成五层：

  1. 客户端层
    • 客户端是用户与系统交互的入口；
    • 主要负责提交用户请求、展示模型回复、展示工具执行结果、展示任务状态；
    • 在 AI 问答或智能体任务中，客户端通常还需要支持流式输出，让用户实时看到生成过程；
    • 项目里可以用 AI 辅助生成前端页面，提高页面开发效率。
  2. API 接口层
    • API 层是后端应用的入口，负责接收客户端请求；
    • 典型职责包括参数校验、鉴权、限流、日志记录、异常处理、请求转发；
    • 可以基于 Spring MVC / Spring WebFlux 实现；
    • 如果涉及大模型流式输出，可以使用 SSE 返回文本流；如果是普通任务提交，则可以返回任务 ID 或同步结果。
  3. AI 应用层
    • 这一层是项目的核心业务编排层；
    • 它不直接等同于大模型，而是负责把用户需求转成具体业务流程；
    • 例如“AI 恋爱大师”偏向对话问答和场景化建议，“YuManus 自主规划智能体”偏向任务规划、工具调用和多步执行；
    • 这一层会决定：该不该调用 RAG、该不该调用工具、是否需要进入 Agent 循环、最终如何组织答案。
  4. AI 能力层
    • AI 能力层封装底层 AI 核心技术，是应用层可复用的能力集合；
    • 主要包括：
      • 大模型能力：提供自然语言理解、生成、推理、总结、规划等能力；
      • RAG 知识库：通过检索增强生成，让 AI 基于外部知识回答问题；
      • 工具调用：让 AI 调用外部函数或服务，比如文件操作、联网搜索、数据库查询；
      • MCP 协议：用标准化方式接入外部工具、资源和服务；
    • 项目中可以基于 Spring AI 封装这一层，并结合业务场景做定制开发。
  5. 外部服务层
    • 外部服务层为 AI 能力层提供数据和工具支撑；
    • 常见服务包括：
      • 向量数据库：存储文档向量，支持 RAG 相似度检索；
      • 三方 API：如搜索 API、地图 API、天气 API、支付接口等；
      • 文件存储：用于保存用户上传文件、知识库文件、执行结果文件；
      • MCP 服务：外部提供的、符合 MCP 标准的工具服务。

面试回答思路

面试时不要只背模块名，最好按照“分层架构 + 每层职责 + 调用链路 + 扩展性”来回答：

  1. 先说这是一个分层架构，不是单点调用大模型；
  2. 再从客户端、API 层、AI 应用层、AI 能力层、外部服务层逐层展开；
  3. 强调 AI 应用层负责业务编排，AI 能力层负责封装 LLM、RAG、Tool、MCP；
  4. 最后说明这种设计的好处：职责清晰、便于维护、便于扩展新的 AI 能力和外部服务。

核心提醒

AI 超级智能体项目的重点不是“大模型能回答问题”，而是工程架构能支撑复杂任务执行。面试时一定要突出：客户端负责交互，API 层负责入口，AI 应用层负责编排，AI 能力层封装智能能力，外部服务层提供数据和工具。

30秒口述版

AI 超级智能体项目我会按五层来设计：客户端负责用户交互，API 层负责接收请求和转发，AI 应用层负责任务编排，AI 能力层封装大模型、RAG、工具调用和 MCP，外部服务层提供向量数据库、三方 API、文件存储和 MCP 服务。这样各层职责清晰，方便维护，也方便后续扩展新的 AI 能力。

Question012: 什么是 AI Agent 中的 Skills？它有什么用？

知识点详解

Skills 可以理解为给 AI Agent 写的操作手册。它本质上通常是一份结构化的指令文件，告诉 Agent 在遇到某类任务时应该怎么做、按什么顺序做、输入输出格式是什么、有哪些注意事项。

如果说大模型本身像一个“通用大脑”，那么 Skills 就是在给它补充某个领域的标准作业流程 SOP。当 Agent 遇到相关任务时，可以读取对应 Skill，按照其中定义的步骤执行，而不是每次都让用户重新解释。

一个典型的 Skill 通常包含以下几部分：

  1. 触发条件
    • 说明什么情况下应该使用这个 Skill；
    • 例如用户要求“创建 Cursor 自定义规则文件”时，就触发 create-rule 相关 Skill。
  2. 任务背景和目标
    • 告诉 Agent 当前任务要解决什么问题；
    • 避免 Agent 只理解字面意思，而不知道最终交付物是什么。
  3. 操作步骤
    • 按顺序写清楚执行流程；
    • 比如先检查目录，再生成文件内容，再校验格式，最后输出结果。
  4. 输入输出规范
    • 输入需要哪些字段；
    • 输出应该是什么格式；
    • 对代码、Markdown、JSON、配置文件这类强格式任务尤其重要。
  5. 常见坑点和约束
    • 告诉 Agent 哪些事情不能做，哪些细节必须注意；
    • 比如不要覆盖用户已有文件、不要泄露密钥、不要生成无效配置。

Skills 的核心价值主要有三点：

  1. 把领域经验固化下来
    • 将某个领域的专业知识、操作流程、注意事项封装成文件；
    • Agent 读完后就能按流程干活，不需要每次由用户重复教学。
  2. 让任务输出更稳定
    • 同一类任务不管执行多少次，都按照 Skill 定义的流程走；
    • 能减少随机发挥，让输出质量更可预测。
  3. 让通用 Agent 具备垂直能力
    • 不需要重新训练模型，只要编写不同 Skills，就能让 Agent 适配不同任务；
    • 例如代码审查 Skill、简历优化 Skill、测试用例生成 Skill、运维排障 Skill。

示例：一个简化版 Skill 文件

核心提醒

Skills 不是工具本身，而是指导 Agent 怎么做事的方法论文件。它解决的是 Agent “会不会做、做得稳不稳、是否符合流程”的问题，而不是“能不能访问外部系统”的问题。

30秒口述版

Skills 就是写给 AI Agent 的操作手册，通常是结构化的 Markdown 指令文件。它会告诉 Agent 什么时候触发、按什么步骤做、输入输出格式是什么、有哪些注意事项。它的价值是把领域经验固化下来，让同类任务输出更稳定，也能让通用 Agent 通过不同 Skills 获得垂直领域能力，而不需要重新训练模型。

Question013: MCP 和 Skills 有什么区别？分别适用于什么场景？

知识点详解

MCP 和 Skills 都是 AI Agent 体系里的重要概念，但它们解决的问题完全不同：

MCP 给 Agent 提供“工具”，Skills 给 Agent 提供“方法论”。

MCP 全称是 Model Context Protocol，它是一套标准化的工具调用协议，主要解决 Agent 如何访问外部服务的问题。比如查数据库、调 API、读文件系统、访问业务系统，都可以通过 MCP Server 暴露成标准能力，再由 MCP Client 调用。

Skills 则是一套结构化指令文档，主要告诉 Agent 遇到某类任务时应该怎么做、按什么步骤做、注意什么规则。它解决的是 Agent 对任务流程和专业方法的掌握问题。

两者可以从几个角度对比：

对比维度 MCP Skills
本质 标准化工具调用协议 结构化任务指令文档
解决问题 Agent 能不能调用外部工具 Agent 会不会按正确流程做事
典型内容 tools、resources、prompts、server、client 触发条件、步骤、输入输出规范、注意事项
关注点 外部能力接入 任务方法沉淀
类比 给厨师配锅碗瓢盆、烤箱、微波炉 给厨师一本菜谱
没有它的结果 Agent 只能“想”，不能真正调用外部系统 Agent 有工具但不知道怎么用，输出不稳定

适用场景

  1. 适合使用 MCP 的场景
    • 需要调用外部 API；
    • 需要查询数据库；
    • 需要读写文件系统；
    • 需要接入企业内部系统；
    • 需要把已有服务标准化暴露给多个 Agent 使用。
  2. 适合使用 Skills 的场景
    • 某类任务有固定流程；
    • 希望 Agent 输出风格和格式稳定；
    • 希望复用领域经验；
    • 希望不重新训练模型也能增强 Agent 的垂直能力；
    • 任务重点不是“调用哪个接口”，而是“怎么把事情做对”。
  3. 两者组合使用的场景
    • 复杂 Agent 项目里通常 MCP 和 Skills 会一起用；
    • MCP 负责提供工具能力，Skills 负责告诉 Agent 如何选择工具、如何组织步骤；
    • 例如“生成一份竞品分析报告”：Skill 规定调研流程和报告结构，MCP 负责联网搜索、读取文档、调用数据库。

面试回答思路

面试中可以用一个类比快速讲清楚：

MCP 像厨房里的工具，Skills 像菜谱。只有工具，没有菜谱，厨师不知道怎么做；只有菜谱，没有工具，厨师也做不出来。

所以在 Agent 系统设计里，MCP 偏工程接入，Skills 偏任务知识沉淀。二者不是替代关系，而是互补关系。

核心提醒

MCP 解决的是“Agent 能不能做”的问题，Skills 解决的是“Agent 会不会做、做得好不好”的问题。回答时千万不要把 Skills 说成外部工具协议，也不要把 MCP 说成提示词模板。

30秒口述版

MCP 和 Skills 解决的是两个层面的问题。MCP 是 Model Context Protocol，本质是工具调用协议，让 Agent 能调用外部服务，比如查数据库、调 API、读文件；Skills 是结构化指令文档，告诉 Agent 遇到某类任务该怎么做、按什么步骤做。简单说，MCP 是工具，Skills 是方法论；MCP 像厨具，Skills 像菜谱，复杂 Agent 项目里两者通常配合使用。

参考来源：面试鸭题号 13252

Question014: 最近 OpenClaw 这么火，你知道它的原理吗？

知识点详解

OpenClaw 可以理解为一个开源的多渠道 AI 网关（Multi-channel AI Gateway）。它不是只做大模型推理，也不是普通聊天机器人，而是把大模型的推理能力翻译成对操作系统、软件 API、硬件设备和外部服务的实际控制能力。

和普通聊天机器人最大的区别在于：普通机器人更多是“回答文字”，而 OpenClaw 这类系统要让 Agent 真正“动手干活”。

理解 OpenClaw 的原理，可以抓住两个核心：

架构层面：Gateway + Agent Runtime
运行机制：Agentic Loop + 工具生态

架构分层

OpenClaw 可以分成两大层：

  1. Gateway 网关层
    • 网关层不仅是消息入口，也是整个系统的常驻基础设施；
    • 它通常以守护进程方式长期运行，负责连接用户、平台、任务和 Agent；
    • 可以理解成 API Gateway + 进程管理器 + 任务调度器的结合体。

Gateway 层常见职责包括：

  • 多平台接入：接入不同渠道，把各种消息格式统一成内部 MsgContext；
  • 会话隔离：为每个用户、群组、Agent 维护独立的 Session Key，避免上下文串扰；
  • 排队控制：并发消息太多时进入队列，避免 Agent 被打爆；
  • 心跳巡检：通过 Heartbeat 周期性唤醒 Agent 做主动任务；
  • Cron 定时调度：支持基于 cron 表达式的定时任务；
  • 记忆刷盘：在上下文压缩前，将关键记忆持久化到磁盘或数据库。
  1. Agent Runtime 运行时层
    • Runtime 层才是真正“干活”的地方；
    • 它接收 Gateway 分发的消息，组装上下文，调用大模型，解析模型输出，执行工具，然后把结果返回；
    • 运行时的核心机制就是 Agentic Loop，也就是智能体循环。

Agentic Loop 运行机制

OpenClaw 这类系统的核心不是一次 API 调用，而是一个循环：

这套机制本质上类似 ReAct 思路：模型先思考，再行动，再观察结果，再决定下一步。它让 Agent 不只是回答，而是能多步执行任务。

核心提醒

OpenClaw 的关键不是“接了一个大模型”，而是通过网关和运行时机制，把大模型变成能持续执行任务的 Agent 系统。面试时要重点讲清楚 Gateway 负责接入、会话、队列、调度和记忆，Agent Runtime 负责模型调用、工具执行和循环决策。

30秒口述版

OpenClaw 本质是一个多渠道 AI 网关，不只是聊天机器人，而是把大模型推理能力转换成实际工具执行能力。它可以分成 Gateway 网关层和 Agent Runtime 运行时层：Gateway 负责多平台接入、会话隔离、排队、心跳、定时任务和记忆持久化；Runtime 负责加载上下文、调用 LLM、解析 tool call、执行工具、追加结果并循环，直到任务完成。核心就是 Gateway + Agentic Loop。

参考来源：面试鸭题号 13216

Question015: 什么是 AI Agent？它和直接调用大模型 API 做一次问答有什么本质区别？

知识点详解

直接调用大模型 API，本质上是一次“一问一答”：用户发一条 prompt，模型返回一条 response，然后流程结束。

AI Agent 则完全不同。Agent 是一个有状态的循环决策系统，它不仅能生成文本，还能感知环境、规划步骤、调用工具、观察结果，并根据结果决定下一步做什么，直到任务完成。

二者的本质区别可以总结为三点：

  1. Agent 有工具调用能力
    • 直接调用大模型 API 通常只能返回文本；
    • Agent 可以调用外部工具，比如读写文件、执行代码、查数据库、调第三方接口、联网搜索；
    • 这意味着 Agent 能操作外部世界，而不只是“说”。
  2. Agent 有记忆和上下文
    • 单次 API 调用是无状态的，每次都从零开始；
    • Agent 会维护任务上下文、历史步骤、工具执行结果、用户偏好、长期记忆；
    • 这样它能知道自己已经做过什么、拿到了什么结果、下一步应该怎么做。
  3. Agent 有自主决策循环
    • 单次 API 调用是被动的，用户问一句，它答一句；
    • Agent 会主动规划步骤，并在每一步后观察结果，再判断任务是否完成；
    • 如果任务没完成，它会继续执行下一步，而不是直接结束。

单次 API 调用流程

这种方式适合简单问答、文本生成、摘要、翻译、代码解释等任务。它的优点是简单、稳定、成本可控；缺点是不能主动执行复杂任务，也不能天然保存状态。

Agent 运行流程

Agent 更适合多步骤任务，例如：

  • “帮我搜索资料并生成一份报告”；
  • “读取这个项目代码，找出可能的 bug”；
  • “根据文件内容生成测试用例”；
  • “定时检查网页变化，有更新就通知我”；
  • “调用多个业务系统完成一个审批流程”。

面试回答思路

回答这个问题时，可以先用一句话定性：

直接调用大模型 API 是无状态的一次问答，AI Agent 是有状态、能调用工具、能循环决策的任务执行系统。

然后展开三点：工具调用、记忆上下文、自主循环。最后用流程对比收尾，说明 Agent 的价值在于让大模型从“生成文本”升级成“执行任务”。

核心提醒

Agent 不是简单套一层 Prompt，也不是普通聊天机器人。它至少要具备任务状态、工具调用和循环决策能力。没有工具调用和状态管理的系统，通常只能算大模型应用，不能算完整 Agent。

30秒口述版

直接调用大模型 API 是一次无状态问答：用户发 prompt，模型回 response，结束。AI Agent 是有状态的循环决策系统，它有工具调用能力，可以读文件、查数据库、调接口；有记忆和上下文，知道自己做过什么；还有自主决策循环，会规划步骤、执行工具、观察结果，没完成就继续迭代。所以 Agent 的本质是把大模型从“回答问题”升级成“执行任务”。

参考来源：面试鸭题号 13217

总结

这一组问题主要围绕 AI Agent 工程化落地 展开，重点不再只是大模型本身，而是如何把大模型变成可执行、可扩展、可维护的智能系统。

需要重点记住以下几条主线：

  1. AI 超级智能体架构：客户端、API 层、AI 应用层、AI 能力层、外部服务层；
  2. Skills：给 Agent 的操作手册，用来沉淀流程和经验；
  3. MCP：给 Agent 的工具调用协议，用来标准化接入外部能力；
  4. OpenClaw：多渠道 AI 网关，核心是 Gateway + Agent Runtime + Agentic Loop；
  5. AI Agent：有状态、能调用工具、能循环决策的任务执行系统。

面试时可以用一句话串起来：
