Question006: 什么是 SSE？为什么 AI 应用接口选择 SSE 而不是 WebSocket？

知识点详解

SSE 全称是 Server-Sent Events，中文一般叫服务器发送事件。它是一种基于 HTTP 协议的服务端向客户端单向推送技术，浏览器端可以通过 EventSource 接收服务端持续推送的数据。

在 AI 应用里，SSE 最常见的场景就是流式输出大模型回复。比如用户问一个问题后，模型不是等整段答案生成完再返回，而是一个 token、一个词、一个句子地逐步返回，前端就可以像“打字机”一样实时展示内容。

从通信模型看：

  1. 普通 HTTP 请求：客户端请求一次，服务端返回一次，适合短响应；
  2. SSE：客户端建立连接后，服务端可以持续向客户端推送消息，适合服务端持续产生数据；
  3. WebSocket：客户端和服务端建立全双工连接，双方都可以随时主动发送消息，适合强实时双向交互。

AI 应用接口常选 SSE，而不是 WebSocket，核心原因可以从下面几方面回答：

  1. 通信方向更匹配
    • 大模型流式回答主要是“服务端生成内容，然后推给客户端”；
    • 客户端通常只需要先发送一次问题，后续就是等待模型不断返回结果；
    • 这种场景本质上是服务端到客户端的单向数据流，正好符合 SSE 模型；
    • WebSocket 虽然能力更强，但它的双向通信在普通 AI 问答流式输出里不一定用得上。
  2. 实现成本更低
    • SSE 基于标准 HTTP，不需要额外设计一套复杂的连接协议；
    • 前端使用原生 EventSource 或 fetch 读取流即可；
    • 后端只要返回 Content-Type: text/event-stream，按事件格式持续写出数据；
    • WebSocket 需要管理连接生命周期、心跳、断线重连、消息协议、连接池等，工程复杂度更高。
  3. 对网关和基础设施更友好
    • SSE 走 HTTP，通常更容易通过 Nginx、API Gateway、负载均衡和鉴权中间件；
    • 对现有 REST API 架构改造成本较低；
    • AI 接口常常还需要接入鉴权、限流、日志、审计等 HTTP 体系能力，SSE 更容易复用。
  4. 浏览器自带重连机制
    • SSE 规范中包含自动重连机制；
    • 可以通过 id / Last-Event-ID 记录事件位置；
    • 网络短暂中断后，客户端可以带着最后收到的事件 ID 重新连接，服务端可选择从断点继续推送；
    • WebSocket 的重连、恢复、消息补偿通常需要业务自己实现。
  5. 更适合“文本流”场景
    • SSE 本身就是面向文本事件流设计的；
    • 大模型返回的 token、片段、结束标记、错误信息，都可以封装成事件；
    • 例如 event: message、event: done、data: xxx，语义非常清晰。

核心提醒

SSE 不是 WebSocket 的替代品，而是适合不同场景的技术选择。
如果是AI 流式输出、日志推送、通知推送这类服务端单向推送场景，SSE 简单、稳定、够用；如果是在线聊天室、协同编辑、游戏对战、实时双向控制这类强双向交互场景，WebSocket 更合适。

30秒口述版

SSE 是基于 HTTP 的服务端单向推送技术，浏览器可以用 EventSource 接收流式数据。AI 应用里，大模型回答通常是服务端不断把 token 推给前端，本质是单向输出，所以 SSE 更简单、更贴合 HTTP 体系，也自带重连机制。WebSocket 适合真正需要双向实时交互的场景，比如聊天室和协同编辑；普通 AI 流式问答用 SSE 更轻量。

Question007: 什么是 RAG？RAG 的主要流程是什么？

知识点详解

RAG 全称是 Retrieval Augmented Generation，中文叫检索增强生成。

一句话解释：RAG 就是让大模型在回答之前，先去外部知识库查资料，再基于查到的资料生成答案。

大模型本身有两个典型短板：

  1. 知识有滞后性
    • 模型训练完成后，参数里的知识就相对固定了；
    • 它不知道训练之后发生的新事情；
    • 对企业内部新文档、新制度、新接口也天然不了解。
  2. 容易产生幻觉
    • 模型可能会根据语言模式“编”出看似合理但实际错误的答案；
    • 尤其是问公司内部文档、业务规则、接口参数时，单靠模型记忆风险很高。

RAG 的核心价值就是：把外部知识库挂到大模型旁边，让模型开卷考试。

RAG 的主要流程

RAG 流程可以用三个字概括：找、缝、写。

1）找：先检索相关资料

用户提问后，系统不会直接把问题丢给大模型，而是先把问题转成向量，去向量数据库里检索相关文档片段。

典型流程：

  • 对用户问题做 Embedding；
  • 在向量库中做相似度检索；
  • 找出和问题最相关的 TopK 文档片段；
  • 必要时结合关键词检索、元数据过滤、重排序等方式提升召回质量。

例如用户问：

“公司报销制度里，差旅住宿标准是多少？”

系统应该先从知识库里检索“报销制度”“差旅住宿”“住宿标准”等相关文档片段。

2）缝：把资料和问题拼成 Prompt

检索到资料后，系统会把这些资料作为上下文，和用户原始问题一起拼成一个更完整的 Prompt。

3）写：让大模型基于上下文生成答案

最后，把拼好的 Prompt 发给大模型。
模型根据检索到的资料生成回答，输出更准确、更可追溯的内容。

如果检索内容足够准确，RAG 可以显著降低幻觉；但如果检索阶段找错了资料，后面模型再强也很难答对。

RAG 的完整工程链路

实际项目中，RAG 通常分为两个阶段：

  1. 离线知识入库
    • 文档采集：PDF、Word、网页、数据库、接口文档；
    • 文档清洗：去水印、页眉页脚、乱码、重复内容；
    • 文档切分：按标题、段落、小节等语义边界切块；
    • 向量化：使用 Embedding 模型把文本块转成向量；
    • 入库：存入 Milvus、Elasticsearch、pgvector、Pinecone 等检索系统。
  2. 在线问答检索
    • 用户提问；
    • 问题向量化；
    • 检索相关文档；
    • 重排序和过滤；
    • 组装 Prompt；
    • 调用大模型生成答案；
    • 返回答案和引用来源。

核心提醒

RAG 的重点不是“把资料塞给大模型”这么简单，而是检索质量决定生成质量。
如果文档切分差、Embedding 不匹配、检索策略不合理，最终答案依然会不准。

30秒口述版

RAG 是检索增强生成，就是让大模型先查资料再回答。流程可以概括为找、缝、写：先把用户问题向量化，到向量库里找相关文档片段；再把原问题和检索内容拼成 Prompt；最后让大模型基于资料生成答案。它主要解决模型知识滞后和幻觉问题，但效果很依赖分块和检索质量。

Question008: 如何优化 RAG 的检索效果？

知识点详解

RAG 检索效果差，问题通常不在大模型本身，而是在文档切片、查询理解、检索策略这三个环节。

面试时可以按“三层优化法”回答：先优化数据，再优化查询，最后优化召回和排序。

1）文档预处理：从源头提高知识库质量

RAG 的第一步是把文档变成可检索的知识块。
如果原始文档质量差，后面 Embedding、向量检索、重排序都会受到影响。

常见优化点：

  1. 清洗无效内容
    • 去掉页眉、页脚、水印、目录噪声；
    • 清理乱码、重复段落、广告、版权声明；
    • 对 PDF 解析错误、表格错位、换行异常做修复。
  2. 按语义边界切分
    • 不建议简单按固定 512 tokens 一刀切；
    • 更推荐按标题、段落、小节、列表、代码块等语义结构切分；
    • 可以使用类似 RecursiveCharacterTextSplitter 的递归切分策略；
    • 先按大结构切，再按段落切，最后才按固定长度兜底。
  3. 设置合理 chunk 大小
    • chunk 太小：上下文不完整，答案缺关键条件；
    • chunk 太大：召回不精准，还会占用 Prompt 空间；
    • 常见范围是 512～1024 tokens，具体要根据文档类型和模型窗口测试。
  4. 增加 chunk overlap
    • 对相邻切片保留一定重叠区域；
    • 避免一句话或一个定义被切断；
    • 例如 overlap 设置为 50～100 tokens。
  5. 补充元数据
    • 给每个切片加上来源文档、章节、标题、日期、作者、业务线等标签；
    • 查询时可按时间、部门、文档类型过滤；
    • 对企业知识库尤其重要。

2）查询增强：让用户问题变得更适合检索

用户的自然语言问题经常不适合直接拿去检索。

比如用户问：

“鱼皮是谁？”

但知识库里写的是：

“程序员鱼皮，原名某某，是编程导航创始人……”

这时直接用“鱼皮是谁”做向量检索，可能召回不稳定。
因此需要做查询增强，把口语化问题转换成更规范、更容易匹配的检索表达。

常见做法：

  1. 查询重写
    • 用大模型把用户问题改写成更标准的检索句；
    • 例如“这个怎么报销？”改成“公司报销流程、报销材料、审批规则”。
  2. 查询扩展
    • 为同一个问题生成多个角度的子查询；
    • 例如“如何优化 RAG 检索效果”可以扩展为：
      • “RAG 文档切分优化”
      • “RAG query rewrite”
      • “RAG hybrid search”
      • “RAG rerank 重排序”
  3. 查询拆解
    • 对复杂问题拆成多个子问题分别检索；
    • 例如“对比 A 和 B 的价格、性能、适用场景”，可以拆成 A 的信息、B 的信息、对比维度三类检索。
  4. 元数据过滤
    • 从问题中抽取时间、部门、产品、版本号等条件；
    • 用 metadata filter 缩小检索范围；
    • 例如“2024 年 Q4 的销售政策”可以加上 year=2024、quarter=Q4。

3）混合检索与重排序：提高召回和精准度

单纯向量检索有一个致命问题：它擅长语义相似，但不擅长精确匹配。

例如下面这些内容，关键词非常关键：

  • 专有名词；
  • 代码函数名；
  • API 路径；
  • 错误码；
  • 数字；
  • 人名；
  • 产品型号。

这些场景下，BM25 关键词检索往往比纯向量检索更可靠。

因此更推荐使用混合检索：

  1. 向量检索
    • 适合语义相似；
    • 能处理“表达不一致但意思相近”的问题。
  2. 关键词检索 / BM25
    • 适合精确匹配；
    • 对代码、数字、术语、接口路径更敏感。
  3. RRF 融合排序
    • RRF 全称 Reciprocal Rank Fusion；
    • 可以把向量检索和关键词检索的结果合并排序；
    • 避免只依赖一种检索方式。
  4. Reranker 重排序
    • 第一阶段先召回较多候选，例如 Top50；
    • 第二阶段用 reranker 重新计算“问题-文档片段”的相关性；
    • 最终只保留 Top3～Top5 放进 Prompt。

核心提醒

RAG 检索优化不要一上来就调大模型参数。
正确顺序是：先看文档是否干净，再看分块是否合理，再看查询是否被正确理解，最后再看检索融合、rerank 和 topK 参数。
如果检索结果本身不相关，后面生成阶段很难补救。

30秒口述版

优化 RAG 检索效果可以从三层入手：第一是文档预处理，清洗噪声、按语义切分、加元数据；第二是查询增强，把用户口语化问题做重写、扩展和拆解；第三是检索策略，用向量检索加 BM25 混合召回，再用 RRF 和 reranker 重排序。核心原则是先保证召回资料准，再让模型生成答案。

Question009: 什么是结构化输出？Spring AI 是怎么实现结构化输出的？

知识点详解

结构化输出就是让大模型不要返回一段自由文本，而是返回程序可以直接解析的数据结构，比如 JSON、XML、表格字段、Java Bean 等。

它的核心价值是：让大模型的输出能直接进入业务系统，而不是只能给人看。

例如：

  • 智能客服：提取用户意图、工单类型、紧急程度；
  • 简历解析：提取姓名、学历、技能、工作年限；
  • 订单助手：提取商品名、数量、地址、联系方式；
  • RAG 问答：返回答案、引用来源、置信度；
  • Agent 调用：返回工具名和工具参数。

为什么需要结构化输出？

  1. 方便程序解析
    • 后端可以直接把 JSON 反序列化成对象；
    • 不需要再从自然语言里用正则硬提取字段。
  2. 降低系统不确定性
    • 自由文本每次表达可能不同；
    • 结构化输出可以约束字段名、字段类型和返回格式。
  3. 便于后续业务流转
    • 结构化结果可以直接入库、调用接口、触发工作流；
    • 比如把 intent=refund 的用户请求转给退款流程。
  4. 适合和函数调用 / Agent 结合
    • 工具调用需要明确参数；
    • 参数格式越稳定，系统越容易自动化。

Spring AI 如何实现结构化输出？

Spring AI 里结构化输出的核心是 StructuredOutputConverter。

它可以理解为同时做两件事：

  1. 调用前：提供格式说明
    • StructuredOutputConverter 实现了 FormatProvider；
    • 它会生成一段格式要求，追加到 Prompt 中；
    • 告诉模型必须按指定 JSON Schema 或对象结构返回。
  2. 调用后：把文本转成 Java 对象
    • StructuredOutputConverter 同时也承担转换器角色；
    • 模型返回 JSON 字符串后，它会把 JSON 解析成 Java Bean、Map 或 List；
    • 底层通常借助 Jackson ObjectMapper 完成反序列化。

也就是说，Spring AI 的结构化输出不是“模型天然一定听话”，而是通过：

Prompt 格式约束 + 输出文本解析 + Java 类型转换

来完成结构化结果落地。

常见 Converter

  1. BeanOutputConverter
    • 把模型输出解析成 Java Bean / Record；
    • 适合固定字段结构。
  2. MapOutputConverter
    • 把模型输出解析成 Map；
    • 适合字段不完全固定的场景。
  3. ListOutputConverter
    • 把模型输出解析成列表；
    • 适合关键词列表、标签列表、步骤列表等场景。

面试回答重点

面试时可以这样组织：

  1. 结构化输出是让模型返回 JSON/XML/对象等可解析格式；
  2. 目的不是让人读，而是方便程序直接消费；
  3. Spring AI 通过 StructuredOutputConverter 实现；
  4. 调用前由 FormatProvider 生成格式说明放进 Prompt；
  5. 调用后用 Converter 把模型输出文本反序列化成 Java 对象；
  6. 生产环境要做异常处理和结果校验，因为模型仍可能返回不合法 JSON。

核心提醒

结构化输出不是 100% 可靠的强约束。
实际项目中要配合低温度参数、清晰 Schema、输出校验、JSON 修复和失败重试机制。
不要默认模型每次都会严格返回合法 JSON，尤其是在复杂嵌套对象场景下。

30秒口述版

结构化输出就是让大模型返回 JSON、XML 或 Java 对象这类程序能直接解析的格式，而不是普通自然语言。Spring AI 主要通过 StructuredOutputConverter 实现：调用前生成格式说明塞进 Prompt，引导模型按 Schema 返回；调用后再用 Converter 和 ObjectMapper 把返回文本解析成 Java Bean。生产环境还要做校验、异常处理和重试，因为模型不一定每次都严格输出合法 JSON。

Question010: 什么是 MCP 模型上下文协议？如何利用 Spring AI 实现 MCP 客户端和服务端？

知识点详解

MCP 全称是 Model Context Protocol，中文叫模型上下文协议。

它是一种面向 AI 应用的开放协议，目标是标准化大模型与外部工具、资源、服务之间的交互方式。

可以把 MCP 理解成 AI 应用世界里的“USB 接口”：

  • 大模型应用是主机；
  • 外部工具、数据库、文件系统、业务 API 是设备；
  • MCP 负责定义统一的连接和调用标准；
  • 这样 AI 应用就不用为每个工具单独写一套私有适配逻辑。

MCP 解决了什么问题？

在没有 MCP 时，如果大模型要调用外部能力，通常需要为每个系统单独集成：

  • 查数据库写一套接口；
  • 调 GitHub 写一套接口；
  • 读本地文件写一套接口；
  • 查企业知识库再写一套接口；
  • 调天气、地图、日历、工单系统都要分别适配。

这样会导致：

  1. 工具接入成本高；
  2. 协议不统一；
  3. 不同 AI 应用之间难复用；
  4. 权限、上下文、工具描述难标准化；
  5. Agent 能力扩展困难。

MCP 的意义就是把这些外部能力抽象成标准的：

  1. Tools
    • 可被模型调用的工具；
    • 例如查询天气、创建工单、执行 SQL、调用内部 API。
  2. Resources
    • 可读取的上下文资源；
    • 例如文件、数据库记录、文档、配置。
  3. Prompts
    • 可复用的提示模板；
    • 例如代码审查模板、日报总结模板、客服回复模板。

MCP 的角色模型

MCP 通常可以理解为三层角色：

  1. Host
    • AI 应用本体；
    • 例如 IDE 插件、聊天机器人、智能客服系统。
  2. Client
    • Host 内部的 MCP 客户端；
    • 负责连接 MCP Server，发现工具并发起调用。
  3. Server
    • 提供外部能力的一方；
    • 把工具、资源、Prompt 暴露给 MCP Client。

一句话：MCP Server 暴露能力，MCP Client 连接能力，Host 借助这些能力增强大模型。

如何利用 Spring AI 实现 MCP 服务端？

Spring AI 可以把 Spring Bean 中的方法注册为工具，并通过 MCP Server 暴露出去。

核心步骤：

  1. 引入 MCP Server 相关依赖；
  2. 定义业务工具方法；
  3. 用工具注解或 ToolCallbackProvider 暴露工具；
  4. 配置 MCP Server 名称、版本、传输方式；
  5. 启动服务，让客户端发现和调用工具。

面试回答重点

面试时不要只背概念，可以从“为什么需要协议”说起：

  1. MCP 是 Model Context Protocol，用来标准化模型和外部工具/资源的交互；
  2. 它把工具、资源、Prompt 抽象成统一协议；
  3. 好处是不同 AI 应用可以复用同一批工具服务；
  4. Spring AI 可以作为 MCP Server，把 Spring Bean 方法暴露成工具；
  5. Spring AI 也可以作为 MCP Client，连接外部 MCP Server，把工具注册给 ChatClient；
  6. 最终模型就可以在对话中自动选择和调用外部工具。

核心提醒

MCP 的重点不是“又一个 RPC 框架”，而是面向大模型上下文和工具调用的标准化协议。
它关注的不只是怎么调接口，还包括工具描述、参数 Schema、资源暴露、Prompt 模板、模型可发现性和上下文组织方式。
回答时可以用“AI 应用的 USB 接口”来帮助面试官快速理解。

总结：这 5 个问题怎么串起来记？

这 5 个问题其实对应 AI 应用开发的完整链路：

  1. SSE：解决模型回答如何流式返回给前端；
  2. RAG：解决模型如何接入外部知识库；
  3. RAG 检索优化：解决外部知识怎么查得准；
  4. 结构化输出：解决模型结果如何被程序稳定解析；
  5. MCP：解决模型如何标准化调用外部工具和服务。

面试时可以按这条主线表达：

一个完整 AI 应用里，用户提问后，系统可能先通过 RAG 检索知识库，再把上下文交给大模型生成答案；如果需要调用外部工具，可以通过 MCP 标准化接入；如果返回给前端，可以用 SSE 做流式输出；如果返回给后端系统处理，就要用结构化输出保证结果可解析。

复习口诀

SSE 管输出，RAG 管查资料，优化管查得准，结构化管可解析，MCP 管工具接入。
