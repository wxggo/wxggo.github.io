Notes-项目复盘：把 AI 编程小助手改造成博客数字人面试 Agent

发表于2026-05-05|更新于2026-05-05|项目回顾
|字数总计:5.2k|阅读时长:18分钟|阅读量:

这篇文章是对上一篇 AI 编程小助手 - LangChain4j 项目复现 的继续改造。
原项目的目标是做一个能回答编程学习、求职规划、项目建议和面试题的 AI 小助手；这次我把它进一步接入自己的个人博客，把博客文章、项目复盘、学习笔记和面试题整理变成 RAG 知识库，做成一个可以嵌入 https://wxggo.github.io/ 的「博客数字人面试 Agent」。

一、项目背景

之前复现的 AI 编程小助手更像一个通用型 Demo：

这个 Demo 已经包含了大模型调用、Prompt、流式输出、RAG、Tool Calling、MCP 等能力，但它的问题也很明显：它回答的是通用问题，还没有和“我是谁、我做过什么、我写过什么”真正绑定起来。

而我的个人博客本身已经积累了很多素材：

  1. 项目复盘文章，比如 AI 编程小助手、无人机、智能车等项目记录。
  2. 学习笔记，比如 LLM、Java、计算机网络、算法题整理。
  3. 面试八股和题库内容。
  4. 关于页面、个人介绍、学习路线和求职准备。

这些内容其实天然就是一份“个人知识库”。如果只把它们作为静态文章展示，面试官需要自己翻目录、点文章、找重点；如果做成 RAG Agent，面试官就可以直接问：

于是这次改造的目标就很清晰：
把个人博客从静态内容展示，升级为一个可以对话的个人素材库入口。

二、改造目标

这次改造不是从零写一个新项目，而是在原 AI 编程小助手项目上做二次工程化：

改造方向 原 AI 编程小助手 博客数字人 Agent
产品定位 编程学习和求职问答助手 面向面试官的个人数字人
知识来源 少量本地 docs 文档 Hexo 博客文章 + about 页面 + docs
回答身份 编程领域小助手 使用第一人称回答的 wxggo
前端形态 独立 Vue 页面 / 调试页面 嵌入 Butterfly 博客的悬浮对话窗
访问入口 本地前后端联调 GitHub Pages 博客页面
部署方式 本地运行 Demo 阿里云 ECS 部署后端，GitHub Pages 部署前端
对话体验 普通问答 SSE 流式输出、模型切换、记忆 ID、Markdown 渲染

3.2 Prompt：从“小助手”到“数字人”

原项目的 Prompt 偏向“编程领域小助手”，这次改造的 Prompt 更强调身份感和面试场景：

这部分很关键。因为如果只做 RAG，不做身份约束，模型很容易退化成“资料总结器”；而我的目标是让它像我本人一样和面试官交流，所以 Prompt 中明确了几个规则：

  1. 使用第一人称回答。
  2. 项目问题要说明背景、技术选型、个人负责内容、难点、解决方案和复盘。
  3. 八股和算法问题可以结合博客学习笔记作答。
  4. 知识库没有记录的内容不能编造，要说明“目前博客素材中没有记录”。
  5. 不能泄露 API Key、服务器密码、环境变量等敏感信息。

这样 Agent 的回答风格就从“通用 AI 助手”变成了“基于我的博客素材回答问题的数字人”。

项目中增加了全局 CORS 配置：

5.4 浏览器拦截问题

部署后遇到一个比较现实的问题：因为域名备案、HTTPS、浏览器安全策略等因素，不同浏览器对后端请求的处理不一样。

实测：

浏览器 表现
Google Chrome 可以正常访问后端 Agent
夸克浏览器 会拦截部分对后端智能体项目的请求

这类问题不是业务代码错误，而是部署环境和浏览器安全策略共同导致的。
目前项目通过多个 API 地址 fallback 缓解访问失败问题，但更彻底的方案应该是：

  1. 后端服务绑定备案完成的正式域名。
  2. 使用稳定的 HTTPS 证书。
  3. Nginx 反向代理 Spring Boot 服务。
  4. 对 SSE 增加正确的代理配置，关闭不必要的缓冲。
  5. 前端只保留一个稳定的 HTTPS API 地址，减少混合内容和安全策略问题。

可以说，这次部署让我更直观地感受到：AI 项目不只是“本地能跑”，真正上线还要处理域名、证书、跨域、代理、浏览器兼容这些工程问题。

七、相比原项目的收获

7.1 从 Demo 思维到产品思维

原 AI 编程小助手更像是学习 LangChain4j 的练手项目。
它证明了“我会调用大模型、会接 RAG、会做工具调用”，但对用户来说，它仍然是一个泛化的问答工具。

这次改造成博客数字人后，项目有了更明确的使用场景：

这个目标让很多技术选择都变得自然：

  1. 为什么接博客 Markdown？因为博客就是我的素材库。
  2. 为什么要第一人称 Prompt？因为它要代表我回答。
  3. 为什么要嵌入 GitHub Pages？因为面试官最可能先打开我的个人主页。
  4. 为什么要流式输出？因为对话体验比静态结果更重要。

7.2 RAG 的价值更明显了

在通用问答 Demo 里，RAG 有时会显得像“为了用技术而用技术”。
但在博客数字人项目里，RAG 的价值非常直接：

这也是我对 RAG 更深的理解：
RAG 不只是减少幻觉，它更重要的作用是把大模型和某个具体领域、具体组织、具体个人的私有知识绑定起来。

7.3 部署比本地开发更考验细节

本地开发时，很多问题都被隐藏了：

但上线到 GitHub Pages + 阿里云 ECS 后，就必须面对：

  1. HTTPS 页面请求 HTTP 后端的混合内容问题。
  2. GitHub Pages 静态站点和 Spring Boot 服务跨域问题。
  3. SSE 在代理层的连接保持问题。
  4. 域名备案和浏览器安全拦截问题。
  5. API Key、服务器密码等敏感信息管理问题。

这些问题虽然不如写模型调用代码“酷”，但它们决定了项目能不能真正被别人使用。

八、后续优化方向

当前版本已经可以作为博客数字人 Demo 使用，但还有不少可以继续优化的地方：

  1. 向量库持久化
    目前更偏向内存级别的知识库加载，后续可以接入 Milvus、Elasticsearch、Redis Vector 或 PGVector，避免每次启动都重新向量化。

  2. 知识库构建流水线
    博客文章更新后，可以通过脚本自动清洗 Markdown front-matter、图片标签、HTML 片段，再增量更新知识库。

  3. 回答引用来源
    RAG 片段中已经带有来源信息，后续可以让回答末尾附上“参考文章”，方便面试官继续阅读原文。

  4. 更严格的安全护栏
    当前只做了简单敏感词输入护栏，后续可以增加输出过滤、敏感信息检测、Prompt 注入防护。

  5. 前端体验增强
    可以增加对话清空、复制回答、停止生成、错误重试、移动端更细致适配等功能。

  6. 部署规范化
    后续可以用 Nginx + HTTPS + systemd / Docker Compose 规范部署，把启动、日志、重启和健康检查都纳入流程。

  7. 数字人画像强化
    可以单独整理一份 profile.md，把个人经历、项目亮点、求职目标、技术栈关键词沉淀成更稳定的高优先级知识源。

九、总结

这次项目改造，本质上是把一个“AI 编程小助手 Demo”推进成了一个更贴近个人场景的“博客数字人 Agent”。

它的核心变化可以概括为：

从技术上看，这个项目串起了 Spring Boot、LangChain4j、通义千问、RAG、SSE、Tool Calling、MCP、Hexo、Butterfly、GitHub Pages 和阿里云部署；从产品上看，它把博客从“给人阅读的静态文章集合”，变成了“可以被提问、被检索、被对话的个人知识库”。

对我来说，这个项目最有价值的地方不是用了多少技术点，而是它把过去写下来的文章重新组织成了一个能对外表达自己的系统。
当面试官打开博客时，看到的不只是文章列表，还可以直接和我的“数字人”对话，快速了解我的项目经历、学习轨迹和技术理解。

这也是我对 AI Agent 更实际的一层理解：
Agent 不一定一开始就要做得很复杂，它可以先从一个明确的人、明确的知识库、明确的使用场景开始。
