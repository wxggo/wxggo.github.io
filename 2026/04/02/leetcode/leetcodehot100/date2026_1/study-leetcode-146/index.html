Leetcode-146:LRU缓存机制-双向链表+哈希表最优实现（哨兵节点版）

发表于2026-04-02|更新于2026-04-02|Java学习笔记数据结构与算法
|字数总计:3.8k|阅读时长:14分钟|阅读量:

前言

LRU（Least Recently Used，最近最少使用）缓存机制是面试中数据结构模块的高频压轴题，也是工业界缓存设计的核心思想（如Redis的LRU淘汰策略）。其核心是「数据访问后标记为“最近使用”，缓存满时淘汰“最久未使用”数据」，最优实现是「双向链表 + 哈希表」，而哨兵节点的引入能彻底避免链表操作的空指针问题。本文聚焦LRUCache的哨兵节点优化版实现，重点解答三个核心疑问：为什么要用双向链表？哈希表的作用是什么？哨兵节点如何简化链表操作？拆解LRU实现的底层逻辑。

1、题目

设计并实现一个满足 LRU (最近最少使用) 缓存约束的数据结构。

实现 LRUCache 类：

  • LRUCache(int capacity) 以正整数作为容量 capacity 初始化 LRU 缓存
  • int get(int key) 如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1
  • void put(int key, int value) 如果关键字 key 已经存在，则变更其数据值；如果不存在，则向缓存中插入该组 key-value。如果插入操作导致关键字数量超过 capacity，则应该逐出最久未使用的关键字。

要求

  • 函数 get 和 put 必须以 O(1) 的平均时间复杂度运行。

示例

提示

  • 1 <= capacity <= 3000
  • 0 <= key <= 10^4
  • 0 <= value <= 10^5
  • 最多调用 2 * 10^5 次 get 和 put

2、分析

2.1 问题核心：LRU的「访问-淘汰」规则

LRU的核心诉求是：

  1. 快速访问：根据key快速找到对应值（要求O(1)时间）；
  2. 快速更新：访问key后，将其标记为“最近使用”（要求O(1)时间）；
  3. 快速淘汰：缓存满时，快速找到并删除“最久未使用”的key（要求O(1)时间）。

2.2 核心难点：数据结构的选型与协同

单一数据结构无法满足所有O(1)要求，需组合「双向链表 + 哈希表」，同时引入哨兵节点简化链表操作：

数据结构 作用 为什么选它？
双向链表 维护数据的「使用时间顺序」 表头=最近使用，表尾=最久未使用；双向链表可O(1)删除任意节点（需知道节点引用）
哈希表（HashMap） 映射key到链表节点的引用 O(1)时间根据key找到对应节点，解决链表遍历O(n)的问题
哨兵节点（dummy） 作为链表头尾的虚拟节点 避免空指针判断（如删除最后一个节点、插入第一个节点），简化链表操作

2.3 核心逻辑：三个关键操作的设计

  1. 节点移动（访问后移到表头）：get/put已存在的key时，将节点从原位置删除，插入到链表头部；
  2. 节点添加（新增key）：新节点插入到链表头部，同时哈希表记录key→节点的映射；
  3. 节点淘汰（缓存满）：删除链表尾部节点，同时哈希表移除对应key的映射。

3.2 核心设计逐行解释（重点解答三大疑问）

（1）双向链表节点设计：为什么要存储key？

核心原因：淘汰尾部节点时，需要通过节点获取key来删除哈希表的映射。

  • 若节点仅存储value，淘汰尾部节点时，无法知道对应的key，哈希表的映射无法删除；
  • 存储key后，通过backNode.key可直接拿到要淘汰的key，完成哈希表的O(1)删除。

（2）哨兵节点初始化：dummy.pre = dummy; dummy.next = dummy;

核心原因：让链表永远有“头尾”，避免空指针判断。

  • 无哨兵节点时：插入第一个节点需判断head == null，删除最后一个节点需判断tail == null，逻辑复杂且易出空指针；
  • 有哨兵节点时：链表初始状态是“自环”（哨兵的pre和next都指向自己），无论链表为空/非空，插入/删除操作的逻辑完全统一，无需额外判断；
  • 举例：插入第一个节点时，直接调用putHead(node)即可，无需处理“头节点为空”的特殊情况。

（3）哈希表的核心作用：HashMap<Integer, Node> hashMap

核心原因：解决双向链表“查找节点O(n)”的问题，保证get/put的O(1)时间复杂度。

  • 双向链表的优势是O(1)移动/删除节点，但根据key查找节点需要遍历链表（O(n)）；
  • 哈希表以key为键、节点引用为值，可O(1)时间找到对应节点，完美弥补链表的查找短板；
  • 核心逻辑：所有对节点的操作（移动、删除）都先通过哈希表拿到节点引用，再执行链表操作。

（4）核心辅助方法：getNode/removeNode/putHead

① getNode：查找+移到头部（LRU的“最近使用”核心）
  • 这是LRU的核心方法：任何访问（get/put已存在的key）都会触发该方法，将节点移到链表头部，标记为“最近使用”；
  • 先删除再插入头部，而非直接修改顺序，是双向链表最简洁的移动方式。
② removeNode：删除任意节点（通用化链表删除）
  • 双向链表删除节点的标准写法：只需知道节点引用，无需遍历找前驱/后继；
  • 因哨兵节点的存在，即使删除的是链表唯一节点，也不会出现node.pre/node.next为空的情况。
③ putHead：插入节点到头部（通用化链表插入）
  • 插入顺序是LRU的关键：新节点/被访问的节点必须插入到哨兵节点的next（链表头部），保证“最近使用”的节点在头部；
  • 四行代码的逻辑：先让原头部节点的pre指向新节点 → 新节点的next指向原头部 → 哨兵的next指向新节点 → 新节点的pre指向哨兵，顺序不可颠倒（避免链表断裂）。

（5）put方法的淘汰逻辑：hashMap.size() > capacity

  • 淘汰规则：缓存满时，删除链表尾部节点（dummy.pre），因为尾部是“最久未使用”的节点；
  • 必须同时操作哈希表和链表：仅删除链表节点会导致哈希表存在无效映射，仅删除哈希表会导致链表存在无效节点。

4、算法性能分析

4.1 时间复杂度：O(1)（get/put均满足）

  • get：哈希表查找O(1) + 链表删除O(1) + 链表插入O(1) → 总O(1)；
  • put：哈希表查找/插入O(1) + 链表操作O(1) + 淘汰节点O(1) → 总O(1)；
  • 这是LRU缓存的理论最优时间复杂度，完全满足题目要求。

4.2 空间复杂度：O(capacity)

  • 空间开销主要来自：
    1. 哈希表：最多存储capacity个key→节点的映射，O(capacity)；
    2. 双向链表：最多存储capacity个节点，O(capacity)；
    3. 哨兵节点：固定开销O(1)；
  • 总空间复杂度为O(capacity)，无额外冗余开销。

4.3 核心优势（面试高频考点）

  1. 时间最优：哈希表+双向链表的组合保证所有操作O(1)；
  2. 逻辑简洁：哨兵节点避免了空指针判断，代码鲁棒性高；
  3. 符合工业标准：该实现与Redis的LRU核心逻辑一致，是面试中“最优解”的标准答案。

5、经典示例手动模拟（直观理解）

以示例中capacity=2为例，手动模拟核心步骤的链表和哈希表变化：

初始化

  • LRUCache lru = new LRUCache(2) → capacity=2，dummy自环（dummy.pre=dummy，dummy.next=dummy），hashMap={}。

步骤1：put(1,1)

  • hashMap无1 → 创建Node(1,1)，hashMap.put(1, Node(1,1))；
  • putHead(Node(1,1)) → 链表：dummy ↔ Node(1,1) ↔ dummy；
  • hashMap.size=1 ≤ 2 → 无需淘汰。

步骤2：put(2,2)

  • hashMap无2 → 创建Node(2,2)，hashMap.put(2, Node(2,2))；
  • putHead(Node(2,2)) → 链表：dummy ↔ Node(2,2) ↔ Node(1,1) ↔ dummy；
  • hashMap.size=2 ≤ 2 → 无需淘汰。

步骤3：get(1)

  • hashMap有1 → 调用getNode(1)：
    1. removeNode(Node(1,1)) → 链表：dummy ↔ Node(2,2) ↔ dummy；
    2. putHead(Node(1,1)) → 链表：dummy ↔ Node(1,1) ↔ Node(2,2) ↔ dummy；
  • 返回Node(1,1).value=1。

步骤4：put(3,3)

  • hashMap无3 → 创建Node(3,3)，hashMap.put(3, Node(3,3))；
  • putHead(Node(3,3)) → 链表：dummy ↔ Node(3,3) ↔ Node(1,1) ↔ Node(2,2) ↔ dummy；
  • hashMap.size=3 > 2 → 淘汰尾部节点：
    1. backNode = dummy.pre = Node(2,2)；
    2. hashMap.remove(2) → hashMap={1:Node(1,1), 3:Node(3,3)}；
    3. removeNode(Node(2,2)) → 链表：dummy ↔ Node(3,3) ↔ Node(1,1) ↔ dummy。

最终结果

后续的get(2)返回-1、put(4,4)淘汰Node(1,1)等步骤，均遵循上述逻辑，与示例输出一致。

6、拓展思考（面试延伸考点）

6.1 与无哨兵节点写法的对比

写法 空指针处理 代码复杂度 面试推荐度
哨兵节点版 无需处理（链表永远非空） 低 ★★★★★
无哨兵节点版 需判断head/tail是否为空 高 ★★★☆☆
  • 面试中优先写哨兵节点版：代码更简洁，不易出错，且能体现对链表优化的理解。

6.2 边界情况适配

本解法天然兼容所有边界场景：

  • capacity=1：缓存只能存1个节点，put第二个节点时直接淘汰第一个，逻辑正确；
  • key=0：哈希表支持key=0，节点存储key=0无问题；
  • 高频访问同一key：每次访问都会移到头部，不会被淘汰，符合LRU规则。

6.3 进阶拓展：LRU vs LFU

缓存策略 核心规则 实现难度 适用场景
LRU（最近最少） 淘汰最久未使用的节点 低 访问模式具有“时间局部性”（近期访问的易再次访问）
LFU（最不经常） 淘汰访问次数最少的节点 高 访问模式具有“频率局部性”（访问次数多的易再次访问）
  • 面试追问：LRU的缺陷是“突发流量可能淘汰重要节点”（如一次性访问大量新key，淘汰常用key），LFU可弥补该缺陷，但实现需额外记录访问次数（哈希表+双向链表+频率链表）。

总结

LRUCache的哨兵节点版实现是数据结构组合使用的经典案例，核心知识点总结如下（面试必背）：

  1. 数据结构选型：双向链表维护使用顺序（表头=最近使用，表尾=最久未使用），哈希表O(1)查找节点，两者协同保证所有操作O(1)；
  2. 哨兵节点：自环初始化，避免链表操作的空指针判断，简化插入/删除逻辑；
  3. 核心操作：
    • 访问节点（get/put已存在）：先删除再插入链表头部，标记为“最近使用”；
    • 新增节点：插入链表头部，哈希表记录映射，满则淘汰尾部节点；
  4. 关键细节：节点必须存储key，否则淘汰时无法删除哈希表的映射；
  5. 该实现是LRU缓存的最优解，时间复杂度O(1)、空间复杂度O(capacity)，是面试中回答LRU问题的“标准答案”。
