Notes-竞赛复盘:全国大学生智能汽车竞赛单车越野组总结

发表于2026-04-30|更新于2026-04-30|项目回顾
|字数总计:7.2k|阅读时长:22分钟|阅读量:

本文是对之前参加全国大学生智能汽车竞赛单车越野组别的一次阶段性总结。项目主体是一辆基于 CH32V307 主控的平衡单车，需要在越野场地中完成自平衡、循迹导航、坡道通过、圆环锥桶和 S 弯锥桶等任务。本人在项目中主要负责软件控制部分，包括平衡控制框架、PID 调试、传感器数据处理、GPS/IMU 导航逻辑、部分视觉伺服与参数维护等内容。相比于一篇正式实验报告，本文更偏向个人复盘：记录思路、问题、调参过程和一些当时踩过的坑。

一、前言：为什么单车越野这么折磨人

在参加智能车竞赛之前，我对“智能车”的直觉更多来自四轮车模：循迹、速度控制、转向控制、视觉识别、路径规划等问题当然都不简单，但至少车是天然稳定的。
单车越野组完全不同。
它首先要解决的不是“跑得快”，而是“别倒”。
平衡单车只有前后两个轮子，横滚方向本身是不稳定的。只要控制稍微慢一点、传感器噪声稍微大一点、电机响应稍微不及时一点，车就会立刻往一侧倒下。更麻烦的是，越野组的场地并不是干净的直线跑道，还要面对坡道、锥桶、调头、GPS 漂移、地面不平、机械结构松动等各种干扰。
所以这个项目给我的最大感受是：

平衡单车不是一个单独算法的问题，而是机械结构、传感器、驱动电路、控制周期、滤波、导航策略和调参经验共同决定的系统工程。

本文主要从软件控制角度，整理我们在项目中的实现思路和调试经验。

二、赛题任务与系统整体框架

2.1 单车越野组任务理解

我们当时的目标是让平衡单车在指定越野场地中完成往返运行，主要任务包括：

  1. 从出发区出发，沿设定路径行驶到调头区；
  2. 在调头区完成转向后返回出发区；
  3. 途中依次通过坡道、圆环锥桶、S 弯锥桶等区域；
  4. 全程保持车身平衡，不能严重偏航、撞桶或摔车。

从任务特点来看，单车越野组和普通四轮智能车最大的区别在于：

对比项 四轮车模 平衡单车
静态稳定性 天然稳定 横滚方向不稳定
控制重点 转向、速度、路径 平衡优先，其次路径
主要传感器 摄像头、电感、编码器等 IMU、GPS/北斗、霍尔/编码器、视觉
调试难点 路径识别和速度提升 平衡、导航、抗干扰同时满足
容错性 较高 很低，参数不合适就倒车

因此我们的软件策略始终围绕一个原则展开：

先保证平衡，再考虑导航；先保证能走，再考虑走得准；先保证能完赛，再考虑优化速度。

2.2 硬件与传感器组成

整车系统大致由以下部分组成：

后轮无刷电机 提供整车前进动力

软件上主要分为五个部分：

  1. 底层驱动层：GPIO、PWM、定时器、串口、SPI、IIC 等；
  2. 传感器层：IMU、GPS、霍尔/编码器、视觉模块数据读取；
  3. 数据处理层：滤波、零偏校准、角度解算、航向角融合；
  4. 控制算法层：平衡 PID、速度控制、舵机控制、GPS 路径规划；
  5. 调试交互层：OLED UI、Flash 存取、蓝牙遥控、参数在线调整。

三、软件控制整体思路

3.1 控制任务拆解

从软件角度看，整车控制并不是一个单独的 PID 就能解决的，而是多个控制任务的叠加：

控制任务 控制对象 主要输入 主要输出
横滚平衡控制 动量轮无刷电机 Roll 角、角速度、动量轮速度 动量轮 PWM
前进速度控制 后轮无刷电机 目标速度、后轮速度 后轮 PWM
舵机方向控制 前轮舵机 当前航向角、目标航向角 舵机角度
GPS 路径规划 目标点序列 当前经纬度、目标经纬度 目标航向角
姿态解算 IMU 加速度、角速度 Roll / Pitch / Yaw
视觉辅助 OPEN ART mini 锥桶位置、识别框 避障或转向修正

其中优先级最高的是横滚平衡控制。因为只要车倒了，后面的路径规划、视觉识别、速度优化全部没有意义。

四、无刷电机控制：先让电机听话

4.1 为什么不用 FOC

无刷电机控制有很多方案，FOC 当然更优雅，力矩控制也更细腻，但考虑到当时的主控算力、调试时间、硬件条件和任务需求，我们没有选择 FOC，而是采用了更直接的六步换相控制。

对竞赛来说，六步换相的优势是：

  1. 原理相对清晰；
  2. 实现复杂度较低；
  3. 对主控算力要求较小；
  4. 配合霍尔传感器能快速实现闭环换相；
  5. 对动量轮和平衡控制已经基本够用。

代价也很明显：

  1. 换相存在转矩脉动；
  2. 低速控制细腻程度不如 FOC；
  3. 电机噪声和振动会更明显；
  4. 对霍尔信号质量和换相表正确性依赖较大。

4.2 六步换相基本逻辑

BLDC 的三相绕组每次两相导通、一相悬空，按照固定顺序切换即可产生旋转磁场。关键是要根据霍尔状态判断当前转子位置，然后输出对应的三相状态。

简单来说就是：

步骤 导通状态
1 U+ V- W悬空
2 W+ V- U悬空
3 W+ U- V悬空
4 V+ U- W悬空
5 V+ W- U悬空
6 U+ W- V悬空

实际调试中，无刷电机最容易出现的问题有：

  1. 相序接错：电机抖动、不转或反转；
  2. 霍尔顺序不匹配：换相时机错误，电机发热严重；
  3. PWM 方向反了：控制输出越大，车越不稳；
  4. 死区和直通问题：驱动板发热甚至损坏；
  5. 低速抖动：平衡时电机响应不够平滑。

所以在正式上车之前，必须先单独测试电机，让它在不同占空比、不同方向下稳定运行。否则一旦把电机问题和平衡 PID 问题混在一起，调试会非常痛苦。

五、平衡控制：从并级 PID 到串级 PID

5.1 平衡控制需要哪些量

对于平衡单车来说，最核心的状态量有三个：

状态量 来源 作用
车身横滚角 Roll IMU 姿态解算 判断车身偏离机械中值多少
横滚角速度 Gyro 陀螺仪 判断车身倒下的速度和方向
动量轮转速 Speed 霍尔/编码器 判断动量轮当前运动状态

这三个量分别对应三个控制环：

  1. 角度环：车歪了多少；
  2. 角速度环：车歪得有多快；
  3. 速度环：动量轮当前转得多快。

刚开始调车时，最直观的目标是：车身稍微往右倒，动量轮就要给出一个让车身往左恢复的反作用力矩；车身稍微往左倒，也要及时反向补偿。

5.2 并级 PID：更容易调，但耦合也明显

前期我们采用的是并级 PID，也就是三个环分别计算，最后把输出相加：

并级 PID 的优点是结构简单，每个环都可以单独看效果。调试时一般按下面顺序：

  1. 先调角速度环，确认方向正确；
  2. 再调角度环，让车能短暂保持直立；
  3. 最后加速度环，让动量轮更积极地输出角动量。

我们当时的经验是：

  • 角速度环主要提供“阻尼感”；
  • 角度环决定车身回正趋势；
  • 速度环不要简单理解为抑制速度，它在这里更像正反馈，用来让动量轮获得更强响应。

调试角速度环时，可以用手左右晃动车身。如果方向正确，会明显感觉到动量轮在“反抗”你的晃动；如果方向反了，就会感觉车在帮你一起倒，这时候必须立刻反向输出。

并级 PID 最大的问题是：三个环之间虽然形式上独立，但实际输出强耦合。某个环参数稍微变大，可能会让整体响应变得非常暴躁。

5.3 串级 PID：更符合响应速度层级

后期我们尝试将控制策略优化为串级 PID。串级 PID 的核心思想是：响应越快的量越靠内层。

5.4 我的调参顺序

实际调参时，我不建议一上来就追求理论上的完美参数。平衡单车的调参更像一个迭代过程。

我的调试顺序大致是：

1. 只开角速度环

目标不是立起来，而是确认方向和阻尼。

  • 左右晃动车身；
  • 感受动量轮是否抵抗晃动；
  • 如果有“越晃越倒”的趋势，说明方向反了；
  • 参数不要太大，否则容易高频抖动。

2. 加入角度环

目标是让车能在机械中值附近短暂保持。

  • 找到车身机械中值；
  • 调整 Roll 目标值；
  • 逐渐增大角度环 Kp；
  • 观察车是否能直立 1-3 秒；
  • 如果来回猛摆，说明参数太大或滤波延迟明显。

3. 加入速度环

目标是让动量轮输出更充分，延长稳定时间。

  • 从较小参数开始；
  • 一点点增加；
  • 观察是否能从“短暂直立”变为“持续直立”；
  • 注意速度环过大时会引入振荡。

4. 再回头微调内环

平衡能立起来以后，不代表参数已经好。此时要重新微调角速度环和角度环，让车身抖动更小，响应更顺滑。

我的体会是：

不可能先把某一个环调到完美，再去调下一个环。更合理的方式是先让整个闭环跑通，再反复迭代每一层。

六、IMU 数据处理：滤波和零偏比想象中重要

6.1 姿态数据不是拿来就能用

IMU 是平衡控制最核心的传感器，但原始数据并不能直接使用。加速度计容易受振动和低频干扰影响，陀螺仪会存在零偏和积分漂移。

如果直接把原始角速度积分成角度，会遇到两个问题：

  1. 静止时角度也会慢慢漂；
  2. 车身振动时姿态角会出现明显噪声。

因此我们需要做几件事：

  1. 静止采样，估计陀螺仪零偏；
  2. 每次读取角速度时减去零偏；
  3. 对 Roll、Gyro、Speed 做滑动平均或低通滤波；
  4. 对加速度计和陀螺仪做互补滤波或卡尔曼滤波；
  5. 对异常值进行限幅或舍弃。

6.2 零偏处理

陀螺仪静止时理论输出应该是 0，但实际会有一个小偏置。我们的处理方式是启动时静止采样多次，然后取平均值作为零偏。

6.3 数据滤波

平衡控制中，滤波不能只追求“平滑”，还要考虑延迟。滤波太弱，控制输出会抖；滤波太强，车已经倒了，控制器才反应过来。

我们对不同变量采用不同思路：

数据 处理方式 原因
Roll 角 卡尔曼/互补滤波 融合加速度计和陀螺仪
Gyro 角速度 零偏校正 + 简单滤波 保留快速响应
动量轮速度 均值滤波/限幅 抑制霍尔测速毛刺
GPS 位置 距离阈值 + 后续考虑 EKF GPS 更新慢且有漂移
舵机输出 平滑滤波 避免舵机频繁大幅摆动

核心思路是：

平衡环的数据要快，导航层的数据可以稳。

七、GPS + IMU 路径规划：让车知道往哪走

7.1 基本导航逻辑

单车越野组没有传统循迹线，主要依赖 GPS/北斗、IMU 和后续视觉辅助完成路径规划。我们的基本思路是：

  1. 先人工采集路径点；
  2. 将经纬度点保存到 Flash；
  3. 运行时实时读取当前 GPS 坐标；
  4. 计算当前位置到目标点的距离和方位角；
  5. 使用 IMU 解算当前航向角；
  6. 计算目标航向与当前航向的偏差；
  7. 用 PID 控制舵机转向；
  8. 当距离目标点小于阈值时切换到下一个点。

7.3 航向角融合

GPS 可以提供航向角，但它有明显缺点：

  1. 更新频率低；
  2. 需要车运动后才比较可靠；
  3. 低速或静止时航向角不稳定。

IMU 的陀螺仪积分可以提供高频航向变化，但长期会漂移。

因此我们采用一种类似互补滤波的思路：

  1. 短时间相信陀螺仪积分；
  2. 长时间用 GPS 航向角修正；
  3. GPS 变化慢，所以修正不能太暴力；
  4. 修正系数需要实车调试。

八、锥桶与视觉：GPS 绕桶真的不稳

8.1 GPS 绕锥桶的问题

在开阔区域，GPS 采点循迹可以完成大致路线。但一旦进入锥桶区域，问题就明显了。

锥桶之间的距离可能只有 1m - 2m，而 GPS 误差在实际场地中可能接近这个量级。天气、卫星分布、采点时间间隔、场地遮挡都会影响定位稳定性。

这会导致：

  1. 舵机频繁修正；
  2. 车身受到额外扰动；
  3. 路径点实际位置偏移；
  4. S 弯区域容易穿错方向或撞桶。

所以单靠 GPS 绕桶并不可靠。我们后续考虑用 OPEN ART mini 做视觉辅助，识别锥桶位置后进行视觉伺服。

8.2 YOLOv3 锥桶识别思路

视觉部分的基本流程是：

  1. 采集锥桶图片；
  2. 数据增强，包括缩放、裁剪、镜像等；
  3. 标注锥桶目标框；
  4. 使用 YOLOv3 训练模型；
  5. 将模型部署到 OPEN ART mini；
  6. 输出锥桶中心位置和置信度；
  7. 根据识别结果修正舵机控制。

视觉识别结果可以用于两种策略：

策略 思路
触发式动作 检测到锥桶后执行预设转向动作
视觉伺服 根据锥桶中心偏差实时调整舵机
GPS + 视觉融合 GPS 负责全局路线，视觉负责局部避障
路径拟合预测 根据多个锥桶位置拟合可行路径

当时受限于硬件迭代和调试时间，视觉并没有完全成为主控制链路，但它明确指出了后续优化方向：GPS 管大方向，视觉管近距离细节。

九、OLED UI、Flash 与蓝牙遥控：调参效率很重要

9.1 为什么一定要做 UI

平衡单车调参非常频繁。如果每改一个 PID 参数都要重新编译、下载、上电测试，效率会极低。

所以我们设计了 OLED + 按键 UI，用来在线修改参数，包括：

  1. 平衡 PID 参数；
  2. 舵机 PID 参数；
  3. GPS 距离阈值；
  4. 舵机限幅；
  5. 电机速度目标；
  6. 传感器校准状态；
  7. Flash 保存与读取；
  8. 调试模式切换。

UI 看起来不是核心算法，但它极大提高了调试效率。

尤其是 Flash 存取很重要。否则每次关机参数都会丢失，第二天又要从头开始调。

9.2 蓝牙遥控的作用

蓝牙遥控主要用于：

  1. 发车；
  2. 急停；
  3. 手动控制速度；
  4. 手动控制转向；
  5. 单独测试舵机和后轮；
  6. 在 GPS 不稳定时进行人工接管。

在平衡车这种容易摔的系统里，急停非常重要。任何时候只要车身角度超过安全阈值，或者控制状态异常，都应该立刻关闭电机输出。

安全保护逻辑类似：

这个保护可能救不了每一次摔车，但至少可以减少电机继续狂转带来的二次损伤。

十、实际调试中踩过的坑

10.1 机械中值比想象中重要

平衡控制里，Roll 的目标值不一定是数学上的 0°，而是整车实际机械结构下的“最舒服位置”。

如果电池、主板、传感器、动量轮安装位置不对，机械中值就会偏。软件上虽然可以用目标角修正，但如果机械结构本身重心太高、左右不对称，调参会非常困难。

我的经验是：

软件可以补偿机械误差，但不能替机械结构背所有锅。

10.2 传感器安装位置会影响控制效果

IMU 安装得越高，车身摆动时测得的线加速度影响越明显，姿态解算也更容易受干扰。后来我们尽量将 IMU 固定在更合适的位置，并保证安装牢固。

松动的传感器比噪声更可怕，因为它会让控制器以为车身在动，实际动的是传感器本身。

10.3 参数不是越大越好

刚开始调 PID 时，很容易觉得响应不够就加 Kp。但在平衡单车上，参数过大会导致：

  1. 动量轮高频抖动；
  2. 车身左右振荡；
  3. 电机电流增大；
  4. 舵机频繁摆动；
  5. 导航误差进一步放大。

合适的参数应该是“够用但不过度”。尤其是角速度环，太小抗干扰差，太大高频抖动明显，需要在两者之间折中。

10.4 GPS 数据不能盲信

GPS 漂移是越野组里非常现实的问题。即使同一条路线，上午采点和下午运行也可能不完全一致。天气变化、卫星状态变化都会导致偏差。

所以 GPS 控制不能写得太“死”：

  1. 目标点距离阈值要合理；
  2. 舵机输出要限幅；
  3. 航向角要融合 IMU；
  4. 锥桶区域最好有视觉辅助；
  5. 路径点要根据实际场地反复修正。

10.5 控制周期被打断会直接影响平衡

OLED 刷新、串口打印、Flash 写入这些任务，如果写得不注意，都可能影响核心控制周期。

尤其是调试阶段，串口打印很容易加太多。一旦打印阻塞，平衡控制就会延迟。后来我们尽量减少核心控制中的打印，只保留必要状态，或者降低打印频率。

十一、阶段性结果与不足

11.1 已实现内容

经过多轮调试，整车基本实现了：

  1. 动量轮无刷电机六步换相控制；
  2. 后轮无刷电机驱动；
  3. IMU 姿态读取与滤波；
  4. Roll / Gyro / Speed 多环平衡控制；
  5. GPS 采点与路径点 Flash 存储；
  6. GPS + IMU 航向角融合；
  7. 舵机 PID 转向控制；
  8. OLED UI 在线调参；
  9. Flash 参数断电保存；
  10. 蓝牙遥控和急停；
  11. OPEN ART mini 锥桶识别初步验证。

11.2 仍然存在的问题

问题 表现 后续优化方向
GPS 漂移 路径偏移，锥桶区不稳定 引入视觉辅助、EKF 融合
后轮测速误差 速度偶发异常 加编码器或优化测速滤波
PID 参数未完全最优 抗干扰与抖动之间折中 系统辨识 + 更细参数整定
视觉未完全闭环 检测结果未充分参与路径规划 视觉伺服 + 路径拟合
控制耦合明显 转向会影响平衡 舵机动态零点补偿
硬件迭代影响软件 驱动、电源、结构变化导致参数重调 模块化驱动与配置管理

十二、个人收获

这次单车越野项目让我对嵌入式控制有了更真实的理解。

以前写 PID，更多是在课程实验或者仿真里看曲线；这次调平衡单车，PID 参数不再只是几个数字，而是直接对应车身是否能站住、电机是否发抖、舵机是否乱摆、车会不会摔。

我最大的几个收获是：

1. 控制系统一定要分层

平衡、速度、导航、视觉、UI 不能混在一起写。每一层都应该有清晰输入输出，否则后期调试会非常混乱。

2. 数据质量决定控制上限

滤波、零偏、异常值处理这些看似基础的内容，实际上决定了 PID 能不能发挥作用。控制器再好，输入数据乱跳也没用。

3. 调参是工程能力，不是玄学

虽然调参看起来像“试出来的”，但它背后依然有逻辑：先方向、再单环、再闭环、再耦合、再优化。每一步都应该知道自己在验证什么。

4. 软件不能脱离硬件

电源纹波、驱动板发热、IMU 安装位置、车架重心、舵机虚位都会影响软件效果。嵌入式项目里，很多“软件问题”最后都能追到硬件或机械结构上。

5. 能跑完全程比局部最优更重要

比赛项目不是论文仿真。稳定完赛永远优先于某个模块看起来很高级。GPS 不够准就加阈值，视觉没完全稳定就先做触发策略，PID 不是最优但能稳就先保留。这种取舍也是工程能力的一部分。

十三、总结

全国大学生智能汽车竞赛单车越野组对我来说是一段非常典型的“从理论走向工程”的经历。

这个项目里，我们做过动力学分析，写过无刷电机驱动，调过并级 PID 和串级 PID，处理过 IMU 零偏和滤波，做过 GPS 路径规划，也尝试过 YOLOv3 锥桶识别。每一个模块单独看都不算特别新奇，但把它们放到一辆会倒、会抖、会受场地影响的平衡单车上，难度就完全不一样了。

如果用一句话总结这次经历，我觉得是：

平衡单车的难点不只是让车立起来，而是让所有不完美的模块在同一个系统里尽可能稳定地协同工作。

后续如果继续优化，我会优先考虑三个方向：

  1. 使用编码器进一步提高速度测量稳定性；
  2. 用 EKF 或更合理的融合方式处理 GPS/IMU 数据；
  3. 将视觉识别真正接入局部路径规划，而不只是作为触发信号。

这次比赛留下了很多遗憾，但也留下了很多可以继续深入的方向。对我来说，它不只是一辆车，更像是一次完整的嵌入式控制系统训练。
