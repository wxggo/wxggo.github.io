Notes-项目复盘:苍穹外卖后端项目学习总结

发表于2026-04-30|更新于2026-04-30|项目回顾
|字数总计:6.3k|阅读时长:20分钟|阅读量:

本文是对之前在 B 站学习并完成的 苍穹外卖项目 的一次总结整理。该项目是一个比较完整的 Java 后端练手项目，主要模拟外卖平台的后台管理端和用户下单端，涉及员工管理、分类管理、菜品管理、套餐管理、店铺状态、用户登录、购物车、订单、地址簿、Redis 缓存、JWT 认证、微信小程序接口等内容。相比单纯学习 Spring Boot 语法，这个项目更接近真实业务开发流程，也让我第一次比较系统地理解了一个后端项目从接口、数据库到业务逻辑之间是如何串起来的。

一、前言

在学习 Java 后端的过程中，单独学 Spring Boot、MyBatis、Redis、JWT 这些技术点时，常常会有一种感觉：每个知识点似乎都懂一点，但不知道它们在真实项目里到底怎么组合起来。

苍穹外卖项目正好解决了这个问题。

它不是单纯写几个 Controller 接口，也不是只做增删改查，而是围绕一个完整的外卖业务场景，把后台管理端和用户端都串联起来。通过这个项目，我对后端开发中的几个问题有了更直观的认识：

  1. 一个项目的分层结构应该怎么设计；
  2. Controller、Service、Mapper 分别应该负责什么；
  3. DTO、VO、Entity 为什么要分开；
  4. 登录认证、权限校验、JWT 令牌是怎么用在项目里的；
  5. Redis 缓存应该放在哪些业务场景；
  6. 用户下单、购物车、订单状态流转这种业务应该如何拆解；
  7. 后端接口如何服务后台管理端和微信小程序端。

这篇文章主要是对学习过程的一次复盘，不会贴太多完整源码，而是重点整理项目思路和自己学到的内容。

三、项目分层与对象设计

3.1 Controller 层

Controller 层主要负责接收请求、解析参数、调用 Service，并返回统一格式的数据。

它不应该写复杂业务逻辑，更多是做“入口”和“出口”。

例如员工登录接口的逻辑大致是：

Controller 层的特点是：

  1. 接收 DTO；
  2. 调用 Service；
  3. 返回 VO；
  4. 不直接操作数据库；
  5. 不写太复杂的业务判断。

3.2 Service 层

Service 层是业务逻辑的核心。

例如新增菜品时，不只是插入一条菜品数据，还可能涉及：

  1. 保存菜品基本信息；
  2. 保存菜品口味信息；
  3. 检查分类是否存在；
  4. 设置默认状态；
  5. 清理缓存。

所以 Service 层通常是一个业务动作的完整封装。

Service 层不是简单转发 Mapper，而是负责把一次业务操作完整地组织起来。

七、项目中印象比较深的业务

7.1 菜品和口味的一对多关系

刚开始写菜品模块时，我以为新增菜品就是插入一条 dish 表数据。但实际业务中，一个菜品可能有多个口味，所以需要同时维护 dish 和 dish_flavor 两张表。

如果用户已经添加过相同菜品，并且口味也相同，就应该数量加一；如果口味不同，则应该作为新的购物车项。

7.4 缓存一致性

Redis 缓存刚开始看起来只是提高性能，但后面会遇到一个问题：

解决方式一般是：修改、删除、停售菜品时，清理对应缓存。下次用户查询时重新从数据库加载。

这让我意识到缓存并不是简单地“加上就行”，还要考虑数据什么时候失效。

八、项目调试过程中的一些问题

8.1 接口路径和参数不一致

刚开始对接前端时，经常出现接口访问失败。常见原因包括：

  1. 请求路径写错；
  2. GET 和 POST 用错；
  3. 参数名和 DTO 字段不一致；
  4. JSON 格式错误；
  5. Header 中没有携带 token。

后来习惯先用 Knife4j 或 Postman 单独测试接口，再和前端联调。

8.2 数据库字段映射问题

MySQL 中常用下划线命名，例如：

Java 中常用驼峰命名：

8.3 JWT过期或解析失败

登录相关问题也比较常见，例如：

  1. token 没传；
  2. token 过期；
  3. token 密钥不一致；
  4. 管理端和用户端 token 配置混淆；
  5. 拦截器没有放行登录接口。

调试这类问题时，最好先确认请求头中是否真的携带了 token。

8.4 Redis序列化问题

使用 RedisTemplate 时，如果序列化方式没有配置好，可能会出现存进去的数据看不懂，或者取出来类型转换异常。

这部分让我意识到，使用中间件时，配置细节也很重要。

九、个人学习收获

9.1 第一次完整理解后端项目结构

在做这个项目之前，我对后端项目的理解比较碎片化。做完之后，对整体结构清楚了很多：

这个流程看似简单，但贯穿了几乎所有业务模块。

9.2 理解了DTO、VO、Entity分层的意义

以前会觉得 DTO、VO、Entity 分开写很麻烦。现在能理解，它们是为了让不同层之间职责更清晰。

数据库表怎么设计，不一定等于前端需要什么；前端传什么，也不一定能直接作为数据库实体保存。

9.3 业务逻辑比语法更重要

项目里最难的并不是某个 Java 语法，而是业务逻辑：

  1. 下单时要检查哪些数据；
  2. 删除菜品前要不要判断是否关联套餐；
  3. 套餐停售时是否影响用户端展示；
  4. 订单状态能不能直接修改；
  5. 缓存什么时候应该删除。

这些问题更接近真实开发。

9.4 学会了接口调试流程

通过这个项目，我熟悉了后端接口开发的基本流程：

  1. 设计接口路径；
  2. 定义 DTO；
  3. 编写 Controller；
  4. 实现 Service；
  5. 编写 Mapper 和 SQL；
  6. 使用 Knife4j 测试接口；
  7. 联调前端；
  8. 处理异常情况。

这套流程以后做其他项目也能复用。

9.5 对Redis和JWT有了实际认识

以前学 Redis 和 JWT 时只是知道概念。项目中真正用到后，才知道它们分别适合解决什么问题：

技术 解决的问题
JWT 用户登录状态和身份认证
Redis 高频数据缓存和临时状态保存
ThreadLocal 当前请求用户信息传递
AOP 公共逻辑统一处理
事务 保证多表操作一致性

10.1 权限控制更细化

目前管理端主要是登录认证，后续可以加入更细粒度的权限控制，例如：

  1. 超级管理员；
  2. 普通员工；
  3. 订单管理员；
  4. 菜品管理员。

不同角色拥有不同接口权限。

10.2 订单支付流程更真实

学习项目中的支付流程通常会做简化。后续可以进一步接入更完整的微信支付流程，包括：

  1. 预支付订单；
  2. 支付回调；
  3. 签名验证；
  4. 支付状态查询；
  5. 退款处理。

10.3 缓存策略更完善

目前缓存主要用于菜品、套餐、店铺状态等数据。后续可以考虑：

  1. 缓存预热；
  2. 缓存穿透处理；
  3. 缓存击穿处理；
  4. 缓存雪崩处理；
  5. 分布式锁。

项目中可以进一步完善：

  1. 全局异常处理；
  2. 操作日志；
  3. 错误日志记录；
  4. 接口耗时统计；

十一、总结

苍穹外卖项目对我来说，是一个比较完整的 Java 后端入门实战项目。

它覆盖了从后台管理端到用户端的完整业务流程，也让我把 Spring Boot、MyBatis、MySQL、Redis、JWT、Swagger、微信小程序接口等知识点串联了起来。

如果用一句话总结这个项目：

苍穹外卖让我从“会写接口”进一步理解了“如何围绕业务写一个完整后端项目”。

通过这个项目，我不仅练习了增删改查，更重要的是理解了业务模块之间的关系，比如菜品和口味、套餐和菜品、用户和购物车、订单和订单明细、登录认证和当前用户上下文等。

后续如果继续学习 Java 后端，我会重点补充以下内容：

  1. Spring Security 或 Sa-Token 权限框架；
  2. Redis 高级缓存问题；
  3. RabbitMQ 消息队列；
  4. 分布式事务；
  5. Docker 部署；
  6. 项目上线与服务器运维；
  7. 更规范的接口设计和异常处理。

苍穹外卖不是终点，但它确实是一个很适合作为后端学习阶段性总结的项目。
