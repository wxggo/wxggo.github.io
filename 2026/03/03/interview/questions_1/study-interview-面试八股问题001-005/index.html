Question001: Java 中有哪些集合类？请简单介绍

知识点详解

Java 集合主要分为两大体系：Collection 接口下的单列集合、Map 接口下的键值对集合，面试重点关注底层结构、性能、使用场景。

  • Collection 常用子接口

    • List：有序、可重复
      • ArrayList：数组实现，随机访问快，中间插入/删除效率低
      • LinkedList：双向链表，头尾增删高效，随机访问慢
    • Set：不可重复
      • HashSet：基于 HashMap，查找/添加 O(1)，无序
      • LinkedHashSet：保留插入顺序
      • TreeSet：自然排序，底层红黑树
    • Queue：用于任务调度
      • ArrayDeque：数组双端队列
      • ConcurrentLinkedQueue：无锁线程安全队列
      • DelayQueue：适合定时任务

  • Map 键值对体系

    • HashMap：数组+链表/红黑树，非线程安全，性能优秀
    • Hashtable：全方法加锁，性能差，基本弃用
    • ConcurrentHashMap：分段锁+CAS，高并发首选
    • LinkedHashMap：保留插入顺序，常用于 LRU 缓存
    • TreeMap：支持按键排序

核心总结

  1. ArrayList 查找快、增删慢
  2. LinkedList 增删快、查找慢
  3. 单线程用 HashMap，并发用 ConcurrentHashMap
  4. 需排序用 TreeSet/TreeMap
  5. 需保留插入顺序用 LinkedHashMap

30秒口述版

Java 集合分 Collection 和 Map 两大类。Collection 下有 List、Set、Queue，List 里 ArrayList 基于数组、查询快，LinkedList 基于链表、增删快；Set 用来去重，HashSet 性能最高。Map 是键值对，HashMap 日常使用最多，并发场景用 ConcurrentHashMap，需要有序可以用 LinkedHashMap 或 TreeMap。

Question002: MySQL 的索引类型有哪些？

知识点详解

MySQL 以 InnoDB 为主，索引基于 B+ 树 实现，常见索引类型如下：

  1. 主键索引（聚簇索引）
    叶子节点存储整行数据，一张表只能有一个，查询与范围查询效率最高。

  2. 二级索引（普通索引）
    叶子节点存储主键值，查询需要先查二级索引，再回表查聚簇索引。

  3. 唯一索引
    列值唯一，允许一个 NULL，创建唯一约束时自动生成。

  4. 联合索引
    多列组合索引，遵循最左前缀原则。

  5. 全文索引
    用于文本关键词搜索，实际项目多用 Elasticsearch 替代。

  6. 前缀索引
    对字符串前 N 个字符建索引，节省空间，但要保证区分度。

重要优化：覆盖索引
查询字段全部包含在索引中，无需回表，可大幅提升查询速度。

30秒口述版

MySQL InnoDB 的索引基于 B+ 树，主要有主键索引、二级索引、唯一索引、联合索引、全文索引和前缀索引。主键索引存整行数据，二级索引存主键，需要回表。联合索引要遵循最左前缀，覆盖索引可以避免回表，是常用的优化手段。

Question003: MySQL 索引的最左前缀匹配原则是什么？

知识点详解

最左前缀匹配是联合索引的核心规则：
MySQL 会从联合索引最左列开始连续向右匹配，遇到范围查询（>、<、BETWEEN、LIKE 等）或列中断时，停止匹配。

以索引 (a, b, c) 为例：

  • 可命中索引：a、a+b、a+b+c
  • 不可命中：b、c、b+c
  • 范围查询截断：a=1 AND b>2 AND c=3 只用到 a+b，c 不生效

建索引经验
等值查询放前面，范围查询放后面，高频查询字段靠左。

30秒口述版

最左前缀原则就是联合索引必须从第一列开始连续使用，不能跳过中间列。遇到大于、小于这类范围查询时，后面的列就无法使用索引了。所以建联合索引时，要把等值查询的字段放前面，范围查询放后面。

参考链接：https://www.mianshiya.com/bank/1860871861809897474/question/1780933295450648578

Question004: 在 MySQL 中建索引时需要注意哪些事项？

知识点详解

索引会提升查询速度，但会降低写入性能，建索引需遵守以下原则：

  1. 选择区分度高的列，避免 gender 这类低区分度字段
  2. 联合索引严格遵循最左前缀原则
  3. 不在索引列上使用函数或计算，避免索引失效
  4. 尽量使用覆盖索引，减少回表 IO
  5. 字符串使用前缀索引，控制长度与区分度
  6. 大表加索引使用在线 DDL，避免锁表
  7. 定期清理冗余与未使用索引

30秒口述版

建索引要选区分度高的列，联合索引注意最左前缀，不要在索引列上做函数操作。尽量用覆盖索引避免回表，字符串可以建前缀索引。大表加索引要在线操作防止锁表，同时定期清理没用的冗余索引，避免拖累写入性能。

参考链接：https://www.mianshiya.com/bank/1860871861809897474/question/1780933295459037187

Question005: MySQL 中的事务隔离级别有哪些？

知识点详解

事务隔离级别用于解决并发问题：脏读、不可重复读、幻读。MySQL 支持四种标准隔离级别：

  1. 读未提交（Read Uncommitted）
    可读到未提交数据，存在脏读，基本不用。

  2. 读已提交（Read Committed）
    只能读已提交数据，解决脏读，存在不可重复读，是 PostgreSQL 默认级别。

  3. 可重复读（Repeatable Read）
    MySQL 默认级别，通过 MVCC 实现，解决不可重复读，InnoDB 用间隙锁大幅降低幻读。

  4. 串行化（Serializable）
    最高级别，事务串行执行，无并发问题，但性能极差。

业务常用：可重复读、读已提交。

30秒口述版

MySQL 有四种事务隔离级别：读未提交、读已提交、可重复读、串行化。读未提交会脏读，读已提交解决脏读但有不可重复读。可重复读是 MySQL 默认，用 MVCC 实现，能解决不可重复读。串行化最安全但性能最差，金融场景常用可重复读，普通业务用读已提交即可。
