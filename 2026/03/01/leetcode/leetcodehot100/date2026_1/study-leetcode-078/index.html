Leetcode-078:子集-DFS回溯经典解法（基础最优版）

Created2026-03-01|Updated2026-03-01|Java学习笔记Leetcode Hot100
|Word Count:3.2k|Reading Time:11mins|Post Views:

前言

子集问题是Leetcode数组模块的经典高频题，也是面试中考察「回溯算法（DFS）」的核心题目之一。本题的核心难点在于如何枚举数组的所有子集（包括空集），且不重复、不遗漏。其中「DFS回溯+选/不选」是最直观、最易理解的基础最优解，该解法通过对每个元素做「选」或「不选」的递归决策，结合「恢复现场」的操作，完整遍历所有子集可能性。

1、题目

给定一个整数数组 nums，数组中的元素互不相同。返回该数组所有可能的子集（幂集）。

示例

示例 1：
输入：nums = [1,2,3]
输出：[[],[1],[2],[1,2],[3],[1,3],[2,3],[1,2,3]]
解释：输出包含数组所有可能的子集，顺序不做要求。

示例 2：
输入：nums = [0]
输出：[[],[0]]
解释：空集和仅包含0的子集是唯一可能的结果。

提示

  • 1 <= nums.length <= 10
  • -10 <= nums[i] <= 10
  • nums 中的所有元素 互不相同

2、分析

2.1 问题核心：如何枚举所有子集？

子集的本质是「数组元素的任意组合（包括空组合）」，每个元素有且仅有两种状态：选入当前子集 或 不选入当前子集。

比如 nums = [1,2]，对1的决策是「选」或「不选」，对2的决策也是「选」或「不选」，所有决策组合恰好对应4个子集：

  • 1不选 + 2不选 → []
  • 1选 + 2不选 → [1]
  • 1不选 + 2选 → [2]
  • 1选 + 2选 → [1,2]

这是回溯解法的核心依据——通过递归遍历每个元素的「选/不选」决策，覆盖所有子集可能。

2.2 核心难点：如何避免重复/遗漏 & 理解恢复现场？

暴力解法的思路是，枚举所有可能的元素组合（如双层循环枚举长度1、2的子集），但该方法仅适配短数组，且无法通用（数组长度变化时需修改循环层数）。

DFS回溯+选/不选 是解决该难点的最优思路：

  1. 递归遍历每个元素，对当前元素做「选」或「不选」的决策；
  2. 「选」决策后需「恢复现场」（移除已选元素），保证递归返回后不影响后续「不选」决策的执行；
  3. 递归终止条件为「遍历完所有元素」，此时当前路径即为一个完整子集；
    整体时间复杂度为O(n×2ⁿ)（2ⁿ个子集，每个子集复制耗时O(n)），完全适配题目输入规模。

2.3 核心解题思路

  1. 初始化结果集与路径：ans 存储所有子集，path 存储当前递归路径（即当前正在构建的子集）；
  2. DFS递归遍历：
    • 递归参数：i（当前遍历到的元素下标）、nums（原数组）、ans（结果集）、path（当前路径）；
    • 终止条件：i == nums.length（遍历完所有元素），将path的副本加入ans；
    • 不选当前元素：直接递归下一个元素（i+1）；
    • 选当前元素：将元素加入path → 递归下一个元素 → 恢复现场（移除该元素）；
  3. 返回结果集：递归结束后，ans 包含所有子集，直接返回。

2.4 关键疑问解答（对应代码注释）

疑问1：为什么选了元素后不直接添加到ans？

递归过程中，path 是「动态构建的临时路径」：选元素1后，path = [1]，此时递归处理元素2，若直接添加path到ans，会提前记录不完整的路径（如[1]还未处理元素2的决策）。只有当遍历完所有元素（i == nums.length），path 才是一个完整的子集，此时添加副本才能保证正确性。

疑问2：选、不选、恢复现场的顺序是怎么确定的？

  • 「不选」优先：先处理「不选当前元素」的分支，递归到下一层，再处理「选」的分支，符合「从左到右遍历元素」的直觉；
  • 「恢复现场」必须在「选元素+递归」之后：选元素后，path 被修改，递归返回后需要将元素移除，才能保证后续「不选」分支的path 是干净的（比如处理完元素1的「选」分支后，移除1，才能正确处理元素1的「不选」分支）。

3.2 代码核心点逐行解释

（1）初始化结果集与路径

  • ans：最终返回的所有子集，需存储每个完整路径的副本；
  • path：递归过程中动态构建的临时子集（引用类型），避免频繁新建列表，提升效率；
  • 递归起点：从下标0开始，即第一个元素的决策。

（2）DFS终止条件

  • 核心逻辑：当i等于数组长度时，所有元素的决策已完成，path 是一个完整子集；
  • 关键细节：必须新建ArrayList<>(path)添加到ans——若直接添加path，后续递归修改path会导致ans中的元素同步变化（引用传递特性）。

（3）不选当前元素分支

  • 逻辑：放弃当前元素，直接处理下一个元素（i+1），path 无任何修改；
  • 优先级：先处理「不选」分支，符合「从左到右遍历元素」的直觉，也保证「选」分支的恢复现场不影响该分支。

（4）选当前元素分支 + 恢复现场

  • 选元素：将nums[i]加入path，此时path包含当前元素；
  • 递归下一层：处理后续元素的决策；
  • 恢复现场：递归返回后，移除刚加入的元素——这是回溯算法的核心，保证path回到「选元素前」的状态，不影响后续其他分支的执行。

4、算法性能分析

本解法是子集问题的基础最优解，兼顾效率和理解性，完全适配题目输入规模。

4.1 时间复杂度：O(n×2ⁿ)

  • n：数组长度；
  • 2ⁿ：子集的总数（每个元素有选/不选两种可能）；
  • 核心耗时：每个子集生成时，需要复制path到ans，复制操作耗时O(n)；递归遍历的总次数为2ⁿ（每个子集对应一次递归终止）；
  • 适配性：面对n≤10的数组长度，2¹⁰=1024，总操作数为10×1024=10240，完全无性能压力。

4.2 空间复杂度：O(n)

  • 空间开销主要来自：
    1. 递归调用栈的深度（最多为n，对应遍历到最后一个元素时的栈层级）；
    2. path 列表的长度（最多为n，即选入所有元素时）；
  • 必要性：递归栈和path是回溯算法的基础开销，该空间复杂度为最优值（无法进一步降低）。

4.3 核心优势

  1. 逻辑直观：完全贴合「每个元素选/不选」的子集定义，递归思路易理解、易推导；
  2. 代码简洁：核心逻辑仅20行左右，无冗余操作，面试书写效率高；
  3. 无重复子集：因元素互不相同且按下标遍历，不会生成重复子集；
  4. 扩展性强：是理解组合、排列类回溯问题的基础，掌握后可快速迁移至「组合总和」「全排列」等题目。

6、拓展思考（面试高频考点）

6.1 边界情况天然适配

本解法无需额外处理各类边界场景，代码天然兼容：

  1. 空数组：nums = []，递归直接触发终止条件，ans = [[]]，正确；
  2. 单元素数组：如nums = [0]，返回[[], [0]]，正确；
  3. 多元素数组：如nums = [1,2,3]，覆盖所有8个子集，正确。

6.4 恢复现场的必要性（面试必问）

若不移除「path.remove(path.size() - 1)」，会导致path 累积所有选过的元素，最终生成错误子集：

  • 比如nums = [1,2]，选1后递归处理2，选2后path = [1,2]，递归返回后未移除2，后续处理「不选2」分支时，path 仍为[1,2]，最终生成错误子集。
  • 结论：恢复现场是回溯算法的「灵魂」，保证递归分支之间的独立性。

7、总结

子集问题的DFS回溯+选/不选解法是本题的基础最优解，也是理解回溯算法的核心，本文核心总结如下：

  1. 子集的核心特征是「每个元素选/不选」，递归遍历所有决策组合是最直观的解法；
  2. 回溯算法的核心是「决策+恢复现场」：选元素后必须移除，保证递归分支的独立性；
  3. 递归终止条件是「遍历完所有元素」，此时需添加path的副本到结果集（避免引用传递问题）；
  4. 面试中可通过「迭代解法」「去重逻辑」拓展解题思路，体现对问题的深度理解；
  5. 解法具备强扩展性，是掌握组合、排列类回溯问题的关键基础。
