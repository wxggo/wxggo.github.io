Question016: 说说 TCP 的四次挥手？

知识点详解

TCP四次挥手是TCP连接可靠释放的标准流程，核心目的是保证双工通信的两端都能安全关闭读写通道，避免数据丢失，关键要点如下：

  1. 挥手核心流程与状态变化：
    • 第一次挥手：客户端→服务端，发送FIN=1报文，表明客户端无数据要发送，客户端进入FIN_WAIT_1状态；
    • 第二次挥手：服务端→客户端，返回ACK=1, ack=客户端FIN序列号+1报文，确认客户端的关闭请求，此时进入半关闭状态（服务端仍可向客户端发数据），客户端进入FIN_WAIT_2，服务端进入CLOSE_WAIT；
    • 第三次挥手：服务端→客户端，发送FIN=1报文，表明服务端无数据要发送，服务端进入LAST_ACK状态；
    • 第四次挥手：客户端→服务端，返回ACK=1, ack=服务端FIN序列号+1报文，客户端进入TIME_WAIT，服务端收到后关闭连接，客户端等待2MSL后彻底关闭。
  2. 关键细节：
    • TIME_WAIT的核心作用：防止最后一个ACK丢包导致服务端重传FIN失败，等待2倍最大报文生存时间（2MSL），确保网络中残留的该连接报文完全消失；
    • 高并发问题：短连接场景下客户端易堆积大量TIME_WAIT状态，占用端口资源，可通过开启SO_REUSEADDR、调整tcp_tw_reuse等内核参数缓解。

30秒口述版

TCP四次挥手是为了可靠关闭双工的TCP连接，客户端先发FIN，服务端回ACK进入半关闭；等服务端发完数据再发FIN，客户端回ACK后进入TIME_WAIT状态，等2MSL后关闭。TIME_WAIT是防止最后一个ACK丢包，高并发下这个状态堆积会占端口，可调内核参数优化，四次挥手的核心是保证双向读写通道都安全关闭。

Question017: TCP 超时重传机制是为了解决什么问题？

知识点详解

TCP超时重传是保障数据可靠传输的核心机制，专门解决网络传输中数据包丢失、ACK确认丢失的问题，屏蔽底层网络的不可靠性，关键要点如下：

  1. 核心解决问题：网络丢包（数据包/ACK回执丢失），确保发送方能检测到丢包并重新传输，最终让应用层收到完整数据；
  2. 核心流程：
    • 发送方发送报文段后立即启动定时器（超时时间为RTO）；
    • 若定时器超时前收到ACK，清除定时器；
    • 若超时未收到ACK，判定数据包丢失，触发重传，并采用指数退避策略（重传等待时间加倍），避免频繁重试加剧网络拥堵；
  3. 动态超时时间（RTO）：
    • RTO并非固定值，会根据网络实际往返时间（RTT）动态调整；
    • 例如初始RTT为200ms，RTO设为200ms，若网络变慢RTT升至500ms，RTO也会同步调整，避免误判重传；
  4. 极限情况：连续重传15次仍无响应时，TCP会主动断开连接，防止无效资源占用。

30秒口述版

TCP超时重传主要解决网络丢包问题，发送方发数据后启动定时器，超时没收到ACK就重传，还会根据RTT动态调整超时时间RTO，重传时等待时间指数加倍。这个机制对上层透明，能保证数据不丢失，连续重传多次失败就会断开连接，是TCP可靠传输的关键。

Question018: TCP/IP 四层模型是什么？

知识点详解

TCP/IP四层模型是网络通信的分层设计范式，将复杂的网络通信拆解为四层，每层专注解决特定问题，实现职责隔离与解耦，各层核心职责、典型协议/设备如下：

  1. 链路层（数据链路层/网络接口层）：
    • 核心：处理物理介质的传输，负责MAC地址寻址、帧的封装/解封装；
    • 典型设备/协议：交换机、以太网、Wi-Fi；
  2. 网络层（网际层）：
    • 核心：实现主机到主机的寻址与路由，负责IP地址分配、数据包转发；
    • 典型协议/设备：IP、ICMP（网络控制）、ARP（地址解析）、路由器；
  3. 传输层：
    • 核心：提供端到端的通信能力，区分不同应用的通信；
    • 典型协议：TCP（可靠字节流）、UDP（高效无连接）；
  4. 应用层：
    • 核心：为具体业务场景提供通信协议，直接对接应用程序；
    • 典型协议：HTTP/HTTPS、FTP、SMTP、DNS。

核心逻辑：数据传输时会逐层封装（应用层→传输层→网络层→链路层），接收时逐层解封装，比如HTTP请求会先封装TCP头，再封装IP头，最后封装以太网帧头后发送。

30秒口述版

TCP/IP四层模型把网络通信拆成链路层、网络层、传输层、应用层。链路层管物理传输和MAC，网络层靠IP和路由器做主机寻址，传输层用TCP/UDP做端到端通信，应用层是HTTP这些业务协议。数据传输时逐层封装，接收时逐层解包，每层各司其职，拆解复杂的网络通信问题。

Question019: 什么是服务雪崩？

知识点详解

服务雪崩是微服务架构下的级联故障现象，核心是单个服务故障引发整个调用链的系统性崩溃，关键要点如下：

  1. 核心成因：
    • 基础服务（如Redis、数据库）故障/响应变慢，导致依赖它的上游服务请求堆积；
    • 同步阻塞调用+无熔断/限流机制，上游服务线程池被占满，无法处理新请求，进而故障向上扩散；
  2. 典型场景：
    • 订单服务调用Redis查数据，Redis响应超时，订单服务线程一直阻塞等待，线程池耗尽后无法处理用户请求；
    • 用户服务调用订单服务失败，线程池也被占满，最终整个用户下单链路崩溃；
  3. 解决思路：
    • 超时控制：为每个依赖调用设置最大等待时间，避免线程长期阻塞；
    • 限流/熔断：使用Hystrix、Sentinel等工具，限制并发数，故障时快速失败（熔断）；
    • 降级：故障时返回兜底数据（如默认库存、缓存数据），避免服务完全不可用。

30秒口述版

服务雪崩是微服务里的级联故障，一个基础服务响应慢或挂了，上游服务调用它的线程阻塞堆积，线程池满了就处理不了新请求，然后一层一层往上崩。核心原因是同步阻塞+无熔断限流，解决办法是加超时控制、用Sentinel/Hystrix做熔断限流，再配合降级策略，防止单点故障扩散。

Question020: MyBatis 中 #{} 和 ${} 的区别是什么？

知识点详解

MyBatis中#{}和${}都是参数占位符，但处理机制、安全特性、适用场景差异极大，核心要点如下：

  1. #{}：预编译参数占位符（推荐）：

    • 处理机制：MyBatis将#{}替换为?，通过PreparedStatement的setXxx()方法赋值，参数参与预编译而非SQL解析；
    • 核心优势：能有效防止SQL注入，参数值会被自动转义，性能也更优；
    • 示例代码： 最终执行SQL：SELECT * FROM user WHERE id = ?，参数id通过setInt()赋值。

  2. ${}：字符串直接拼接：

    • 处理机制：直接将变量值原样插入SQL，不经过预编译，参数参与SQL解析；
    • 核心风险：存在严重SQL注入风险，若参数可控性差（如用户输入），可能导致数据泄露/删除；
    • 适用场景：仅用于动态表名、动态排序字段等无法预编译的场景，且必须做白名单校验；
    • 示例代码（风险场景）： 若sortColumn传入id; DROP TABLE user;，会执行恶意SQL。

  3. 核心原则：能使用#{}时绝不使用${}；使用${}必须确保参数完全可控（如内部枚举、白名单过滤）。

30秒口述版

MyBatis里#{}是预编译占位符，会转成?通过PreparedStatement赋值，能防SQL注入，是首选；${}是直接字符串拼接，有注入风险，只能用在动态表名、排序字段这些没法预编译的场景，还要做白名单校验。核心区别就是预编译vs直接拼接，安全优先级远高于便利性。
