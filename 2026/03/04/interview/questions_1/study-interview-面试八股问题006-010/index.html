Question006: MySQL 默认的事务隔离级别是什么？为什么选择这个级别？

知识点详解

MySQL 默认的事务隔离级别是可重复读（Repeatable Read），该级别在数据一致性和并发性能之间实现了较好的平衡，也是 InnoDB 引擎的核心设计考量：

  1. 解决核心并发问题：
    • 杜绝脏读：不会读取其他未提交事务的数据，满足生产环境最基础的一致性要求；
    • 解决不可重复读：通过 MVCC（多版本并发控制）为事务提供一致的快照，避免同一事务内多次查询结果不一致。
  2. 对幻读的优化：
    可重复读本身不彻底解决幻读，但 InnoDB 引入间隙锁 + 行锁组合，大幅遏制幻读问题，无需升级到串行化级别。
  3. 性能与场景适配：
    • 对比串行化（完全串行执行，性能极差），可重复读保留了大部分并发能力；
    • 大部分业务场景（电商库存、用户积分等）无需绝对杜绝幻读，该级别完全满足需求；
    • 金融核心账务等极致一致性场景，可手动升级为串行化。

Question007: MySQL 中 count(*)、count(1) 和 count(字段名) 有什么区别？

知识点详解

三者均用于统计行数，但底层逻辑和语义存在关键差异，直接影响结果准确性和性能：

  1. count(*)：
    • 标准行计数方式，统计所有行（包含NULL值）；
    • InnoDB 优化器会选择成本最低的索引树遍历，无需全表扫描，性能最优且语义最准确。
  2. count(1)：
    • 1 是常量，每一行都能取到值，效果与 count(*) 几乎一致；
    • 执行计划和性能与 count(*) 无明显差异，可互换使用。
  3. count(字段名)：
    • 仅统计该字段非NULL的行，若字段允许NULL，结果会少于总行数；
    • 语义与前两者不同，易引发统计错误（如 count(email) 会漏掉无邮箱用户）。

核心建议：统计总行数优先用 count(*)，避免使用 count(字段名) 除非明确需要过滤NULL值。

30秒口述版

count(*)是标准的总行数统计，包含NULL值且优化最好；count(1)和count(*)效果几乎一样；count(字段名)只统计非NULL行，容易出错。查总行数就用count(*)，别用count(字段名)。

Question008: 什么是分库分表？分库分表有哪些类型（或策略）？

知识点详解

分库分表是解决单库单表数据量/并发瓶颈的分布式存储方案，核心是将数据拆分到多个库/表，提升存储和吞吐能力，主要分为四类：

  1. 垂直分库：
    按业务模块拆分（如用户库、订单库、商品库），适配微服务架构，降低库间耦合。
  2. 垂直分表：
    将宽表按字段拆分（如用户表拆分为登录信息表+扩展信息表），减少I/O开销，适配“冷热字段分离”场景。
  3. 水平分表：
    同结构表按规则打散数据（如用户表按user_id取模分16张表），降低单表数据量，适配写多读少、数据增长快的场景。
  4. 水平分库：
    水平分表基础上跨数据库实例拆分（如订单数据按order_id分到8个MySQL实例），是分布式拆分的核心方式。

关键要点：

  • 分片键（Sharding Key）决定数据均匀性和查询效率，常用策略：取模、范围、一致性哈希；
  • 中间件（ShardingSphere、MyCat）通过解析SQL+路由规则实现自动寻址；
  • 拆分后需解决跨库事务、全局排序/聚合等问题（如柔性事务、ES汇总查询）。

30秒口述版

分库分表是解决单库单表瓶颈的方案，分垂直和水平两类：垂直分库按业务拆、垂直分表按字段拆；水平分表把同表数据打散，水平分库再跨实例拆分。分片键选得好才均匀，拆分后要处理跨库事务、全局查询的问题。

Question009: Redis 通常应⽤于哪些场景？

知识点详解

Redis 是高性能的内存数据库，核心适配“高频读写、低延迟、临时数据”场景，典型应用如下：

  1. 热点数据缓存：
    存储电商商品详情、首页榜单等热点数据，查询延迟从几十毫秒降至亚毫秒级，扛高并发。
  2. 分布式Session：
    结合 Spring Session 实现集群环境下用户登录状态共享，避免单点Session丢失。
  3. 分布式锁：
    基于 SETNX 或 RedLock 实现，解决多服务节点资源竞争问题（需合理设置锁过期时间，避免误删）。
  4. 排行榜/计数：
    利用 ZSet 按分数排序特性，实现微博热搜、游戏积分榜等场景；String 类型的 INCR 可实现接口调用计数。
  5. 简易消息队列：
    通过 List 结构的 LPOP/RPOP、LPUSH/RPUSH 实现生产消费模型，适合轻量场景（复杂场景推荐 RocketMQ/Kafka）。
  6. 接口限流：
    结合 Lua 脚本实现原子操作，统计单位时间内接口调用次数，超过阈值则拒绝请求，保护后端服务。

注意事项：Redis 持久化（RDB/AOF）有性能损耗，不建议作为主数据库使用，聚焦临时/高频数据场景。

30秒口述版

Redis主要用在热点数据缓存，能大幅提升查询速度；还能做分布式Session、分布式锁，排行榜用ZSet实现，也能做简易消息队列和接口限流。它适合高频读写场景，不适合当主库。

Question010: Redis 中常⻅的数据类型有哪些？

知识点详解

Redis 支持多种核心数据结构，每种适配特定业务场景，核心类型如下：

  1. 字符串（String）：
    最基础类型，可存储文本、序列化对象，支持 INCR/DECR 原子计数（如接口调用次数统计）。
  2. 列表（List）：
    底层双向链表实现，支持头尾增删（LPUSH/RPOP），适合简易消息队列；随机访问性能差，不建议当数组使用。
  3. 集合（Set）：
    无序、不重复的字符串集合，底层哈希表实现（增删查 O(1)），适配用户标签管理、社交关系去重等场景。
  4. 有序集合（Sorted Set）：
    Set 基础上加 score 排序，member 唯一、score 可重复，适配排行榜（ZADD 加数据、ZRANGE 取排名）。
  5. 哈希（Hash）：
    适合存储对象（如用户信息），field 对应属性名、value 存值，可单独更新字段，节省带宽和内存。

扩展类型：Stream（消息队列）、Geo（地理定位）、Bitmap（位图）等，适配特定小众场景。

30秒口述版

Redis核心数据类型有五种：String做计数器，List做简易消息队列，Set去重，ZSet实现排行榜，Hash存储对象字段。这五种日常用得最多，Stream、Geo这些是扩展类型，用得少。
