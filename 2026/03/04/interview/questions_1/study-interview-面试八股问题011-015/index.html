Question011: 你在项目中使用的 Redis 客户端是什么？

知识点详解

在 Java 生态中，常用的 Redis 客户端主要有 Jedis 和 Lettuce。目前 Spring Boot 2.0 以后默认集成了 Lettuce。
Lettuce（首选）：

  • 技术底层：基于 Netty 实现，支持响应式编程。
  • 连接模型：采用长连接，多个线程可以共享同一个连接，资源利用率高。
  • 生态配合：与 Spring Data Redis 配合非常顺手，API 清晰（如 ReactiveRedisTemplate）。

Jedis：

  • 技术底层：默认是同步阻塞的。
  • 连接模型：每个操作通常独占一个连接，在高并发下连接数容易打满。
  • 现状：除轻量场景外，目前基本不再作为首选。

避坑与实践经验：

  1. 连接泄露案例：曾因直接使用 Lettuce 异步 API 未处理好 Future 回调导致 Netty 线程堆积，建议统一使用 RedisTemplate 封装。
  2. 生产配置：必须配置连接池（通过 LettucePoolingClientConfiguration 启用）。
  3. 粒度控制：大 key 或频繁批量操作要控制粒度，避免阻塞主线程。
  4. 安全监控：上线前务必开启 SSL 并设置合理的超时时间。

30秒口述版

项目里Redis客户端首选Lettuce，它基于Netty实现，支持异步和响应式编程，多线程能共享单个长连接，也是Spring Boot 2.0后的默认客户端，和Spring Data Redis配合很顺畅；Jedis是早期的同步阻塞客户端，线程不安全，高并发下容易打满连接，现在基本不用。生产环境要给Lettuce配连接池，避免连接泄漏，还要控制大key操作的粒度。

Question012: 到底什么是 TCP 连接？

知识点详解

TCP连接并非物理通信线路，而是通信双方操作系统内核中维护的一对状态上下文，核心作用是为可靠传输提供状态支撑，关键要点如下：

  1. 连接的本质：内核中socket对象保存的核心状态信息，包括缓冲区大小、滑动窗口、序列号、收发状态等，是实现可靠传输的“上下文容器”。
  2. 连接的建立与断开：
    • 建立：通过三次握手交换双方初始序列号（ISN），确认彼此收发能力，最终双方状态机进入ESTABLISHED；
    • 断开：通过四次挥手关闭连接，因TCP是双工协议，读写通道需分别确认关闭。
  3. 高并发场景问题与优化：
    • Nginx、Redis等高并发服务易出现TIME_WAIT状态堆积、文件描述符耗尽问题，本质是连接资源未及时释放；
    • 需调优内核参数（如开启SO_REUSEADDR、调整tcp_tw_reuse）。
  4. 核心目标：通过状态同步，支撑TCP的序列号、确认应答、超时重传等机制，最终实现可靠的字节流传输。

30秒口述版

TCP连接不是物理线路，而是双方操作系统内核里维护的状态信息，比如序列号、缓冲区、滑动窗口这些。建立连接要三次握手，断开要四次挥手，因为TCP是双工的。高并发服务最怕TIME_WAIT堆积或文件描述符不够，需要调内核参数，而TCP连接的核心就是为了做状态同步，保证数据的可靠传输。

Question013: TCP 是用来解决什么问题？

知识点详解

TCP是传输层协议，核心解决不可靠IP网络下的可靠字节流传输问题，屏蔽底层网络的丢包、乱序、重复、中断等问题，让数据能完整、有序地从一端交付到另一端，其核心实现机制分四大类：

  1. 可靠传输机制：为每个字节分配序列号，接收方返回ACK确认已接收的字节范围，超时未收到ACK则触发重传，保证数据不丢、不重复、不乱序。
  2. 滑动窗口机制：摒弃“发一个等一个ACK”的低效模式，允许发送方在窗口范围内连续发送数据，大幅提升传输效率。
  3. 流量+拥塞控制：
    • 流量控制：接收方通过窗口字段告知发送方自身处理能力，防止发送方数据压满接收方缓冲区；
    • 拥塞控制：通过Reno/Cubic等算法感知网络拥堵，主动降低发送速率，避免网络拥塞加剧。
  4. 连接管理机制：通过三次握手建立连接（同步初始序列号，避免历史连接干扰），四次挥手关闭连接（确保双向读写通道都正常关闭）。

场景适配：TCP为上层提供“可靠的字节流管道”，适用于对数据准确性要求高的场景（如HTTP、数据库连接、文件传输）；而音视频通话、直播等容忍少量丢包但要求低延迟的场景，更适合使用UDP。

30秒口述版

TCP主要解决的是不可靠IP网络下的可靠传输问题，底层网络会丢包、乱序，TCP通过序列号、确认应答、滑动窗口这些机制，保证数据完整、有序地交付。它还做了流量控制和拥塞控制，避免压垮接收方或造成网络拥堵，像HTTP、数据库这些要求零丢包的场景都用TCP，核心是为上层屏蔽底层网络的复杂性。

Question014: TCP 和 UDP 有什么区别？

知识点详解

TCP和UDP均为传输层协议，核心差异是设计目标不同：TCP追求数据可靠性，UDP追求传输效率与实时性，二者关键特性、使用场景对比如下，且代码实现上也存在明显差异。

对比维度 TCP UDP
连接特性 面向连接，需三次握手 无连接，直接发送
传输可靠性 可靠，重传/排序/校验 不可靠，发后即忘
状态维护 有状态，维护连接上下文 无状态，数据包独立处理
头部开销 至少20字节 仅8字节
适用场景 HTTP、数据库、文件传输 音视频、DNS、DHCP、游戏
广播/组播 不支持 天生支持

TCP和UDP的核心区别是TCP追求可靠，UDP追求效率和实时性。TCP需要三次握手建立连接，有重传、排序机制，保证数据准确，适合HTTP、数据库这些场景；UDP无连接，发后即忘，延迟低，头部开销小，适合音视频、游戏这些能容忍少量丢包的场景。没有谁更好，只是场景适配不同。

Question015: 说说 TCP 的三次握手？

知识点详解

TCP三次握手是建立可靠连接的必经流程，核心目的是同步双方的初始序列号（ISN），并双向验证彼此的收发能力，确保连接建立后能正常传输数据，具体流程、状态变化与关键细节如下：

  1. 三次握手具体流程

    • 第一次握手：客户端→服务端，发送SYN=1, seq=x的报文，请求建立连接，客户端进入SYN_SENT状态；
    • 第二次握手：服务端→客户端，返回SYN=1, ACK=1, ack=x+1, seq=y的报文，确认客户端的序列号，同时同步自身初始序列号，服务端进入SYN_RCVD状态；
    • 第三次握手：客户端→服务端，发送ACK=1, ack=y+1的报文，确认服务器的序列号，双方最终进入ESTABLISHED状态，连接建立完成。

  2. 关键细节

    • 为何不能两次握手：两次握手无法验证服务器的接收能力，若服务端的ACK报文丢失，客户端无法知晓服务端是否准备就绪，直接发数据会造成资源浪费，三次握手能确保双向通道均通畅；
    • 超时重传：若SYN报文丢失，客户端会触发超时重传，一般重试5次，重传间隔从1秒开始指数增长；
    • 高并发优化：高并发场景下易出现半连接队列（SYN队列） 满的问题，导致新连接请求被丢弃，需调大内核参数somaxconn和tcp_max_syn_backlog；
    • 最后一次ACK丢失：服务器会重发SYN+ACK报文，客户端收到后会重新回复ACK，避免连接卡住。

30秒口述版

TCP三次握手的核心是同步双方的初始序列号，验证双向收发能力。客户端先发SYN包，服务端回SYN+ACK包，客户端再回ACK包，三次完成后双方进入连接状态。不能两次是因为没法验证服务器的接收能力，高并发时SYN队列容易满，需要调大somaxconn这些内核参数，丢包的话会触发超时重传。
