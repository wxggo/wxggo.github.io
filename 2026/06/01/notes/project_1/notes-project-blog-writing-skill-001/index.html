Notes-项目复盘:Blog-writing Skills工具包设计与生成流程整理

发表于2026-06-01|更新于2026-06-01|项目回顾
|字数总计:6.4k|阅读时长:20分钟|阅读量:

本文是对 Blog-writing Skills 工具包 设计与生成过程的一次复盘整理。这个工具包的目标不是简单写一篇博客，而是把我常见的学习材料整理流程沉淀成一个可复用的 Agent 能力模块：当我后续输入 LeetCode 代码、八股问题、面经截图、项目链接、Word/PDF 文件或论文文献时，Agent 可以按照固定风格自动生成一篇完整的个人 Blog Markdown 文件。

这次整理主要记录：Agent 的 Skill 模块有什么作用、为什么要设计成工具包、刚才 Blog-writing Skill 是如何拆分和生成的，以及后续应该如何调用和迭代。

一、前言

最近我在整理个人学习 Blog 时，遇到一个很明显的问题：不同类型的内容虽然主题不同，但整理方式其实有很多重复部分。

比如：

  • 刷 LeetCode 时，需要整理题目、思路、代码、复杂度和边界情况；
  • 准备面试八股时，需要把问题整理成知识点详解和 30 秒口述版；
  • 看面经时，需要先提取题目，再整理成简洁回答、重点说明和面试可背版；
  • 复现项目时，需要写项目背景、技术栈、核心模块、复现过程、调试问题和个人收获；
  • 阅读论文时，需要关注研究方向、创新点、方法设计、实验结果和后续启发。

如果每次都从零开始写提示词，不仅效率低，而且很难保证文章风格统一。因此这次我尝试把这些写作规则整理成一个 Blog-writing Skills 工具包，让 Agent 在面对类似任务时可以自动加载固定规范，并按照我的个人 Blog 风格输出 Markdown 文件。

我的理解是：

Skill 的价值不在于“替我写一次文章”，而在于把一套稳定的处理流程沉淀下来，让 Agent 每次都能按相同标准完成任务。

二、Agent 的 Skill 模块是什么

2.1 Skill 的基本理解

Skill 可以理解为 Agent 的一个“能力插件”或“任务说明书”。

普通对话中，我们往往需要在每次提问时重复说明：

而 Skill 的作用，就是把这些重复要求固化成一个可复用模块。之后只要触发这个 Skill，Agent 就能知道：

  1. 什么时候应该使用这套流程；
  2. 输入内容可能有哪些类型；
  3. 每种类型应该如何处理；
  4. 输出文件应该是什么命名规则；
  5. Markdown 文章应该包含哪些固定结构；
  6. 哪些情况需要联网、读文件、看图片或检查 PDF；
  7. 最终应该如何交付 .md 文件。

所以，Skill 不是简单的 Prompt，而是由 主说明文件 + 参考文档 + 脚本 + 示例样式 组成的一套小型工作流。

2.2 Skill 与普通 Prompt 的区别

我觉得 Skill 和普通 Prompt 最大的区别在于：

对比项 普通 Prompt Skill 工具包
使用方式 每次手动描述需求 通过名称或触发条件自动调用
规则保存 存在于当前对话中 存在于 Skill 文件结构中
适合场景 一次性任务 高频、重复、风格固定的任务
输出稳定性 容易受当次提示影响 更容易保持统一格式
可维护性 修改困难，容易遗漏 可拆分为多个参考文件迭代
工程化程度 偏临时对话 偏可复用工作流

因此，如果任务只是临时问答，不一定需要 Skill；但如果任务具有明显的重复性、格式约束和分类流程，就非常适合封装成 Skill。

这次 Blog-writing 就属于典型的 Skill 场景：它不是只生成一篇文章，而是要长期帮助我整理不同类别的学习 Blog。

三、Blog-writing Skill 的设计目标

这次工具包的核心目标可以概括为一句话：

当我输入任意学习材料时，Agent 能自动判断材料类型，并按照我的个人博客风格生成一篇完整、可发布、可复习的 Markdown 文章。

具体来说，它需要覆盖七类内容：

类型 输入内容 输出目标
LeetCode 刷题类 题目、代码、解法 生成算法学习 Blog
八股问题类 一组面试问题 生成知识点总结 + 30 秒口述版
面经类 图片、截图、题目列表 提取问题并整理答案
链接阅读分析类 项目文章、公众号链接、网页 阅读链接并生成项目学习复盘
文件内容整理类 Word/PDF/项目报告 生成项目实验总结 Blog
文献整理类 PDF 论文、学位论文 生成文献阅读笔记
其他内容 零散学习材料 参考最接近模板整理

这个设计其实很像给 Agent 做一个“个人知识整理助手”。

它不只是把文字润色一下，而是要完成下面这条链路：

4.1 SKILL.md：入口和总控说明

SKILL.md 是整个 Skill 的入口文件，主要负责告诉 Agent：

  4. 应该按什么默认流程执行；
  5. 遇到不同类别内容时应该去看哪些参考文件。

4.2 references：分类规则与写作规范

references 目录是这个 Skill 最核心的知识区，里面放的是写作规则，而不是最终文章。

我把它拆成了几个文件：

文件 作用
style-guide.md 规定整体语气、frontmatter、封面、Markdown 风格
category-workflows.md 规定不同内容类型的判断逻辑、命名规则和文章结构
source-handling.md 规定链接、图片、代码、Word/PDF、论文等输入如何处理
output-templates.md 提供通用模板、八股模板、面经模板、LeetCode 模板等

这样拆分的好处是：

  1. SKILL.md 不会变得过长；
  2. Agent 可以按需读取相关规则；
  3. 后续维护时可以只改某一个文件；
  4. 不同类型任务之间不会相互干扰。

例如，当用户提供的是八股问题时，Agent 重点参考 category-workflows.md 中的“Interview 八股 blog”部分和 output-templates.md 中的“Interview question block”；当用户提供的是论文 PDF 时，则重点参考“Paper / literature reading blog”部分和 source-handling.md 中的论文处理规则。

比如 frontmatter 缺字段、代码块没闭合、残留 TODO 这类问题，本质上都适合用脚本检查。这样可以降低生成文章时的格式错误率。

五、刚才 Skill 的生成流程复盘

整个 Blog-writing Skill 的生成过程可以拆成下面几步。

5.1 第一步：分析用户需求

最开始的需求其实比较完整：用户已经明确给出了七类文章场景，并提供了六份参考 Markdown 样式。

这些样式文件分别对应：

  1. LeetCode 刷题文章；
  2. 面试八股问题文章；
  3. 面经学习文章；
  4. 链接项目复盘文章；
  5. 文件/课程实验复盘文章；
  6. 论文阅读文章。

因此这次不需要再反复追问输出格式，而是可以直接从样例中抽取规则，包括：

  • frontmatter 字段；
  • 标题命名方式；
  • 分类和标签风格；
  • 正文标题层级；
  • 代码块和表格使用方式；
  • 个人学习总结的语气；
  • 每类文章的固定模块。

这一步相当于把“隐含在样例中的风格”显式抽象出来。

5.2 第二步：抽象通用 Blog 风格

从参考样式中可以看到，所有文章都有类似的基本结构：

正文通常会以一段引用开头，用来说明本文的目的：

然后进入主体部分，并在最后加入个人收获、总结或核心流程伪代码。

这说明 Blog-writing Skill 的底层风格不是“新闻文章”，也不是“百科说明”，而是：

研究生个人学习笔记风格，重点关注理解、复盘、可背诵、可回顾和可迁移。

所以在 style-guide.md 中，我专门强调了这些写作要求：

  1. 用自然的中文解释；
  2. 保留第一人称学习反思；
  3. 不要堆砌原文；
  4. 不要写成过于官方的报告；
  5. 要适当加入学习收获和后续优化方向；
  6. 输出必须是完整 Markdown，而不是大纲。

5.4 第四步：设计输入处理规则

Blog-writing 面对的输入并不只有纯文本，还可能包括：

  1. Java / Python / C++ 代码；
  2. 面经截图；
  3. 公众号链接；
  4. Word 文档；
  5. PDF 项目报告；
  6. PDF 论文；
  7. 混合输入材料。

所以我又设计了 source-handling.md，专门说明不同输入应该怎么处理。

例如：

输入类型 处理方式
代码 识别题目和算法，优化代码解释，补复杂度和边界情况
图片 提取可见题目，不清楚的地方说明不确定性
链接 尽量访问链接，若不可访问则提示解析失败，不能编造内容
Word/PDF 提取正文，必要时查看图表，改写为个人复盘风格
论文 识别研究方向、创新点、方法、实验、局限和参考文献

这部分规则解决的是“信息从哪里来、可信度如何处理”的问题。

六、Skill 设计原则总结

通过这次 Blog-writing Skill 的设计，我觉得一个好用的 Skill 至少要遵循下面几个原则。

6.1 触发条件要清晰

Skill 的描述要明确告诉 Agent：什么时候该用它，什么时候不该用它。

比如 Blog-writing 的触发场景应该包括：

6.2 主文件要短，细节要拆分

SKILL.md 应该像“总控文件”，负责说明整体流程和资源导航，而不是把所有规则都塞进去。

这样做的好处是：

  1. 结构更清楚；
  2. 修改更方便；
  3. Agent 可以按需读取；
  4. 后续扩展新文章类型时不会破坏原规则。

6.3 样例比抽象描述更重要

这次 Skill 能设计得比较具体，一个重要原因是有参考 Markdown 文件。

仅仅说“写成个人博客风格”其实很模糊，但给出 study-leetcode-141.md、notes-project-aicode-001.md、notes-paper-HRL-001.md 这类样例后，就能抽取出非常明确的规则：

  1. 标题怎么写；
  2. frontmatter 怎么写；
  3. 段落长度如何控制；
  4. 哪些地方用表格；
  5. 哪些地方写个人收获；
  6. 代码和伪代码怎么放；
  7. 总结部分应该写到什么程度。

所以我理解到：

设计 Skill 时，与其写很多抽象要求，不如提供几个高质量输入输出样例。

6.4 能程序化检查的地方就用脚本

文章内容质量需要模型判断，但格式完整性可以用程序检查。

比如：

  • frontmatter 是否存在；
  • 必要字段是否齐全；
  • 是否有未闭合代码块；
  • 是否有 TODO 占位；
  • 公式是否需要 mathjax: true。

这些检查不复杂，但能明显提升输出稳定性。

这也是 Skill 和普通 Prompt 的区别之一：Skill 可以携带脚本，把部分不稳定的自然语言任务变成确定性检查。

6.5 输出必须面向真实使用

Blog-writing 的最终输出不是“看起来像文章”，而是要能直接用于博客发布。

因此必须满足：

  1. 文件命名符合习惯；
  2. frontmatter 完整；
  3. Markdown 可直接复制；
  4. 文章结构完整；
  5. 没有空标题和 TODO；
  6. 有个人理解和总结；
  7. 如果用户需要，还要生成实际 .md 文件。

这提醒我：设计 Skill 时要从最终交付物倒推，而不是只关注中间回答过程。

九、这次设计中的个人理解

9.1 Skill 是把经验沉淀成流程

以前写 Blog 时，我更多依赖临时 Prompt。现在通过 Skill，我能把自己的写作经验拆成稳定流程。

例如：

  • 八股必须有 30 秒口述版；
  • 面经必须有面试可背版；
  • 项目复盘必须有个人收获；
  • 论文阅读必须重点关注方法；
  • 链接不可访问时不能编造；
  • 公式出现时要加 mathjax: true。

这些规则如果每次手动提醒，很容易遗漏；但放进 Skill 后，就可以长期复用。

9.2 Skill 让 Agent 更像一个专用助手

没有 Skill 时，Agent 更像一个通用聊天模型；有了 Skill 后，它会更像一个专门负责某类任务的助手。

Blog-writing Skill 的角色就是：

它不需要解决所有问题，只需要把“学习材料 → 个人 Blog”这件事做好。

这种思路也可以迁移到其他方向，比如：

  1. 论文精读 Skill；
  2. 面试复习 Skill；
  3. 项目 README 生成 Skill；
  4. 实验报告复盘 Skill；
  5. 代码仓库分析 Skill。

9.3 好的 Skill 需要不断迭代

这次生成的 Blog-writing 只是第一版。真正好用的 Skill 应该在后续使用中继续调整。

比如我后面可能会发现：

  1. 某类文章标题还不够统一；
  2. 论文阅读部分需要增加“方法流程图”；
  3. 八股题需要增加“易错点”；
  4. 面经题需要增加“追问概率”；
  5. 项目复盘需要增加“可放进简历的项目亮点”；
  6. Markdown 校验脚本还可以检查图片路径是否为空。

这些都可以继续修改 Skill 的 reference 文件或脚本，而不用每次重新写一长串提示词。

十、后续优化方向

后续如果继续完善 Blog-writing Skill，我觉得可以从以下方向扩展。

10.1 增加更多文章模板

目前主要覆盖 LeetCode、八股、面经、项目、文件、论文六类。后续还可以增加：

  1. 源码阅读类；
  2. 开源项目学习类；
  3. 课程笔记类；
  4. 工具使用教程类；
  5. 实验结果分析类；
  6. 简历项目包装类。

10.2 增强 Markdown 校验脚本

当前脚本主要检查基础格式，后续可以继续检查：

  1. 标题编号是否连续；
  2. 是否存在空链接；
  3. 图片路径是否缺失；
  4. 代码块语言是否标注；
  5. frontmatter 日期格式是否正确；
  6. 是否符合对应类别的必选章节。

这样可以让生成结果更接近真正可发布状态。

10.3 加入个人知识库规范

目前 Skill 主要学习的是文章样式，后续还可以加入个人知识库规范，例如：

  1. 常用技术栈名称；
  2. 常用分类目录；
  3. 图片存放路径规则；
  4. Blog 编号递增规则；
  5. 常用参考链接来源；
  6. 不同研究方向的关键词库。

这样生成的 Blog 会更贴合我自己的长期写作体系。
