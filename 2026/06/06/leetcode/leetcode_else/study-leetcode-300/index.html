Leetcode-300:最长递增子序列-动态规划与贪心二分

第二种是贪心加二分：

维护一个数组 g，其中 g[i] 表示长度为 i + 1 的递增子序列的最小结尾值。

为什么要维护“最小结尾值”？

因为在长度相同的情况下，结尾越小，后面越容易接上更大的数字，形成更长的递增子序列。

例如有两个长度为 3 的递增子序列：

虽然长度都为 3，但第二个序列结尾是 4，比 8 更小，后面更容易继续扩展。

因此，贪心二分的核心就是：

对每个数字 x，找到 g 中第一个大于等于 x 的位置，用 x 替换它。

如果找不到这样的位置，说明 x 比当前所有结尾都大，可以把最长长度加一。

4、算法性能

解法 时间复杂度 空间复杂度 适合理解程度
动态规划 O(n²) O(n) 最容易理解
贪心 + 二分 O(n log n) O(1) 或 O(n) 更适合优化

动态规划需要两层循环，所以是 O(n²)。

贪心二分中，每个元素只遍历一次，每次通过二分查找位置，所以是 O(n log n)。

如果像代码中一样复用 nums 数组作为 g，额外空间可以认为是 O(1)；如果单独开一个 g 数组，则空间复杂度是 O(n)。

6、扩展思考

这道题最容易混淆的是“子序列”和“子数组”。

子数组要求元素必须连续，而子序列不要求连续，只需要保持原来的相对顺序。

另外，贪心二分写法中维护的 g 数组也容易误解。它并不是用来保存最终答案路径的，而是用来保存“不同长度递增子序列的最小结尾”。

如果题目只要求返回长度，贪心二分就足够了。

如果题目要求返回具体的最长递增子序列，则还需要额外记录前驱节点和位置关系，不能只靠当前这份 g 数组。

动态规划解法帮助我理解“以谁结尾”的状态设计；贪心二分解法则进一步说明，很多动态规划问题在只关心长度时，可以通过维护某种最优信息来降低复杂度。

总结

Leetcode 300 是一道非常经典的动态规划题，也是一道很好的二分优化题。

核心结论可以总结为：

动态规划关注每个位置作为结尾时的最优长度；贪心二分关注每种长度下尽可能小的结尾值。

面试中如果时间有限，可以先写 O(n²) 的动态规划，保证思路正确；如果面试官继续要求优化，再写 O(n log n) 的贪心二分版本。
