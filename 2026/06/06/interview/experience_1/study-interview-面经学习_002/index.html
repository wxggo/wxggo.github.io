Question002：手撕算法：有效的括号

题目描述

给定一个只包括 (、)、{、}、[、] 的字符串 s，判断字符串是否有效。

有效字符串需要满足：

  1. 左括号必须用相同类型的右括号闭合；
  2. 左括号必须以正确的顺序闭合；
  3. 每个右括号都有一个对应的相同类型的左括号。

简洁回答

这题可以用栈解决。遍历字符串时，如果遇到左括号就入栈；如果遇到右括号，就判断栈是否为空，以及栈顶左括号是否和当前右括号匹配。如果不匹配，直接返回 false。遍历结束后，只有栈为空才说明所有括号都正确闭合。

重点说明

这道题的核心是“后出现的左括号要先闭合”，这正好符合栈的后进先出特性。需要注意三个边界情况：

  1. 字符串长度为奇数，必然不合法；
  2. 遇到右括号时栈为空，说明没有对应左括号；
  3. 遍历结束后栈不为空，说明还有左括号没有闭合。

面试可背版

这题我会用栈做。因为括号匹配要求最近出现的左括号最先被闭合，符合栈的后进先出特性。遍历字符串时，左括号入栈，右括号就取栈顶判断是否匹配。如果栈为空或者类型不匹配，返回 false。最后判断栈是否为空，空才说明全部匹配成功。时间复杂度 O(n)，空间复杂度最坏 O(n)。

Question003：企业场景中大量请求打到 Redis，应该怎么处理？

简洁回答

企业场景下如果有成千上万个请求打到 Redis，需要先判断请求类型：是读多还是写多，是不是热点 Key，是否存在大量缓存未命中。整体可以从应用层、缓存设计、Redis 访问优化、架构扩展和监控几个方面处理。

重点说明

可以从以下几个层次回答：

层次 应对方案
应用层 接口限流、削峰、降级、熔断
缓存设计 热点数据预热、合理 TTL、TTL 随机化、本地缓存
Redis 访问优化 连接池、Pipeline、批量查询、避免 big key 和慢命令
架构层 主从读写分离、哨兵高可用、Redis Cluster 分片
监控层 监控 QPS、连接数、命中率、慢查询、内存、网络带宽、延迟

结合商城项目，菜品、套餐、店铺状态属于典型读多写少数据，适合放入 Redis。用户端高频访问时优先读缓存，后台修改数据后删除或更新缓存，从而降低 MySQL 压力。

面试可背版

我会先区分请求是读多还是写多、是不是热点 Key、是否有大量缓存未命中。如果是大量读请求打到 Redis，首先在缓存设计上要做好热点数据预热、合理 TTL、TTL 加随机值，避免同一时间大量过期；其次在应用层可以做限流、降级和削峰；访问 Redis 时要合理配置连接池，尽量用批量操作或 Pipeline，避免 big key、慢命令和 keys 这种阻塞命令。如果单机 Redis 承载不住，可以进一步做主从读写分离、哨兵高可用或者 Redis Cluster 分片扩展。同时还要监控 QPS、连接数、命中率、慢查询、内存和网络带宽等指标。

延伸追问

  • Redis 为什么快？
  • Redis 是单线程吗？为什么单线程还可以支撑高并发？
  • Redis 连接数打满怎么办？
  • Redis 热点 Key 怎么发现和治理？
  • Redis Cluster 如何做数据分片？

Question004：Redis 吞吐量剧增会出现什么情况？QPS 怎么理解？

简洁回答

Redis 吞吐量突然剧增时，可能出现延迟升高、请求超时、连接数打满、CPU 飙高、网络带宽打满、慢查询增多等情况。严重时，如果缓存未命中率升高，请求还可能继续打到 MySQL，造成数据库压力。

QPS 是每秒请求数。在 Redis 中更常关注每秒执行命令数，也就是 ops/sec。一个业务接口的 QPS 和 Redis 的 ops/sec 不一定相等，因为一个接口可能会访问多次 Redis。

重点说明

例如：

  • 某接口 QPS 是 10000；
  • 每次请求需要查 3 次 Redis；
  • 那么 Redis 侧可能承受约 30000 ops/sec。

因此排查 Redis 压力时，不能只看业务接口 QPS，还要看具体命令类型、单次请求访问 Redis 的次数、缓存命中率和网络流量。

常见监控指标：

指标 含义
instantaneous_ops_per_sec Redis 每秒操作数
connected_clients 当前连接数
used_memory 内存使用量
keyspace_hits / keyspace_misses 缓存命中和未命中次数
slowlog 慢查询日志
evicted_keys 被淘汰的 key 数量

面试可背版

Redis 吞吐量剧增时，可能会出现响应延迟升高、请求超时、连接数打满、CPU 或网络带宽打满、慢查询增加等问题。如果大量请求没有命中缓存，还可能进一步把压力传递到 MySQL。QPS 可以理解为每秒请求数，但业务接口 QPS 和 Redis 的 ops/sec 不一定相同，因为一个接口可能访问多次 Redis。所以排查时我会结合命中率、连接数、慢查询、内存、网络和具体命令类型一起分析。

延伸追问

  • Redis 的慢查询如何查看？
  • Redis CPU 打满可能是什么原因？
  • Redis 网络带宽打满怎么办？
  • 如何减少一个接口对 Redis 的访问次数？
  • Pipeline 的作用是什么？

Question005：缓存击穿除了加锁和热点 Key 不过期，还有什么更好的方案？

简洁回答

除了互斥锁和热点 Key 永不过期，更好的方案是“逻辑过期 + 异步刷新”。缓存数据本身不依赖 Redis 的物理过期时间，而是在 value 中保存一个逻辑过期时间。请求发现数据逻辑过期后，仍然先返回旧数据，同时只让一个后台线程异步重建缓存。

重点说明

缓存击穿的典型场景是：某个热点 Key 过期的瞬间，大量并发请求同时访问该 Key，发现缓存不存在，于是同时打到数据库。

常见解决方案对比：

方案 优点 缺点
互斥锁 保证只有一个线程重建缓存 其他线程可能阻塞，影响响应时间
热点 Key 不过期 简单，避免过期瞬间击穿 数据更新依赖主动删除或刷新
逻辑过期 + 异步刷新 请求不阻塞，可用性好 短时间内可能返回旧数据
热点 Key 预热 避免冷启动高并发打库 需要提前知道热点数据
本地缓存 减少 Redis 压力 数据一致性更复杂

逻辑过期适合读多写少、允许短时间弱一致性的场景，比如菜品列表、店铺状态、首页配置等。

面试可背版

除了加锁和热点 Key 不过期，我会考虑逻辑过期加异步刷新。具体做法是缓存 value 中保存数据和逻辑过期时间，请求过来后如果数据没过期就直接返回；如果逻辑上过期，也先返回旧数据保证请求不阻塞，然后只让一个线程去异步重建缓存。这样热点 Key 过期时不会有大量请求阻塞等待，也不会全部打到数据库。缺点是短时间内可能读到旧数据，所以更适合对强一致性要求不高的热点读场景。

延伸追问

  • 缓存穿透、缓存击穿、缓存雪崩有什么区别？
  • 布隆过滤器解决什么问题？
  • 热点 Key 永不过期有什么风险？
  • 逻辑过期如何保证只有一个线程重建缓存？
  • 缓存和数据库如何保证一致性？

Question006：项目过程中有没有遇到比较慢的 SQL？

简洁回答

可以诚实回答：项目不是生产级大流量系统，没有遇到真实线上慢 SQL 事故，但在订单模块设计时关注过慢 SQL 风险，尤其是订单列表按用户、状态、时间分页查询的场景。

重点说明

订单列表查询是商城项目中比较容易产生慢 SQL 的场景。例如用户端查询订单列表时，可能按照 user_id、status、order_time 过滤和排序；商家端查询订单时，可能按照 shop_id、status、order_time 查询。

如果没有合适索引，数据量上来后可能会出现：

  1. 全表扫描；
  2. 排序使用 filesort；
  3. 分页 offset 较大时查询变慢；
  4. 查询字段过多，回表成本高。

优化思路：

排查慢 SQL 时，可以用 EXPLAIN 查看：

字段 关注点
type 访问类型，尽量避免 ALL
key 实际使用的索引
rows 预估扫描行数
Extra 是否出现 filesort、temporary

面试可背版

我的项目不是生产级大流量系统，所以没有遇到真实线上慢 SQL 事故，但在订单模块设计时我有关注这个问题。比如订单列表查询，如果按照用户、订单状态、下单时间倒序分页查询，数据量上来后如果没有合适索引，就可能出现全表扫描或者 filesort。我的处理思路是先用 EXPLAIN 看执行计划，重点关注 type、key、rows、Extra。如果发现没有走索引，就会根据查询条件建立组合索引，例如用户端可以考虑 (user_id, status, order_time)，商家端可以考虑 (shop_id, status, order_time)。同时尽量只查必要字段，避免 select *，深分页场景可以考虑基于 id 或时间游标优化。

延伸追问

  • EXPLAIN 中 type 字段有哪些常见取值？
  • 联合索引的最左前缀原则是什么？
  • 哪些情况会导致索引失效？
  • 深分页为什么慢？如何优化？
  • 覆盖索引是什么？
  • 回表是什么？

Question007：Agent 项目都是你自己做的吗？主要做了哪些工作？

这个问题本质是在确认项目真实性。回答时不能只说“是我自己做的”，而要按照模块拆分自己的工作。

可以把 Agent 项目拆成四层：

  1. 后端服务层：SpringBoot + LangChain4j + Qwen；
  2. 知识库层：Markdown 加载、切分、Embedding、语义召回；
  3. 工具调用层：Tool Calling、MCP 外部搜索；
  4. 交互部署层：Flux + SSE 流式回答、博客嵌入、跨域配置、ECS 部署。

重点说明

这个项目的目标是把个人博客文章、项目笔记和学习资料接入 RAG 知识库，构建一个可以交互问答的数字人 Agent。

项目回答时要突出“不是简单套壳调用大模型 API”，而是完成了完整的 RAG 应用链路：

面试可背版

这个项目是我个人主导完成的，核心链路都是我自己做的。它的目标是把我的个人博客文章、项目笔记和学习资料接入 RAG 知识库，做成一个可以交互问答的数字人 Agent。主要工作分成几块：第一，后端服务层，我用 SpringBoot 和 LangChain4j 封装 Agent 服务，接入 Qwen 模型，并做了 Prompt 身份约束和会话记忆；第二，知识库层，我设计了 Markdown 文档加载和预处理流程，包括按段落切分、重叠窗口、Embedding 向量化和语义召回；第三，工具调用层，我接入了 Tool Calling 和 MCP 外部搜索能力，让 Agent 可以做资料补充和复杂问题分步回答；第四，交互层，我用 Flux 和 SSE 实现了流式回答，前端可以实时渲染输出，最后完成了博客嵌入、跨域配置和 ECS 部署。

Question008：Agent 项目最困难的是哪个模块？

简洁回答

最困难的模块可以回答为：RAG 知识库构建与召回质量优化。因为这部分最能体现项目深度，也能说明自己不是只调 API。

重点说明

一开始如果只是简单按照固定长度切分 Markdown，容易出现两个问题：

  1. chunk 太大：召回内容噪声多，模型上下文中无关信息变多；
  2. chunk 太小：上下文不完整，模型回答容易缺少关键信息。

因此可以结合 Markdown 的标题、段落结构做切分，并保留标题等元信息。同时设置重叠窗口，使每个 chunk 既不会太长，也尽量保留上下文连续性。

除了切分，召回后的 Prompt 约束也很重要，需要提示模型尽量基于知识库回答。如果知识库中没有相关内容，应明确说明不知道，而不是自由发挥。

面试可背版

我觉得最困难的是 RAG 知识库构建和召回质量优化。一开始如果只是简单按固定长度切分 Markdown，会出现两个问题：切得太大，召回内容噪声多；切得太小，上下文不完整，模型回答容易缺信息。所以我后面会结合 Markdown 的标题、段落结构做切分，并保留标题等元信息，同时设置重叠窗口，让每个 chunk 既不会太长，也尽量保留上下文。另外，召回后还要通过 Prompt 约束模型尽量基于知识库回答，如果上下文没有相关信息，就明确提示不知道，而不是让模型自由发挥。从工程角度看，SSE 流式返回也有一些细节，比如前端实时渲染、连接中断、跨域部署等，但整体上我认为最核心的难点还是知识库质量和召回准确率。

延伸追问

  • 为什么要设置重叠窗口？
  • 如何评估召回效果？
  • 如何减少大模型幻觉？
  • 如果召回不到正确内容，如何排查？
  • RAG 和微调有什么区别？

Question009：如果把 Agent 项目拓展到企业场景，有哪些不足？

简洁回答

个人博客 Agent 更像是一个原型系统，如果拓展到企业知识库，需要重点补齐权限控制、数据接入、检索质量、稳定性、成本控制、安全和可观测性。

重点说明

企业级 Agent 需要考虑的问题比个人项目复杂很多：

方向 需要补齐的能力
权限控制 按用户、部门、角色、项目做文档权限隔离
数据接入 接入飞书、Confluence、Git、PDF、数据库等多源数据
数据更新 增量同步、版本管理、删除同步、失败重试
检索质量 关键词检索 + 向量检索 + rerank + 元数据过滤
稳定性 超时控制、重试、限流、降级、熔断
成本控制 模型调用缓存、token 控制、模型分级调用
安全性 防 Prompt 注入、敏感信息过滤、输出合规检查
可观测性 记录召回内容、模型输出、耗时、错误、用户反馈

面试可背版

如果从个人博客 Agent 拓展到企业知识库，我认为主要有几个不足需要补齐。第一是权限控制。个人博客内容基本都是公开的，但企业知识库有部门、角色、项目权限，RAG 检索时必须做到权限感知，不能让用户检索到无权限文档。第二是数据接入和更新机制。企业文档来源可能包括飞书、Confluence、Git、PDF、数据库等，需要做增量同步、版本管理、失败重试和文档删除同步。第三是检索质量。企业场景只做向量召回可能不够，可能要结合关键词检索、向量检索、rerank、元数据过滤，提高准确率。第四是稳定性和成本控制。大模型调用有延迟和费用，需要做限流、缓存、模型降级、超时控制、重试机制。第五是安全和可观测性。需要记录调用链路、召回内容、模型输出、用户反馈，同时防止 Prompt 注入、敏感信息泄露和不合规输出。所以我现在这个项目更像是一个个人场景下的原型，已经跑通了 SpringBoot + LangChain4j + RAG + SSE 的主链路，但如果要企业化，还要重点补权限、数据治理、检索评估、安全和可观测性。

  • 文档删除后，向量库如何同步删除？
  • 如何防止用户通过 Prompt 注入绕过限制？
  • 如何评估一个 RAG 系统回答是否准确？
  • 如何降低大模型调用成本？
  • 如果模型响应很慢，怎么优化？

1. Redis 高频问题

问题 复习重点
缓存穿透、缓存击穿、缓存雪崩分别是什么？ 定义、场景、解决方案
Redis 为什么快？ 内存存储、I/O 多路复用、单线程模型、数据结构
Redis 单线程为什么还能高性能？ 避免线程切换、事件驱动、命令执行快

Big Key 是什么？有什么危害？ 阻塞、网络传输慢、内存不均、删除风险

Redis 内存淘汰策略有哪些？ LRU、LFU、TTL、random、noeviction

问题 复习重点
MySQL 索引为什么能加速查询？ B+ 树结构、减少扫描行数

深分页为什么慢？ offset 扫描大量数据
深分页如何优化？ 游标分页、延迟关联、覆盖索引
什么是 MVCC？ undo log、ReadView、隔离级别
MySQL 事务隔离级别有哪些？ 读未提交、读已提交、可重复读、串行化

问题 复习重点

七、可直接背诵的高频回答

1. Redis 高并发回答模板

我会先区分请求是读多还是写多、是不是热点 Key、是否有大量缓存未命中。如果是大量读请求打到 Redis，首先在缓存设计上做好热点数据预热、合理 TTL、TTL 加随机值，避免同一时间大量过期；其次应用层可以做限流、降级和削峰；访问 Redis 时合理配置连接池，尽量用批量操作或 Pipeline，避免 big key、慢命令和 keys 这种阻塞命令。如果单机承载不住，可以进一步做主从读写分离、哨兵高可用或者 Redis Cluster 分片扩展。同时要监控 QPS、连接数、命中率、慢查询、内存和网络带宽等指标。

2. 缓存击穿回答模板

缓存击穿是指热点 Key 在过期瞬间，大量并发请求同时访问，发现缓存不存在，于是同时打到数据库。除了互斥锁和热点 Key 不过期，我会考虑逻辑过期加异步刷新。请求发现数据逻辑过期后，仍然先返回旧数据保证可用性，同时只让一个线程异步重建缓存。这样可以避免大量请求阻塞或打到数据库，但缺点是短时间内可能读到旧数据，适合读多写少、允许弱一致性的热点场景。

3. 慢 SQL 回答模板

我的项目不是生产级大流量系统，所以没有遇到真实线上慢 SQL 事故，但在订单模块设计时我有关注这个问题。比如订单列表查询，如果按照用户、订单状态、下单时间倒序分页查询，数据量上来后如果没有合适索引，就可能出现全表扫描或者 filesort。我的处理思路是先用 EXPLAIN 看执行计划，重点关注 type、key、rows、Extra。如果发现没有走索引，就根据查询条件建立组合索引，比如 (user_id, status, order_time)。同时避免 select *，深分页场景可以考虑基于 id 或时间游标优化。

4. Agent 项目真实性回答模板

这个项目是我个人主导完成的，核心链路都是我自己做的。它的目标是把个人博客文章、项目笔记和学习资料接入 RAG 知识库，做成一个可以交互问答的数字人 Agent。我主要做了后端服务封装、Qwen 模型接入、Prompt 身份约束、会话记忆、Markdown 文档加载和切分、Embedding 语义召回、Tool Calling、MCP 外部搜索、Flux + SSE 流式输出，以及博客嵌入和 ECS 部署。

5. Agent 企业化不足回答模板

如果从个人博客 Agent 拓展到企业知识库，我认为主要要补齐权限控制、数据接入、检索质量、稳定性、成本控制、安全和可观测性。企业知识库需要按用户、部门、角色做权限隔离；文档来源可能很多，需要增量同步、版本管理和删除同步；检索上需要结合关键词、向量检索、rerank 和元数据过滤；模型调用还要考虑限流、缓存、超时、降级和成本控制；同时需要防 Prompt 注入、敏感信息泄露，并记录召回内容、模型输出和用户反馈，方便后续评估和优化。

八、总结

这次安克创新后端开发实习生面试的核心特点是：问题都来自简历，追问集中在真实项目和企业场景。后续准备时，我需要把简历中的每一个技术点都变成可以展开讲 2-3 分钟的内容。

尤其要注意：

  1. Redis 不能只说“用来做缓存”，要能讲高并发、击穿、热点 Key、集群和监控；
  2. MySQL 不能只说“建索引”，要能结合订单场景讲慢 SQL 排查；
  3. Agent 项目不能只说“调用大模型”，要能完整讲 RAG 链路和工程化不足；
  4. 自我介绍要主动引导到自己准备充分的项目上；
  5. 反问要体现对岗位、团队技术栈和成长路径的关注。

这场面试本身也是一次很好的提醒：后端面试的关键不是把所有八股都背完，而是要让简历上的每一行都经得起追问。
